Classes e Instâncias
Construindo Objetos Robustos em C#

Definição de Classes

As classes em C# são moldes para criar objetos:

  • Membros: campos, propriedades, métodos
  • Construtores: inicialização de instâncias
  • Modificadores de acesso: public, private, protected, internal
  • Tipos de referência e valor

Código Exemplo

class Cliente { public int Id { get; set; } public string Nome { get; set; } public string Email { get; set; } public Cliente(string nome, string email) { Nome = nome; Email = email; } public bool Validar() { return !string.IsNullOrEmpty(Nome) && !string.IsNullOrEmpty(Email); } }
Herança e Polimorfismo
Extensão e Flexibilidade em C#

Herança em C#

C# suporta herança simples:

  • Classe derivada herda de uma classe base
  • Uso da palavra-chave : ClasseBase
  • Palavras-chave virtual, override, abstract
  • Classe base object para todos os tipos

Polimorfismo

Vários tipos podem ser tratados como uma superclasse:

class Pessoa { public virtual void Apresentar() { Console.WriteLine("Sou uma pessoa"); } } class Funcionario : Pessoa { public override void Apresentar() { Console.WriteLine("Sou um funcionário"); } }
Interfaces e Classes Abstratas
Contratos e Especializações em C#

Interfaces

Definem contratos que classes devem seguir:

  • Todos os membros são implicitamente públicos
  • Podem incluir propriedades, métodos, eventos
  • Classes podem implementar múltiplas interfaces
  • Usadas para desacoplamento e testabilidade

Classes Abstratas

Classe base incompleta que define estrutura:

interface IPagamento { decimal CalcularTaxa(); void Processar(decimal valor); } abstract class PagamentoBase { public abstract decimal Taxa { get; } public virtual void Confirmar() { Console.WriteLine("Pagamento confirmado"); } }

Relações em Diagramas UML
Associação, Agregação e Composição

Tipos de Relações

  • Associação: Relação direta entre classes (linha simples)
  • Agregação: "Tem um" - relação fraca (losango vazio)
  • Composição: "É parte de" - relação forte (losango preto)
  • Dependência: Relação de uso temporário (seta tracejada)

Hierarquias de Herança - Modelagem Estática
Representação UML de Herança e Polimorfismo

Notação de Herança UML

  • Seta de Extensão: Direcionada da classe filha para a classe pai
  • Rótulo: Geralmente omitido para herança
  • Estereótipos: Como <<abstract>> para classes abstratas

Herança permite reuso e especialização de comportamentos e atributos.

Hierarquia UML

Model-View-Controller (MVC) em C#
Organizando Aplicações com Separação de Responsabilidades

Componentes do MVC

  • Model: Representa os dados e a lógica de negócio
  • View: Interface de usuário (HTML, Razor)
  • Controller: Orquestra a interação entre Model e View

O MVC separa preocupações, facilitando manutenção e testes unitários.

Exemplo de Estrutura em C#

// Model public class Produto { public int Id { get; set; } public string Nome { get; set; } public decimal Preco { get; set; } } // Controller public class ProdutoController : Controller { public IActionResult Detalhes(int id) { var produto = _repositorio.ObterPorId(id); return View(produto); } }

Modelagem Arquitetural do Padrão

Classes MVC

  • Model: Classes de domínio e DTOs

Boas Práticas em Programação Orientada a Objetos
SOLID e Outros Princípios em C#

Princípios SOLID

  • S: Single Responsibility Principle
  • O: Open/Closed Principle
  • L: Liskov Substitution Principle
  • I: Interface Segregation Principle
  • D: Dependency Inversion Principle

Outras Práticas

  • Nomenclatura clara e consistente
  • Evitar classes e métodos excessivamente grandes
  • Utilizar interfaces para desacoplamento
  • Implementar padrões de projeto apropriadamente

🎯 Foco Empresarial: Em sistemas ERP e de informação, seguir boas práticas POO é essencial para escalabilidade, manutenção e integração com outros sistemas.

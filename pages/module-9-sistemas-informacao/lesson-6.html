Por que Arquitetura em Camadas?
A Fundação de Sistemas ERP em Blazor

Separação de Preocupações (SoC)

  • Escalabilidade: Facilita o crescimento e manutenção
  • Testabilidade: Permite testes isolados por componente
  • Reuso: Lógica de negócio compartilhável entre Web/Mobile
  • Interoperabilidade: Camadas conversam via contratos (Interfaces)

As 4 Camadas Clássicas

  • Presentation: Blazor (UI / Componentes)
  • Application: Services (Orquestração / DTOs)
  • Domain: Lógica Pura (Entidades / Negócio)
  • Infrastructure: Dados (EF Core / Repositórios)

📊 Reflexão: Mudar de banco ou interface deve ter impacto zero na lógica de domínio (o coração do ERP).

Classes e Instâncias na Arquitetura
Moldes e Ciclos de Vida no Blazor

Classes (Os Moldes)

Cada camada é composta por classes específicas:

  • Entidades: Classes que representam tabelas
  • Serviços: Classes com lógica de negócio
  • DTOs: Classes para transporte de dados

Instâncias (Os Objetos)

No Blazor, as instâncias são gerenciadas pela Injeção de Dependência:

// Classe (Molde) public class PedidoService { ... } // Instância (Objeto vivo) // O ASP.NET cria para você via @inject @inject PedidoService pedidoService
DTOs vs Entidades de Domínio
Protegendo o Coração do ERP

Entidades (Domain)

  • Regras de Negócio: Lógica central do sistema
  • Persistência: Mapeadas ao banco (via EF)
  • Integridade: Validações complexas

DTOs (Application)

  • Transferência: Dados "achatados" para a View
  • Segurança: Oculta campos sensíveis (senhas, etc)
  • Performance: Somente o necessário para o Blazor

💡 Dica: Jamais exponha Entidades diretamente na View. Use Mappers (como AutoMapper) para converter entre Entidade e DTO.

Injeção de Dependência (DI)
O Elo da Arquitetura Blazor

O Papel da DI

  • Acoplamento: Reduz dependências diretas
  • Flexibilidade: Troca de implementações fácil
  • Vida (Lifetimes): Singleton, Scoped, Transient

Implementação Blazor

// Program.cs builder.Services.AddScoped<IProdutoService, ProdutoService>(); // Componente Razor @inject IProdutoService ProdutoService // Classe de Serviço public class ProdutoService : IProdutoService { private readonly IRepository<Produto> _repository; public ProdutoService(IRepository<Produto> repo) => _repository = repo; }

O Fluxo de uma Requisição
Arquitetura em Camadas na Prática

Passo a Passo

  1. View: Componente Blazor dispara evento (clique em salvar)
  2. Application: Serviço recebe DTO e valida permissões
  3. Domain: Serviço orquestra Entidade e regras de negócio
  4. Infrastructure: Repositório persiste a Entidade no BD

Herança na Arquitetura ERP
Reuso e Especialização em Camadas

Classes Base (BaseClasses)

A herança é o pilar para padronização:

  • EntityBase: Id, DataCriacao, Status
  • ServiceBase: Logs, Validações comuns
  • ComponentBase: Layout, Mensagens de UI

Exemplo em Blazor

public abstract class EntityBase { public int Id { get; set; } } public class Produto : EntityBase { public string Nome { get; set; } } // Produto herda 'Id' de EntityBase
Estrutura de Projetos (Solution)
Como Organizar os Arquivos no Mundo Real

Projetos da Solution

  • SeuProjeto.Web (Blazor)
  • SeuProjeto.Application (Services/DTOs)
  • SeuProjeto.Domain (Entities/Interfaces)
  • SeuProjeto.Infra (DB Context/Repos)
  • SeuProjeto.Shared (Utilidades)

Boas Práticas

  • Referencie apenas o necessário
  • Use CancellationToken em chamadas async
  • Mantenha as Entidades "magras"
  • Trate erros na Camada de Aplicação

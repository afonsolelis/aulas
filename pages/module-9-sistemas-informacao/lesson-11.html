⚠️ Por que precisamos de SOLID?

Sistemas sem bons princípios sofrem de "Codigite Aguda":

Sintomas

  • Rigidez: Difícil de mudar.
  • Fragilidade: Muda aqui, quebra ali.
  • Imobilidade: Impossível de reutilizar partes.
  • Viscosidade: Fazer o "certo" dá muito trabalho.

O SOLID ajuda a reduzir o acoplamento e aumentar a coesão.

"Código limpo parece que foi escrito por alguém que se importa."
— Robert C. Martin (Uncle Bob)
SRP - Single Responsibility Principle
Princípio da Responsabilidade Única
S

"Uma classe deve ter um, e apenas um, motivo para mudar."

❌ Ruim (Multifuncional)
public class RelatorioService {
    public void GerarRelatorio() { /* ... */ }
    public void SalvarNoBanco() { /* ... */ }
    public void EnviarPorEmail() { /* ... */ }
}
✅ Bom (Coeso)
public class RelatorioGerador { ... }
public class RelatorioRepositorio { ... }
public class EmailService { ... }
OCP - Open/Closed Principle
Princípio do Aberto/Fechado
O

"Entidades de software devem estar abertas para extensão, mas fechadas para modificação."

❌ Ruim (IFs Infinitos)
if (tipo == "PDF") GerarPdf();
else if (tipo == "Excel") GerarExcel();
// Preciso mudar a classe para cada novo tipo!
✅ Bom (Abstração)
public interface IExportador { Exportar(); }
public class PdfExportador : IExportador { ... }
// Adiciono novos sem mexer no core.
LSP - Liskov Substitution Principle
Princípio da Substituição de Liskov
L

"Subclasses devem ser substituíveis por suas classes base sem quebrar o sistema."

Se S é um subtipo de T, então objetos do tipo T podem ser substituídos por objetos do tipo S.

A regra de ouro: Nunca lance NotImplementedException em heranças.

❌ O clássico erro do Pinguim
public class Passaro { public virtual void Voar() { ... } }
public class Pinguim : Passaro { 
    public override void Voar() => throw new Exception("Não voo!"); 
}
I - ISP - Interface Segregation Principle
Princípio da Segregação de Interface
I

"Clientes não devem ser forçados a depender de interfaces que não utilizam."

❌ Interface "Gorda"
public interface ITrabalhador {
    void Trabalhar();
    void Comer();
}
✅ Interfaces Específicas
public interface ITrabalhavel { void Trabalhar(); }
public interface IComivel { void Comer(); }
// Robôs não precisam implementar 'Comer'!
DIP - Dependency Inversion Principle
Princípio da Inversão de Dependência
D

"Dependa de abstrações, não de implementações concretas."

❌ Acoplamento Forte
public class Pedido {
    // Depende da classe concreta!
    private SqlServerDb _db = new SqlServerDb();
}
✅ Inversão via Interface
public class Pedido {
    private IDatabase _db;
    public Pedido(IDatabase db) => _db = db;
}
🛠️ Atividade: Auditoria SOLID

Missão: Refatorar o Dashboard

Abra o código-fonte do seu projeto atual e siga estes passos:

  1. Identificar: Escolha uma classe ou componente (ex: Service de dados) que pareça estar fazendo "coisas demais".
  2. Diagnosticar: Quais princípios SOLID ela está violando? (Geralmente SRP e DIP).
  3. Refatorar: Aplique a refatoração para separar responsabilidades ou inverter dependências.
  4. Documentar: No README.md, registre o "Antes" e "Depois".

O objetivo não é perfeição, mas reduzir a dívida técnica!

🚀 Guia Rápido de Refatoração

1. Extrair Classe

Se a classe tem muitos métodos não relacionados, mova-os para uma nova classe.

2. Extrair Interface

Crie uma interface para o seu serviço e use Injeção de Dependência no Blazor.

// No Program.cs (Blazor) builder.Services.AddScoped<IDadosService, MeusDadosService>(); // No componente .razor @inject IDadosService DadosService

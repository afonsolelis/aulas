1. Por que esta aula importa para a prova e para o projeto

Esta aula parece operacional, mas ela define conceitos que sustentam o resto do módulo: ambiente de desenvolvimento, diferença entre ferramentas de build e execução, estrutura inicial da aplicação e a prática de entender os dados antes de desenvolver.

Para a prova, o aluno precisa saber explicar não só “como instalar”, mas por que o SDK é necessário, o que o Blazor SSR resolve e qual é o papel do discovery técnico no início de um projeto.

2. Setup de máquina: objetivo real

Setup não é burocracia. É padronização mínima para que o grupo consiga desenvolver, compilar, testar e depurar sem perder tempo com diferenças irrelevantes de ambiente.

Checklist conceitual

  • Sistema operacional compatível: garantir suporte às ferramentas da stack.
  • Git configurado: permitir versionamento e colaboração.
  • .NET SDK instalado: criar, compilar, rodar e testar aplicações.
  • Editor/IDE configurado: produtividade, debug e autocomplete.
  • Docker: padronizar serviços auxiliares e reduzir diferenças locais.
Um dos erros mais comuns é ter somente o runtime instalado. Nesse caso, a máquina executa aplicações prontas, mas não cria nem compila projeto novo.

3. SDK vs Runtime

Item Função Quando é suficiente
Runtime Executar aplicação já compilada Máquina de produção ou usuário final
SDK Criar, compilar, testar e publicar Máquina de desenvolvimento

Regra para prova: sem SDK você não desenvolve; sem runtime você não executa.

4. C# é compilado ou interpretado?

Do ponto de vista didático, C# é uma linguagem compilada, mas a execução passa por uma etapa intermediária. O código-fonte é compilado para IL (Intermediate Language), e depois o runtime .NET usa o CLR/JIT para transformar isso em código executável pela máquina.

  • Compilação: C# → IL.
  • Execução: CLR/JIT → código nativo.
  • Conseqüência: boa portabilidade com desempenho aceitável e forte suporte de runtime.
Na prova, vale diferenciar “compilação para linguagem intermediária” de “execução imediata como linguagens puramente interpretadas”.

5. Blazor Web App e SSR

Blazor permite construir interface web usando C#. Na disciplina, o ponto de partida é o Blazor Web App com SSR, porque ele reduz atrito inicial e integra bem com a stack .NET.

O que significa SSR

Server-Side Rendering significa que a primeira versão do HTML é montada no servidor e enviada pronta ao navegador. Isso ajuda em tempo de resposta inicial, SEO e simplificação de certas interações.

  • Frontend e backend usando a mesma linguagem.
  • Componentização reutilizável.
  • Boa aderência para dashboards, CRUDs e interfaces corporativas.

6. Anatomia básica do projeto

  • Program.cs: ponto de entrada e configuração da aplicação.
  • Components/Pages: componentes e páginas navegáveis.
  • wwwroot: CSS, imagens, scripts e outros arquivos estáticos.
  • appsettings.json: configurações da aplicação.

A lógica arquitetural da disciplina começa justamente aqui: separar interface, configuração e infraestrutura desde cedo evita confusão quando o projeto cresce.

7. Discovery de dados: o que é e por que acontece antes do desenvolvimento

Discovery é o processo de exploração inicial das fontes de dados para entender estrutura, qualidade, significado e limitações. Desenvolver dashboard sem discovery costuma gerar tela bonita com indicador errado.

Perguntas que o discovery deve responder

  • Quais fontes de dados existem?
  • Quais tabelas, arquivos ou coleções são relevantes?
  • Quais campos identificam entidades e relacionamentos?
  • Há valores ausentes, duplicidade ou inconsistência?
  • Quais campos realmente alimentam os indicadores do dashboard?
SELECT table_name
FROM information_schema.tables
WHERE table_schema = 'grupoX';

SELECT column_name, data_type
FROM information_schema.columns
WHERE table_name = 'nome_da_tabela';

8. Discovery com arquivos e banco

Fonte O que observar Risco comum

Banco relacional Tabelas, chaves, cardinalidades, frequência de atualização Selecionar tabela errada por nome parecido

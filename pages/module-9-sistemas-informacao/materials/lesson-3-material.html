1. Como o SQL realmente funciona

Escrever SQL não é só montar uma string. O banco interpreta, otimiza e executa a consulta em etapas.

  • Parser: verifica sintaxe e semântica.
  • Optimizer: escolhe um plano de execução.
  • Engine: acessa tabelas, índices e realiza operações.

Na prova, isso ajuda a explicar por que duas queries semanticamente equivalentes podem ter custos muito diferentes.

2. As linguagens dentro do SQL

Categoria Função Exemplos
DDL Definir estrutura CREATE, ALTER, DROP
DML Manipular dados INSERT, UPDATE, DELETE, SELECT
DCL Controlar permissões GRANT, REVOKE
TCL Controlar transações BEGIN, COMMIT, ROLLBACK 
BEGIN;
UPDATE conta SET saldo = saldo - 100 WHERE id = 1;
UPDATE conta SET saldo = saldo + 100 WHERE id = 2;
COMMIT;

3. Índices, views e triggers

Índices

Reduzem custo de busca, mas aumentam custo de escrita e ocupam espaço. Não são gratuitos.

Views

Simplificam consultas recorrentes e encapsulam lógica de leitura.

Triggers

Automatizam reações a eventos do banco, mas podem esconder comportamento e dificultar depuração.

Resumo para prova: todo mecanismo de apoio a leitura ou automação tem benefício e custo operacional.

4. SQL Injection

SQL Injection acontece quando o sistema permite que entrada do usuário altere a estrutura do comando SQL. É uma falha clássica e ainda muito cobrada porque nasce de erro conceitual simples: concatenar string em vez de parametrizar.

Nunca construa consulta com concatenação de entrada externa. O correto é separar comando e parâmetro. 
// Correto com Dapper
var sql = "SELECT * FROM usuarios WHERE email = @Email";
var user = connection.QueryFirstOrDefault<Usuario>(sql, new { Email = email });

5. Dapper no contexto do .NET

Dapper é um micro-ORM. Ele não abstrai o SQL como um ORM completo faria. Em vez disso, ele mapeia resultado para objetos e preserva alto controle sobre a query.

  • Mais explícito e leve que ORMs completos.
  • Bom para dashboards com consultas específicas e sensíveis a desempenho.
  • Exige mais cuidado do desenvolvedor com SQL, schema e manutenção.
var sql = "INSERT INTO pedidos(cliente_id, total) VALUES(@ClienteId, @Total)";
connection.Execute(sql, pedido);

6. Modelagem corporativa: além do ER

Modelagem de dados em contexto corporativo não pode ser tratada apenas como diagrama entidade-relacionamento. O sistema precisa respeitar fronteiras de domínio, responsabilidade organizacional e evolução independente.

  • Bounded context: cada contexto modela sua própria visão do domínio.
  • Governança de dados: define responsabilidade sobre qualidade e uso dos dados.
  • Stewardship: papel de curadoria e manutenção da consistência.

7. Quando NoSQL faz sentido

NoSQL não substitui automaticamente o relacional. Ele é vantajoso quando o modelo orientado a documento ou agregado atende melhor ao caso de uso do que várias junções e transações complexas.

Relacional NoSQL orientado a documento
Forte estrutura e normalização Modelagem por caso de uso e agregados
Bom para integridade referencial e consultas relacionais Bom para leitura agregada e flexibilidade de estrutura
Mais aderente a transações complexas Mais aderente a dados semiestruturados e agregados isolados 
db.createCollection("pedidos", {
  validator: {
    $jsonSchema: {
      bsonType: "object",
      required: ["clienteId", "itens"]
    }
  }
});

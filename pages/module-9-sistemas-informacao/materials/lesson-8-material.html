1. Objetivo da Aula

Levar a aplicação para um patamar mais profissional: pipeline automatizado, falhas tratadas de forma explícita e critérios objetivos para dizer se o sistema está pronto para seguir adiante.

CI Timeout Retry Circuit Breaker SLO

2. O que precisa mudar no projeto

Até aqui, o foco costuma estar em fazer funcionar. Nesta aula, o foco muda para fazer funcionar bem sob pressão. Isso significa considerar instabilidade de rede, lentidão do banco, indisponibilidade de serviços e falhas parciais.

Aplicação em nuvem não é apenas deploy. É capacidade de manter comportamento previsível quando o ambiente deixa de ser ideal.
  • Automatizar build e testes a cada push.
  • Simular falhas reais antes que elas apareçam em produção.
  • Definir quais erros merecem retry e quais devem falhar rápido.
  • Medir latência, disponibilidade e taxa de erro como parte do fluxo de engenharia.

3. CI/CD sem romantização

Continuous Integration significa integrar pequenas mudanças com frequência e validar automaticamente se o sistema continua íntegro. O pipeline precisa ser rápido, legível e confiável.

Fluxo mínimo recomendado

  1. Restore: recuperar dependências.
  2. Build: garantir compilação reproduzível.
  3. Test: validar comportamento esperado.
  4. Chaos/Resilience: executar cenários de falha.
  5. Package/Deploy: seguir apenas se os gates passaram.
stages: [restore, build, test, resilience]

build:
  stage: build
  script:
    - dotnet restore
    - dotnet build --no-restore

resilience:
  stage: resilience
  script:
    - dotnet test tests/ChaosTests

4. Engenharia do caos aplicada ao pipeline

Engenharia do caos não é “quebrar por quebrar”. É formular uma hipótese sobre o comportamento do sistema e testá-la sob falha controlada.

Cenário Pergunta Resultado esperado
Banco lento O request fica preso indefinidamente? Timeout + log claro + resposta previsível
Serviço dependente fora do ar O sistema insiste até colapsar? Circuit breaker abre e reduz cascata
Pico de latência Retry piora a situação? Retry com backoff e limite explícito
Não aplique retry cegamente. Em operações não idempotentes, retry pode duplicar efeitos e agravar o problema.

5. Padrões de resiliência em .NET

Quando usar cada um

  • Timeout: para impedir espera infinita.
  • Retry: para falhas transitórias, com backoff e jitter.
  • Circuit Breaker: para parar de insistir quando a dependência está claramente doente.
  • Fallback: para responder com dado degradado ou mensagem controlada.
  • Bulkhead: para isolar recursos e evitar contaminação geral.
services.AddHttpClient<IRelatorioClient, RelatorioClient>()
  .AddPolicyHandler(Policy.TimeoutAsync<HttpResponseMessage>(TimeSpan.FromSeconds(2)))
  .AddPolicyHandler(HttpPolicyExtensions
    .HandleTransientHttpError()
    .WaitAndRetryAsync(3, tentativa => TimeSpan.FromMilliseconds(200 * tentativa)))
  .AddPolicyHandler(HttpPolicyExtensions
    .HandleTransientHttpError()
    .CircuitBreakerAsync(5, TimeSpan.FromSeconds(30)));

6. Observabilidade: como saber se a solução está saudável

Sem observabilidade, resiliência vira fé. O time precisa instrumentar a aplicação para enxergar comportamento real.

  • Logs: eventos relevantes com contexto de erro e correlação.
  • Métricas: latência, throughput, taxa de erro, tempo de recuperação.
  • Traces: onde o request perdeu tempo entre componentes.

SLI, SLO e SLA

  • SLI: indicador medido, como p95 de latência.
  • SLO: meta técnica, como 95% das requisições abaixo de 300 ms.
  • SLA: compromisso externo, normalmente contratual.

7. Checklist prático para o grupo

  1. Escolher uma dependência crítica do sistema.
  2. Definir quais falhas devem ser simuladas no CI.
  3. Implementar políticas de timeout, retry e breaker para essa dependência.
  4. Adicionar logs e métricas mínimas para observar o comportamento.
  5. Documentar o que acontece quando o sistema entra em modo degradado.

8. Entregável esperado

Ao final da aula, o grupo deve sair com evidências práticas, não apenas com intenção arquitetural.

  • Pipeline com estágio adicional de resiliência ou caos controlado.
  • Ao menos dois cenários de falha automatizados.
  • Políticas implementadas no código e justificadas tecnicamente.
  • Critérios de aprovação do pipeline baseados em métrica, não em opinião.

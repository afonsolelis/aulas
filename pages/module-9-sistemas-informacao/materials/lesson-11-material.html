1. Para que SOLID serve de verdade

SOLID não existe para “deixar o código bonito”. Ele existe para reduzir custo de mudança. Em um projeto de dashboard, as regras mudam, integrações mudam, relatórios mudam. Se cada ajuste exige mexer em muitas partes ao mesmo tempo, o projeto entra em atrito constante.

Bom uso de SOLID aumenta testabilidade, clareza de responsabilidade e previsibilidade de impacto.

2. Os cinco princípios em linguagem direta

SRP: uma classe deve ter uma responsabilidade central e um motivo principal para mudar.
OCP: o sistema deve permitir extensão de comportamento sem reescrever a base toda.
LSP: se um tipo substitui outro, ele deve respeitar as expectativas de quem usa a abstração.
ISP: interfaces devem ser pequenas e focadas no que cada cliente realmente usa.
DIP: camadas altas devem depender de abstrações, não de detalhes concretos.

3. Como identificar violações no projeto

  • SRP quebrado: service que lê arquivo, valida regra, grava banco e monta DTO.
  • OCP quebrado: toda nova regra exige adicionar mais um if ou switch.
  • LSP quebrado: implementação concreta exige tratamento especial ou lança exceções inesperadas.
  • ISP quebrado: interface com métodos que metade das classes implementa “no vazio”.
  • DIP quebrado: componente de alto nível instancia diretamente detalhes como API client, arquivo ou repositório.

4. Exemplo de refatoração orientada por SOLID

Um erro comum em Blazor é concentrar leitura de dados, regra de negócio e renderização no mesmo componente.

// Antes
public partial class Dashboard
{
    protected override async Task OnInitializedAsync()
    {
        var linhas = await File.ReadAllLinesAsync("dados.csv");
        // parse, regra, agregação e decisão de interface no mesmo lugar
    }
}
// Depois
public interface IIndicadorService
{
    Task<ResumoIndicadores> ObterResumoAsync();
}

public partial class Dashboard
{
    [Inject] public IIndicadorService Service { get; set; } = default!;

    protected override async Task OnInitializedAsync()
        => resumo = await Service.ObterResumoAsync();
}

5. SOLID e arquitetura em camadas

Os princípios aparecem melhor quando ligados à arquitetura real do projeto:

  • Apresentação: componentes e controllers orquestram interação.
  • Aplicação: casos de uso coordenam fluxo.
  • Domínio: regras centrais e contratos.
  • Infraestrutura: banco, APIs externas, arquivos e mensageria.

DIP é o elo mais visível aqui: a camada interna define o contrato, a externa implementa.

6. Perguntas de revisão para o grupo

  1. Qual classe hoje concentra responsabilidades demais?
  2. Onde há lógica condicional que poderia virar estratégia ou abstração?
  3. Quais interfaces estão grandes demais ou mal nomeadas?
  4. Quais dependências concretas dificultam testar o sistema?

7. Entregável esperado

  • Uma violação SOLID identificada no projeto.
  • Uma refatoração concreta implementada.
  • Justificativa curta explicando qual princípio orientou a mudança.
  • Evidência de que o comportamento continuou correto.

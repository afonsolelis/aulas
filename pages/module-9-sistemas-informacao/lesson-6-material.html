Objetivo da Aula

Compreender como os conceitos de Classes, Instâncias e Herança são aplicados para construir uma Arquitetura em Camadas (MVC/MVVM) robusta em aplicações Blazor, focando na separação de responsabilidades e escalabilidade de sistemas ERP.

1) Classes e Instâncias na Arquitetura

Em um sistema corporativo, não usamos classes apenas para representar "objetos do mundo real", mas para organizar as responsabilidades do software:

  • Classes de Serviço: Atuam como os "Controllers" ou "ViewModels", contendo a lógica de orquestração.
  • Classes de Entidade: Representam o estado e as regras de negócio puras.
  • Instâncias via DI: Diferente do new Classe() manual, no Blazor usamos Injeção de Dependência para que o framework gerencie o ciclo de vida das instâncias (Singleton, Scoped, Transient).
// A Classe (Molde)
public class ProdutoService : IProdutoService { ... }

// A Instância (Injetada no componente)
@inject IProdutoService ProdutoService

2) O Padrão MVC no Blazor

O Blazor adapta o MVC tradicional para uma abordagem baseada em componentes:

  • View: Componentes .razor que cuidam apenas da UI e eventos.
  • Model: DTOs e Entidades que carregam os dados.
  • Controller/Service: Classes de serviço que realizam a ponte entre a UI e a persistência de dados.
Dica: Mantenha seus componentes Razor "magros". A lógica complexa deve residir em classes C# separadas (Serviços).

3) Herança e Especialização

Usamos herança para criar padrões de projeto consistentes entre as camadas:

  • Base Classes para Entidades: Criamos uma EntityBase com propriedades comuns como Id e DataCriacao.
  • Base Classes para Componentes: Podemos ter um componente base para padronizar comportamentos de UI (ex: modais, alertas).
public abstract class EntityBase {
    public int Id { get; set; }
    public DateTime AtualizadoEm { get; set; }
}

public class Cliente : EntityBase { 
    public string RazaoSocial { get; set; } 
}

4) DTOs vs Entidades de Domínio

Uma das práticas mais importantes em arquitetura em camadas é não expor suas tabelas do banco diretamente na tela:

  • Entidade: Tem todas as propriedades do banco, incluindo chaves estrangeiras e dados sensíveis.
  • DTO (Data Transfer Object): Uma classe simplificada que contém apenas o que a View precisa exibir ou capturar.

5) As 4 Camadas do Sistema

  1. Presentation (.Web): Componentes Blazor, CSS, Layouts.
  2. Application (.Application): Interfaces de Serviços, Implementações de Serviços e DTOs.
  3. Domain (.Domain): Entidades, Interfaces de Repositórios e Regras de Negócio.
  4. Infrastructure (.Infra): DB Context (EF Core), Repositórios concretos e integrações externas.

6) Modelagem UML para ERP

Ao desenhar o diagrama de classes do seu sistema, foque nas relações:

  • Associação: Como os serviços conversam entre si.
  • Herança: Como as entidades compartilham propriedades comuns.
  • Realização: Como as classes de serviço implementam seus contratos (Interfaces).

7) Atividade Prática

Aplique os conceitos de POO no seu projeto de grupo:

  1. Crie uma Classe Base para suas entidades.
  2. Implemente a Herança nos seus modelos principais.
  3. Defina os DTOs necessários para uma tela específica do seu ERP.
  4. Esboce o Diagrama de Classes mostrando a interação entre View, Service e Model.

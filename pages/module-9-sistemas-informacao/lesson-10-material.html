Programação por Domínios
🎞️ Ver Slides ← Voltar ao Módulo

Leitura central da aula

Programar por domínios é organizar o software de acordo com o significado do negócio. O objetivo não é ter mais camadas, e sim menos ambiguidade, menos acoplamento e mais coerência entre conversa, código e arquitetura.

Objetivo da Aula

Entender como linguagem ubíqua, bounded contexts, entidades, value objects, agregados e camadas de arquitetura ajudam a estruturar aplicações mais próximas do problema real do negócio.

1. O problema que a aula resolve

Muitos sistemas começam corretos do ponto de vista técnico e incorretos do ponto de vista conceitual. O time faz API, controller, banco e tela, mas os nomes não correspondem ao que o negócio realmente quer dizer. Resultado: o software “funciona”, mas cada mudança exige exceções, mapeamentos improvisados e retrabalho.

Programação por domínios tenta atacar essa raiz: o modelo do software deve representar o domínio, não apenas o armazenamento.

2. O que é domínio

Domínio é o campo de problema do negócio: conceitos, regras, restrições, decisões, exceções e vocabulário usados para operar a realidade.

  • No Projeto 9, domínio inclui indicadores, metas, relatórios gerenciais, ciclos de avaliação e leituras analíticas.
  • Domínio não é sinônimo de tabela, endpoint ou página.

3. O que é DDD

DDD significa Domain-Driven Design. A ideia central é projetar o software guiado pelo modelo de negócio e pela linguagem compartilhada entre especialistas de domínio e equipe técnica.

  • Não é apenas “criar uma pasta Domain”.
  • Não depende obrigatoriamente de microservices.
  • É mais valioso quando o domínio é complexo e muda com frequência.

4. Linguagem ubíqua

Linguagem ubíqua é o vocabulário compartilhado entre time técnico e negócio. Se o negócio fala em ciclo de avaliação, meta mensal ou turnover, esses termos devem aparecer no código, não ser substituídos por nomes genéricos como DataManager ou StatusHandler.

Referência: Ubiquitous Language

5. Bounded Context

Um Bounded Context é uma fronteira onde um modelo e sua linguagem têm significado consistente. Quando a linguagem muda, a tendência é que o contexto também precise mudar.

Exemplo: a palavra produto pode significar uma oferta no comercial, um item faturável no financeiro e uma dimensão analítica no BI. Forçar um único modelo para todos esses usos costuma gerar confusão.

Referência: Bounded Context

6. Entidade, Value Object, Serviço de Domínio e Repositório

  • Entidade: tem identidade e continuidade no tempo.
  • Value Object: é definido por seus atributos, não por identidade.
  • Serviço de Domínio: abriga regra de negócio que não cabe bem em uma única entidade.
  • Repositório: abstrai a persistência para o domínio não depender de SQL, EF ou HTTP.

O importante é distribuir responsabilidade com critério, não apenas criar classes “bonitas”.

7. Agregado

Agregado é um agrupamento de objetos de domínio que precisa preservar consistência através de uma raiz, chamada Aggregate Root. Nem tudo que se relaciona precisa estar no mesmo agregado; a pergunta correta é: “o que precisa mudar junto, respeitando invariantes de negócio?”

Um bom agregado reduz acoplamento e ajuda a manter transações mais coerentes.

8. Camadas no .NET

Projeto9.WebApi
  -> endpoints, controllers, autenticação, DTOs

Projeto9.Application
  -> casos de uso, orquestração, comandos, consultas

Projeto9.Domain
  -> entidades, value objects, agregados, serviços de domínio, interfaces

Projeto9.Infrastructure
  -> persistência, integrações, cache, mensageria

A ideia é impedir que o núcleo de domínio dependa diretamente de detalhes externos de tecnologia.

9. Anti-Corruption Layer (ACL)

Quando o sistema integra com legados ou terceiros, a ACL traduz contratos externos para a linguagem do seu domínio. Isso evita contaminar o núcleo com nomes, formatos e decisões que pertencem a outro contexto.

Integrar bem não é copiar o modelo do outro sistema para dentro do seu.

10. Modelar capacidades, não telas

Telas mudam rápido. Capacidades de negócio tendem a ser mais estáveis. Por isso, prefira modelar FecharCicloDeAvaliacao ou CompararMetaPlanejadaComRealizada em vez de TelaDashboardManager.

O frontend consome capacidades do domínio; ele não deveria definir o significado principal do sistema.

12. Atividade de sala

  • Escolher um subdomínio principal da aplicação.
  • Definir uma linguagem ubíqua mínima para esse subdomínio.
  • Separar pelo menos 2 bounded contexts.
  • Propor entidades, value objects e um agregado raiz.
  • Explicar o que pertence a domínio, aplicação e infraestrutura.

O objetivo não é desenhar uma solução perfeita, e sim mostrar que o grupo já consegue pensar o software pelo negócio.

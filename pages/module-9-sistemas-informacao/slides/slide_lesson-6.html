Por que Arquitetura em Camadas?

🖥️ Apresentação

Componentes Blazor, páginas Razor, bindings de UI. Não deve conter lógica de negócio.

⚙️ Aplicação

Services, Use Cases, DTOs. Orquestra o fluxo sem depender de banco ou framework UI.

🗄️ Infraestrutura

Repositórios, Dapper, EF Core, APIs externas. Implementa as interfaces do domínio.

Regra de Dependência: Camadas externas dependem de internas. Domínio não conhece banco nem UI.

Classes e Instâncias na Arquitetura

📐 Classe = Molde

public class Produto {
public int Id { get; set; }
public string Nome { get; set; } = "";
public decimal Preco { get; set; }
public bool EstaAtivo() => Preco > 0;
}

🔄 Instância = Objeto em memória

// Criar instância
var produto = new Produto {
Id = 1,
Nome = "Notebook",
Preco = 2500.00m
};

// Invocar comportamento
bool ativo = produto.EstaAtivo();

DTOs vs Entidades de Domínio

📦 DTO — Data Transfer Object

  • Apenas dados, sem comportamento
  • Usado para entrada/saída de APIs
  • Protege o domínio de exposição direta
public record CriarProdutoDto(
string Nome, decimal Preco);

🏛️ Entidade — Domínio Rico

  • Identidade única (Id)
  • Comportamento de negócio encapsulado
  • Invariantes protegidas no construtor
public class Produto {
private Produto() {}
public static Produto Criar(string nome, decimal preco) {
if (preco <= 0) throw new ArgumentException();
return new Produto { Nome = nome, Preco = preco };
}
}

Injeção de Dependência (DI)

💉 Por que usar DI?

  • Desacoplamento entre implementação e contrato
  • Facilita testes com mocks
  • Gerenciamento de ciclo de vida pelo container
  • Blazor usa DI nativamente

🔧 Registro no Program.cs

// Program.cs
builder.Services.AddScoped<IProdutoService, ProdutoService>();

// Componente Blazor
@inject IProdutoService ProdutoService
@code {
protected override async Task OnInitializedAsync() {
produtos = await ProdutoService.ListarTodos();
}
}

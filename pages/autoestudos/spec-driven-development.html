📝 Metodologia · Spec-Driven Development

Spec-Driven Development (SDD)

Entenda a importância de especificar contratos, schemas e regras de negócio antes de escrever a primeira linha de código, e como isso potencializa a engenharia de software moderna.

O Objetivo deste Autoestudo

Aprender o conceito de Spec-Driven Development (SDD), compreendendo como especificações técnicas detalhadas guiam o desenvolvimento de software, garantem alinhamento entre times e otimizam a automação e geração de código (inclusive por Inteligências Artificiais).

1. O que é Spec-Driven Development?

Muitas vezes no desenvolvimento de software, a pressa em ver algo funcionando nos faz abrir o editor e começar a programar imediatamente. Essa prática comumente resulta em retrabalho, inconsistência de APIs e desalinamento com stakeholders. O **Spec-Driven Development (SDD)** é uma metodologia de desenvolvimento de software onde a especificação técnica atua como a única fonte de verdade (Single Source of Truth - SSoT).

No SDD, antes de implementar qualquer código ou teste, os engenheiros escrevem especificações rigorosas que descrevem o comportamento planejado do sistema. Essas especificações contêm:

  • Assinaturas e contratos de API (ex: OpenAPI/Swagger).
  • Modelagens e esquemas de dados (ex: Schemas de banco de dados, JSON Schemas).
  • Regras de negócio explícitas e fluxos de exceção.
SDD vs. TDD (Test-Driven Development): Enquanto o TDD foca em escrever testes automatizados antes do código de produção para garantir corretude lógica interna, o SDD foca em descrever as interfaces e limites do sistema externamente. O SDD serve como base para o TDD, fornecendo os contratos e cenários que serão testados.

2. A Tríplice da Especificação

Uma especificação madura no ecossistema de dados e software é composta por três grandes pilares:

🔗 Contrato de API

Criação de rotas, métodos HTTP, códigos de status e payload de requisição/resposta usando especificações de mercado como OpenAPI v3.

💾 Schema de Dados

Definição explícita de tipos, chaves, índices e restrições de integridade no banco de dados e nos modelos de tráfego de dados.

🤖 Regras e Cenários

Critérios de aceitação escritos em linguagem compreensível por humanos e computadores (ex: sintaxe Gherkin / BDD).

3. Spec-Driven Development e Inteligência Artificial

Com o avanço de assistentes de codificação de IA (como Gemini, Claude, Antigravity, Copilots), o SDD tornou-se ainda mais essencial. As IAs são excelentes para gerar códigos estruturados, mas sofrem de alucinações ou desvios de design quando recebem instruções vagas ou prompts imprecisos.

Quando fornecemos um **contrato bem especificado** a um assistente de IA, a taxa de sucesso da geração de código sobe drasticamente. A especificação limita as opções de implementação da IA às regras de arquitetura corretas, permitindo que a IA gere código verde de testes, implementações de repositórios e controllers com altíssima aderência ao design inicial do sistema.

Como o SDD melhora o uso de IAs na programação:
  1. Redução de ambiguidade: A IA não precisa adivinhar o tipo de dados ou a nomenclatura de uma propriedade de banco.
  2. Geração robusta de testes: A especificação serve de base para que a IA gere cenários de teste de unidade e integração completos.
  3. Idempotência na Engenharia: Permite regenerar código a partir de atualizações na especificação de maneira limpa.

4. O Ciclo de Vida do SDD

O fluxo de trabalho seguindo SDD segue quatro etapas obrigatórias:

  1. Escrever a Especificação: Desenhar o contrato OpenAPI ou o schema Markdown detalhando as rotas, inputs, validações e outputs.
  2. Revisar e Validar: Alinhar a especificação com stakeholders técnicos e de negócios (ex: Front-end aprova as respostas do Back-end antes da codificação).
  3. Gerar/Implementar o Código: Codificar seguindo estritamente a especificação (usando IA ou manualmente), sem adicionar propriedades ou comportamentos que não estavam descritos.
  4. Verificar: Executar testes automáticos de contrato ou integração para garantir que o código escrito se comporta exatamente como a especificação prevê.

5. Exemplo Prático de uma Spec

Abaixo, veja um exemplo simplificado de uma especificação técnica Markdown para a criação de um endpoint de criação de cotações em um microsserviço de faturamento:

# Especificação: Endpoint de Cotação (Billing Service)

## Informações Gerais
*   **Rota:** `/api/v1/quotes`
*   **Método:** `POST`
*   **Content-Type:** `application/json`

## Payload de Requisição
```json
{
  "customerId": "uuid (obrigatório)",
  "items": [
    {
      "productId": "string (obrigatório)",
      "quantity": "integer (obrigatório, mínimo 1)",
      "priceUnit": "float (obrigatório, positivo)"
    }
  ]
}
```

## Regras de Negócio e Validações
1.  **Validação de Cliente:** O `customerId` deve ser um UUID válido e pertencer a um cliente ativo no banco de dados. Caso contrário, retornar `400 Bad Request` com o erro correspondente.
2.  **Cálculo do Total:** O microsserviço deve calcular o valor total da cotação multiplicando `quantity` * `priceUnit` para cada item e somando os resultados.
3.  **Desconto por Quantidade:** Se a quantidade de qualquer item for maior ou igual a 10, deve ser aplicado um desconto de 5% sobre o preço unitário desse item.

## Respostas Esperadas
*   `201 Created`: Cotação gerada com sucesso. Retorna o ID da cotação criada e o total final.
*   `400 Bad Request`: Payload inválido ou falha em regra de negócio.
*   `500 Internal Server Error`: Erro inesperado do servidor.

6. Desafio Prático de Autoestudo

Agora é sua vez de exercitar o Spec-Driven Development!

Instruções para o Desafio:
  1. Escolha uma funcionalidade do projeto do seu grupo (ex: Cancelamento de pedido, Consulta de faturamento, Cadastro de produto).
  2. Escreva uma especificação Markdown (como no exemplo da Seção 5) detalhando a rota, payload, validações obrigatórias e regras de negócio.
  3. Utilize sua especificação como prompt em uma IA de código (ex: Gemini ou Claude) para gerar a implementação do Controller e os testes unitários dessa rota.
  4. Analise: o código gerado foi mais limpo e direto ao ponto do que quando você programa sem a especificação prévia?

Por que monolitos quebram em escala 💥
4 sintomas que aparecem antes da arquitetura virar serviço
🚀
DEPLOY

Deploy bloqueia time

Mudou 1 linha do checkout? Sobe o app inteiro. Janela de deploy semanal vira gargalo de todos os squads.

📈
SCALE

Escala tudo ou nada

Se o módulo de busca fica saturado, você sobe todo o monolito. Custo de infra cresce sem precisão.

👥
PEOPLE

Times se atropelam

Cinco squads no mesmo repositório significa merge conflicts, code reviews atravessadas e branches longas.

🔥
FAULT

Falha em cascata

Bug no recommender derruba o login. Sem isolamento, qualquer exceção pode tombar todo o processo.

📺 Caso histórico · Netflix → SOA (2008–2012)

Em agosto/2008 a Netflix sofreu corrupção do banco de dados central por 3 dias — DVDs não saíram. Resposta: 4 anos migrando o monolito para centenas de serviços em AWS, com isolamento de falha e deploys independentes. Hoje cada microsserviço cai sem derrubar o catálogo.

🪜 A ponte: SOA é o passo intermediário — quebra o monolito em serviços com contrato explícito sem ainda exigir a granularidade extrema (e custo operacional) de microsserviços puros.

SOA · 8 Princípios de Erl 🏛️
As regras que separam "tenho serviços" de "tenho arquitetura SOA de verdade"
🔗
1. Loose Coupling

Serviços minimizam dependências. Mudança interna não quebra cliente.

📜
2. Service Contract

Comunicação só pelo contrato (REST/OpenAPI, gRPC). Sem atalho interno.

🛡️
3. Autonomy

Serviço controla seu ambiente e dados. Falha local fica local.

💧
4. Statelessness

Não guarda contexto entre chamadas. Escala horizontal trivial.

♻️
5. Reusability

Lógica de negócio empacotada para reuso por múltiplos consumidores.

🧱
6. Composability

Serviços compõem fluxos maiores (orquestração / coreografia).

🔭
7. Discoverability

Metadados publicados (catálogo, gateway). Cliente acha sem código compartilhado.

🎭
8. Abstraction

Esconde a implementação. Só o contrato é público — interno pode mudar.

📚 Fonte: Thomas Erl · Service-Oriented Architecture: Concepts, Technology, and Design · Prentice Hall. Os 8 princípios são a referência canônica do que distingue SOA de uma simples coleção de APIs.

Monolito × SOA × Microsserviços ⚖️
Mesmos problemas resolvidos em granularidades diferentes — escolha pelo contexto
Dimensão Monolito SOA Microsserviços
Granularidade Aplicação única Serviços de negócio (médios) Serviços pequenos · 1 capacidade
Comunicação Chamadas in-process ESB / gateway · SOAP/REST REST/gRPC + mensageria
Deploy Tudo de uma vez Por serviço (com governança central) Independente por serviço
Equipe 1 time grande no mesmo repo Times por domínio + arquiteto central Two-pizza teams autônomos
Falhas Cascata · pode tombar tudo Isoladas (com circuit breaker) Isoladas + bulkhead obrigatório
Dados 1 banco compartilhado Bancos por serviço (governança) Database-per-service rígido

🟥 Quando monolito ainda vale: equipe ≤ 6 devs, domínio único, MVP em validação.

🟦 Quando SOA é o ponto certo: múltiplos canais (mobile, web, parceiros), legado para integrar, governança forte.

🟩 Quando microsserviços compensam: tráfego enorme, autonomia por squad, maturidade em DevOps/observabilidade.

UML de Componentes · Anatomia 🧩
Lollipop = interface fornecida · Socket = interface requerida · Ball-and-socket = conector

⬜ Componente

Caixa com estereótipo «component». Unidade encapsulada.

▫ Porta

Quadradinho na borda. Ponto de comunicação tipado.

🍭 Lollipop

🥄 Socket

6 serviços internos + 2 externos · Mobile → API Gateway → 4 serviços de negócio

🚪 API Gateway · ponto único de entrada · roteamento, throttling e injeção de auth.

🧱 Serviços de negócio · 4 capacidades (auth, catálogo, reco, notificação) · cada um com seu BD.

🔌 Sistemas externos · pagamento e e-mail desacoplados via contrato — fácil trocar provedor.

RNF · ISO/IEC 25010 · Modelo de Qualidade 📐
8 categorias de qualidade · cada RNF deve cair em uma · com SLA mensurável
🎯1. Functional Suitability

Faz o que diz que faz · completo, correto, adequado.

cobertura ≥ 95% dos use cases
⚡2. Performance Efficiency

Tempo de resposta · throughput · uso de recurso.

latência p99 < 300 ms
🔌3. Compatibility

Coexiste e interopera com outros sistemas (formato, protocolo).

REST/JSON OpenAPI 3.0
👤4. Usability

Aprendível, operável, acessível, agradável.

SUS ≥ 75 / WCAG 2.1 AA
🛡️5. Reliability

Disponibilidade, tolerância a falhas, recuperabilidade.

uptime ≥ 99,9% (≤ 8h/ano)
🔒6. Security

Confidencialidade, integridade, não-repúdio, autenticidade.

OWASP Top-10 mitigado
🛠️7. Maintainability

Modular, reusável, analisável, modificável, testável.

cobertura de testes ≥ 80%
📦8. Portability

Adaptável, instalável, substituível em outros ambientes.

container Docker · 12-factor

📏 Regra de ouro: RNF sem número não é RNF — é desejo. Toda categoria precisa virar SLA mensurável. "Tem que ser rápido" ❌ → "p99 da rota /recomendar < 300 ms com 200 req/s" ✅.

🪝 Para a ponderada: cada um dos 5 RNFs que você vai escrever deve ser classificado em uma destas 8 categorias e deve apontar para um componente do diagrama SOA que é responsável.

Atividade · 4 Critérios de Entrega 📐

01
🧩

Diagrama de Componentes SOA

Notação correta: lollipop / socket / portas.

02
📝

Justificativa do Diagrama

Texto abaixo do diagrama explicando os pontos arquiteturais (por que separou esses serviços, quem se comunica com quem).

⭐ Excelência: mostrar como o desacoplamento facilita expansão, lida com picos e integra com legados.

03
📐

5 RNFs com Justificativa

Cada RNF justifica sua necessidade (por que o sistema precisa dele) com SLA mensurável.

04
🎯

Vinculação RNF → Componente

Para cada RNF, indicar qual componente do diagrama é responsável por garantir aquele requisito.

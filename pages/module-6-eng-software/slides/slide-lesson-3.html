Conexão com a Aula 4 · Stored Procedures e Functions
Serviços precisam de banco performático · lógica próxima dos dados

🗄️ Do serviço para o banco

Cada serviço hoje tem seu DB próprio. Na aula 4 descemos para a camada onde nasce o RNF de desempenho:

CREATE FUNCTION verifica_disponibilidade(p_id INT) RETURNS BOOLEAN
-- lógica no banco · 1 round trip · usa índice

📐 RNF orienta a decisão

RNF-1 (p99 < 800ms) e RNF-3 (5k simultâneos) decidem onde colocar a lógica: banco vs aplicação. Stored procedure economiza viagens.

📋 Descrição: atividade prática e avaliativa estruturada para ser realizada durante a aula. Os alunos deverão aplicar os conceitos de Arquitetura Orientada a Serviços (SOA) diretamente no projeto que estão desenvolvendo. O objetivo é que o aluno construa uma arquitetura SOA através de um diagrama de componentes, considerando os blocos principais e integrações externas, além de listar e justificar 5 requisitos não funcionais.

❓ Pergunta-base: com base na solução tecnológica que o seu grupo está desenvolvendo para o parceiro de mercado, você deve desenhar a estrutura técnica, com UML de componentes, do sistema e mapear os requisitos de qualidade que garantirão o seu funcionamento.

🧠 Assuntos: Análise de sistemas · Elicitação de requisitos · Engenharia de requisitos · Histórias de usuários · Projeto e arquitetura de sistemas · Requisitos funcionais e não funcionais.

📚 Conteúdos: Design Arquitetural SOA e Requisitos Não Funcionais.

Atividade · 2 Critérios de Entrega 📐
Cada critério tem um caminho de Excelência ⭐ que vale ponto extra
CRITÉRIO 01
🏗️
Diagrama SOA + Justificativa

Construa uma arquitetura SOA simples usando um diagrama de componentes que represente os blocos principais do seu sistema (frontend, backend, banco de dados) e a integração com componentes de serviços externos (APIs de terceiros relevantes). Logo abaixo do diagrama, escreva um texto justificando os pontos da sua arquitetura.

⭐ Excelência: a justificativa demonstra claramente como a arquitetura em serviços (desacoplada) facilitará futuras expansões do sistema do parceiro e lidará com cenários críticos de uso (ex.: picos de acesso ou integração com sistemas legados).

CRITÉRIO 02
📐
5 RNFs com Justificativa

Liste pelo menos 5 requisitos não funcionais essenciais para o sistema do parceiro. Justifique a necessidade de cada um e explique a relação com a arquitetura SOA proposta no Critério 1.

⭐ Excelência: RNFs baseados nas categorias da ISO 25010 (adequação funcional, eficiência, compatibilidade, usabilidade, segurança, etc.) e o aluno identifica quais componentes específicos do diagrama SOA são responsáveis por garantir cada um.

📌 Formato de entrega (individual):

  1. Documento em PDF ou .md com texto justificativo + diagrama (PNG ou SVG)
  2. Ferramentas sugeridas: draw.io, Lucidchart, Excalidraw, Mermaid
  3. Upload na pasta ponderada/ do seu GitLab individual (não é o repositório do grupo)

⚠️ Atenção: os 2 critérios são obrigatórios. As marcações ⭐ Excelência diferenciam a nota — ataque-as se sobrar tempo.

O Problema · Race condition na compra 🎟️
Ana e Bruno tentam comprar o último ingresso ao mesmo tempo · sem transação
📦 Estoque atual · espetaculos.disponivel = 1
👩 Ana · t = 10:00.001
SELECT disponível → vê 1
UPDATE espetaculos SET disponivel = disponivel - 1
WHERE id = 42;
INSERT INTO reservas ...;
✅ Compra confirmada
⚡
👨 Bruno · t = 10:00.002
SELECT disponível → também vê 1
UPDATE espetaculos SET disponivel = disponivel - 1
WHERE id = 42;
INSERT INTO reservas ...;
✅ Compra confirmada
💥 Resultado · 2 reservas vendidas, estoque = -1 · overbooking e cliente furioso

🎯 Diagnóstico: os dois SELECTs leem o mesmo estado antes de qualquer UPDATE confirmar. Sem BEGIN/COMMIT agrupando leitura+escrita e sem lock, o banco aceita as duas operações. É exatamente o caso que vamos blindar nos próximos slides.

SQL · Célula 1 · BEGIN / COMMIT
Transação atômica · ou tudo passa, ou tudo volta — nada de meio caminho
In [1]:
-- Reserva atômica: decrementa estoque + insere reserva BEGIN; UPDATE espetaculos SET disponivel = disponivel - 1 WHERE id = 42 AND disponivel > 0; INSERT INTO reservas (espetaculo_id, usuario_id, qtd, criada_em) VALUES (42, 7, 1, NOW()); COMMIT; -- ou ROLLBACK se algo deu erro
▶ Running
Out[1]:
UPDATE 1 INSERT 0 1 COMMIT

⚛️ Atomicidade (A do ACID): se o INSERT falhar (ex: violação de FK), o UPDATE também é desfeito. Tudo ou nada. O WHERE disponivel > 0 ainda não resolve concorrência — só evita estoque negativo isolado.

SQL · Célula 2 · Níveis de Isolamento
SERIALIZABLE vs READ COMMITTED · o que cada um previne em concorrência
In [2]:
-- Sessão A · forçar isolamento mais estrito BEGIN; SET TRANSACTION ISOLATION LEVEL SERIALIZABLE; SELECT disponivel FROM espetaculos WHERE id = 42; UPDATE espetaculos SET disponivel = disponivel - 1 WHERE id = 42; COMMIT;
▶ Running
Out[2]:

⚖️ Trade-off: SERIALIZABLE evita anomalias mas pode lançar could not serialize access e exigir retry no app. READ COMMITTED é o default do Postgres — rápido, mas obriga você a usar locks explícitos (próximo slide). * Postgres trata READ UNCOMMITTED como READ COMMITTED. † REPEATABLE READ no PG já previne phantom (snapshot isolation).

SQL · Célula 3 · SELECT ... FOR UPDATE
Lock pessimista · trava a linha do estoque até o COMMIT — concorrência fica em fila
In [3]:
-- Sessão A (Ana) · pega lock e processa BEGIN; SELECT disponivel FROM espetaculos WHERE id = 42 FOR UPDATE; -- 🔒 trava a linha -- nesse instante a Sessão B fica esperando… UPDATE espetaculos SET disponivel = disponivel - 1 WHERE id = 42; INSERT INTO reservas (espetaculo_id, usuario_id, qtd) VALUES (42, 7, 1); COMMIT; -- 🔓 libera; Sessão B desbloqueia agora
▶ Running
Out[3]:
-- Sessão A BEGIN SELECT 1 (disponivel = 1) UPDATE 1 INSERT 0 1 COMMIT -- Sessão B (estava esperando…) SELECT 1 (disponivel = 0) ← agora vê estado pós-Ana ROLLBACK (app: "Esgotado, Bruno")

🗝️ Lock pessimista: a Sessão B só lê depois que A solta. Sem overbooking. Custo: throughput cai sob alta concorrência. Alternativa: FOR UPDATE NOWAIT (erra na hora) ou SKIP LOCKED (pula linhas travadas — usado em filas).

SQL · Célula 4 · Trigger de Auditoria
BEFORE UPDATE em reservas · grava OLD/NEW em reservas_audit automaticamente
In [4]:
CREATE OR REPLACE FUNCTION audit_reservas() RETURNS TRIGGER AS $$ BEGIN INSERT INTO reservas_audit (reserva_id, qtd_antes, qtd_depois, mudou_em, mudou_por) VALUES (OLD.id, OLD.qtd, NEW.qtd, NOW(), current_user); RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER trg_audit_reservas BEFORE UPDATE ON reservas FOR EACH ROW EXECUTE FUNCTION audit_reservas(); -- teste: UPDATE reservas SET qtd = 2 WHERE id = 101;
▶ Running
Out[4]:
CREATE FUNCTION CREATE TRIGGER UPDATE 1
SELECT * FROM reservas_audit ORDER BY mudou_em DESC LIMIT 1;
reserva_idqtd_antesqtd_depoismudou_emmudou_por
101122026-05-22 14:31:08app_user

🪤 Por que BEFORE UPDATE: a função roda dentro da mesma transação do UPDATE. Se o INSERT em reservas_audit falhar, o UPDATE original é desfeito — auditoria sempre consistente. Toda mudança em reserva passa a deixar rastro sem que o app precise saber.

SQL · Célula 5 · Trigger de Validação
BEFORE INSERT · dispara EXCEPTION se qtd > disponivel antes de tocar dados
In [5]:
CREATE OR REPLACE FUNCTION valida_reserva() RETURNS TRIGGER AS $$ DECLARE v_disponivel INT; BEGIN SELECT disponivel INTO v_disponivel FROM espetaculos WHERE id = NEW.espetaculo_id FOR UPDATE; IF NEW.qtd > v_disponivel THEN RAISE EXCEPTION 'Quantidade % excede disponível %', NEW.qtd, v_disponivel USING ERRCODE = '23514'; END IF; RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER trg_valida_reserva BEFORE INSERT ON reservas FOR EACH ROW EXECUTE FUNCTION valida_reserva(); -- teste · estoque tem 1, tentando comprar 3 INSERT INTO reservas (espetaculo_id, usuario_id, qtd) VALUES (42,9,3);
▶ Running
Out[5]:
CREATE FUNCTION CREATE TRIGGER ERROR: Quantidade 3 excede disponível 1 CONTEXT: PL/pgSQL function valida_reserva() line 9 at RAISE SQLSTATE: 23514

🛑 Bloqueio antes de tocar dados: o INSERT é abortado antes da linha entrar na tabela — não é necessário ROLLBACK manual. O app captura SQLSTATE 23514 e devolve 409 Conflict. Validação que mora no banco não pode ser pulada por bug no backend.

SQL · Célula 6 · Compra concorrente protegida
2 sessões + lock + trigger de auditoria + trigger de validação · resultado real
In [6]:
-- Sessão A (Ana) -- Sessão B (Bruno) BEGIN; BEGIN; SELECT * FROM espetaculos WHERE id = 42 FOR UPDATE; -- 🔒 A pega lock; B trava aqui INSERT INTO reservas (...) VALUES (42,7,1); → trg_valida_reserva: ok → trg_audit_reservas: log inserido UPDATE espetaculos SET disponivel = 0 WHERE id=42; COMMIT; -- 🔓 B desbloqueia, vê disponivel=0 INSERT INTO reservas (...) VALUES (42,9,1); → trg_valida_reserva: ERROR 23514 ROLLBACK;
▶ Running
Out[6]:
SELECT 1 INSERT 0 1 (Ana) UPDATE 1 COMMIT ERROR: Quantidade 1 excede disponível 0 (Bruno) ROLLBACK
✅ Compra atômica · 0 overbooking · 100% rastreabilidade no audit

🧱 A pilha que protege o projeto: 1) BEGIN/COMMIT dá atomicidade · 2) FOR UPDATE serializa a corrida · 3) BEFORE INSERT trigger valida regra invariante · 4) BEFORE UPDATE trigger registra auditoria. Backend recebe erro tratado e devolve mensagem amigável.

A

Trigger de auditoria

BEFORE/AFTER UPDATE em tabela crítica · grava OLD/NEW + usuário + timestamp em *_audit.

B
🛑

Trigger de validação

BEFORE INSERT/UPDATE · RAISE EXCEPTION ao violar regra invariante (ex: estoque, prazo, FK soft).

C
⚛️

Transação multi-tabela

BEGIN/COMMIT envolvendo 2+ tabelas · ROLLBACK em qualquer falha (ex: reserva + pagamento + log).

D
🗝️

Lock pessimista

SELECT FOR UPDATE em recurso disputado · prevenir race condition contra o último ingresso/vaga.

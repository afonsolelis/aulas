SQL · Célula 1 · CREATE FUNCTION simples
Function escalar · retorna ingressos disponíveis para um espetáculo
In [1]:
CREATE OR REPLACE FUNCTION disponibilidade_espetaculo(p_id INT) RETURNS INT AS $$ DECLARE v_disp INT; BEGIN SELECT disponivel INTO v_disp FROM ingressos WHERE espetaculo_id = p_id; RETURN COALESCE(v_disp, 0); END; $$ LANGUAGE plpgsql; SELECT disponibilidade_espetaculo(7) AS cirque;
▶ Running
Out[1]:
CREATE FUNCTION cirque -------- 42 (1 row)

📐 Leitura: function escalar · aparece em SELECT. Cirque tem 42 ingressos livres. É o tijolo mais simples — daqui em diante a coisa engrossa.

SQL · Célula 2 · Function retornando JSONB
Detalhes de espetáculo agregados em um único objeto JSON · ideal para resposta de API
In [2]:
CREATE OR REPLACE FUNCTION detalhes_espetaculo(p_id INT) RETURNS JSONB AS $$ DECLARE v JSONB; BEGIN SELECT jsonb_build_object( 'id', e.id, 'nome', e.nome, 'data', e.data, 'vagas', i.disponivel, 'preco', e.preco, 'reservas', (SELECT COUNT(*) FROM reservas WHERE espetaculo_id = e.id) ) INTO v FROM espetaculos e JOIN ingressos i ON i.espetaculo_id = e.id WHERE e.id = p_id; RETURN v; END; $$ LANGUAGE plpgsql; SELECT detalhes_espetaculo(7);
▶ Running
Out[2]:
{ "id": 7, "nome": "Cirque", "data": "2026-06-12", "vagas": 42, "preco": 180.00, "reservas": 27 }

💡 Por que JSONB? 1 chamada → 1 objeto pronto para a API REST. Em vez de 3 queries no backend (espetáculo + ingressos + count), o banco entrega o payload formatado. Quando o agregado é estável, faz sentido empurrar para cá.

SQL · Célula 3 · CREATE PROCEDURE transacional
reservar_ingresso · checa, atualiza, insere · 3 passos atômicos
In [3]:
CREATE OR REPLACE PROCEDURE reservar_ingresso( p_user INT, p_esp INT, p_qtd INT ) AS $$ DECLARE v_disp INT; BEGIN -- 1) checa disponibilidade SELECT disponivel INTO v_disp FROM ingressos WHERE espetaculo_id = p_esp FOR UPDATE; -- 2) atualiza estoque UPDATE ingressos SET vendidos = vendidos + p_qtd, disponivel = disponivel - p_qtd WHERE espetaculo_id = p_esp; -- 3) insere reserva INSERT INTO reservas (usuario_id, espetaculo_id, qtd) VALUES (p_user, p_esp, p_qtd); COMMIT; END; $$ LANGUAGE plpgsql; CALL reservar_ingresso(101, 7, 3);
▶ Running
Out[3]:
CREATE PROCEDURE CALL id | usuario_id | espetaculo_id | qtd ------+------------+---------------+----- 5004 | 101 | 7 | 3 (1 row inserted · disponivel: 42 → 39)

🔒 Atomicidade: os 3 passos rodam dentro da mesma transação · FOR UPDATE evita que duas requisições concorrentes leiam disponivel = 1 ao mesmo tempo e ambas reservem. Sem SP, esse lock seria responsabilidade do backend.

SQL · Célula 4 · Tratamento de exceção
RAISE EXCEPTION + ROLLBACK · o que acontece quando o estoque não dá
In [4]:
CREATE OR REPLACE PROCEDURE reservar_ingresso( p_user INT, p_esp INT, p_qtd INT ) AS $$ DECLARE v_disp INT; BEGIN SELECT disponivel INTO v_disp FROM ingressos WHERE espetaculo_id = p_esp FOR UPDATE; IF v_disp IS NULL OR v_disp < p_qtd THEN RAISE EXCEPTION 'insufficient_resources: % vagas pedidas, % disponiveis', p_qtd, COALESCE(v_disp,0) USING ERRCODE = '53400'; END IF; UPDATE ingressos SET vendidos = vendidos + p_qtd, disponivel = disponivel - p_qtd WHERE espetaculo_id = p_esp; INSERT INTO reservas(usuario_id,espetaculo_id,qtd) VALUES (p_user,p_esp,p_qtd); COMMIT; EXCEPTION WHEN insufficient_resources THEN ROLLBACK; RAISE; END; $$ LANGUAGE plpgsql; CALL reservar_ingresso(101, 9, 2); -- Bossa = 0 vagas
▶ Running
Out[4]:
ERROR: insufficient_resources: 2 vagas pedidas, 0 disponiveis SQLSTATE: 53400 CONTEXT: PL/pgSQL function reservar_ingresso(int,int,int) line 9 -- ✅ ROLLBACK executado · estado do banco preservado -- ✅ CALL reservar_ingresso(101, 7, 3) → sucesso (Cirque tem vagas)

🚨 SQLSTATE customizado: 53400 é mapeável no backend para HTTP 409 Conflict. Erro de domínio desce do banco até a UI sem perder semântica — RAISE + ROLLBACK é o contrato.

SQL · Célula 5 · recomendar_espetaculos(usuario, k)
RETURNS TABLE · top-k de espetáculos · ponte com a Aula 1/2
In [5]:
CREATE OR REPLACE FUNCTION recomendar_espetaculos( p_user INT, p_k INT DEFAULT 3 ) RETURNS TABLE(espetaculo_id INT, score FLOAT) AS $$ BEGIN RETURN QUERY WITH vistos AS ( SELECT DISTINCT espetaculo_id FROM reservas WHERE usuario_id = p_user ) SELECT e.id, ROUND(AVG(s.similaridade)::numeric, 2)::float AS score FROM espetaculos e JOIN similaridade s ON s.alvo = e.id WHERE e.id NOT IN (SELECT espetaculo_id FROM vistos) AND s.fonte IN (SELECT espetaculo_id FROM vistos) GROUP BY e.id ORDER BY score DESC LIMIT p_k; END; $$ LANGUAGE plpgsql; SELECT * FROM recomendar_espetaculos(101, 3);
▶ Running
Out[5]:
espetaculo_id | score ---------------+------- 8 | 4.18 9 | 3.42 11 | 2.95 (3 rows)
🎯 Top-3 para o usuário 101: Hamlet · Bossa · Pop
SQL · Célula 6 · Quando NÃO usar Stored Procedure
Comparativo objetivo · performance × manutenibilidade × testabilidade

-- benchmark: 10k chamadas · backend Node × stored procedure -- (medido no projeto Aventura · ambiente dev) EXPLAIN ANALYZE SELECT disponibilidade_espetaculo(7) FROM generate_series(1, 10000); -- repete o mesmo via API (loop no backend) -- compara: latencia_total, p99, lock_time
▶ Running
Out[6]:
abordagem | latencia_total | p99 | lock_time ------------------+----------------+--------+----------- SP (in-database) | 285 ms | 1.4ms | baixo Backend Node | 4720 ms | 12.8ms | alto Speedup SP: ~16.5x · economia: ~4.4s em 10k chamadas

✅ Use SP quando…

  • Regra de integridade crítica (atomicidade)
  • Operação quente (10k+ chamadas/min)
  • Cálculo fortemente acoplado aos dados

⛔ NÃO use SP quando…

  • Lógica de UI/apresentação
  • Regra que muda toda semana
  • Equipe não tem PL/pgSQL no fluxo de review

📌 Critério: SP para regra crítica + atomicidade · backend para lógica de UI e regra instável. Sempre versione com migration.

CENÁRIO A
🔒

Validação de regra crítica

Procedure que encapsula uma operação atômica do domínio (reservar, transferir, encerrar). Inclui EXCEPTION e ROLLBACK.

CENÁRIO B
📊

Cálculo agregado pesado

Function que retorna métrica (faturamento por evento, taxa de ocupação, ranking de vendedores). RETURNS TABLE ou JSONB.

CENÁRIO C
🧰

Função utilitária reutilizável

Function pura: sanitizar CPF, calcular idade, normalizar string. Usada em vários lugares por SELECT.

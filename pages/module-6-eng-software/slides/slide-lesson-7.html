Modelagem Dinâmica e Estática
O contrato da arquitetura antes da codificação. Não se programa IA no escuro.
🏛️
Modelagem Estática
Define o que existe. Diagramas de classe, entidades, atributos. É a estrutura base da nossa arquitetura SOA.
⚡
Modelagem Dinâmica
Define como as coisas interagem. Diagramas de sequência, fluxos de caso de uso. Revela as regras de negócio reais.
📜
O Contrato
Na arquitetura SOA, os serviços se comunicam via contratos. A modelagem é o rascunho visual desses contratos inquebráveis.
🤖
IA vs Arquitetura
Se a IA codar sem limites, ela cria atalhos e quebra o SOA. A modelagem precisa virar uma restrição em código.

💡 Ponte para TDD: Como transformar um diagrama UML (estático/dinâmico) em algo que a IA seja obrigada a respeitar na hora de gerar o código? É aqui que entram os testes unitários.

TDD como Guardrail para IA
Não deixe o LLM codar solto. Use os testes como barreira (guardrail).
1
Contrato Rigoroso
O teste unitário define a assinatura das classes, métodos e injeção de dependências.
Se a IA tentar "pular" o repositório e ir direto ao banco, o teste quebra.
2
Regras de Negócio
As métricas vitais da sua aplicação (Score, Juros, Permissões) ficam protegidas.
O Assert valida o Output Exato. Alucinações matemáticas da IA são pegas na hora.
3
Desenvolvimento Direcionado
Ao invés de pedir: "Faça um recomendador", você pede: "Faça o teste passar".
O escopo da IA diminui absurdamente. Ela fica focada no *Green*.
4
Barra Verde Contínua
Se o AIOX quebrar algo do SOA durante a geração, você faz revert na hora.
git reset --hard é seu melhor amigo.

🧠 O novo fluxo de Engenharia: O humano faz o trabalho nobre (Pensar a modelagem e definir os contratos/testes). A IA faz o trabalho braçal (Escrever o miolo da classe para os testes passarem).

🔴 Red · escreva o teste primeiro

Teste deve falhar. Se passa de cara, é teste tautológico ou código já existia. Failure message tem que ser claro.

🟢 Green · código mínimo para passar

Use o caminho mais ridículo possível. Hard-code, return fixo, "fake it till you make it" — tudo permitido. Só não pode quebrar a barra verde.

🔵 Refactor · limpe sem quebrar

Renomear, extrair método, remover duplicação. dotnet test deve continuar verde após cada mudança micro.

📐 Regra dos 3 (Beck)

Tolere duplicação até a 3ª ocorrência — aí extraia. Antes disso, é cedo demais para abstrair.

xUnit · Anatomia de um teste · AAA
Arrange · Act · Assert — 1 cenário por teste, nome que vira documentação
Test.cs
using Xunit; using System.Collections.Generic; public class CalculadoraScoreTests { [Fact] public void DeveRetornarZeroQuandoUsuarioNuncaInteragiu() { // Arrange — monta o SUT (system under test) e os dados var sut = new CalculadoraScore(); var historico = new List<Interacao>(); // Act — chama exatamente UMA ação var score = sut.Calcular(historico); // Assert — verifica UMA propriedade do resultado Assert.Equal(0, score); } }

🎯 Arrange (Monta o Palco)

Obriga a IA a usar as dependências certas. Ex: Injeção de repositórios Mock.

⚡ Act (Ação)

Define a assinatura do método. A IA não pode inventar rotas alternativas.

🛡️ Assert (A Trava)

Garante a regra de negócio exata. É o guardrail que impede lógicas quebradas.

🔴 Red · O humano escreve o Contrato
A IA não cria a regra. VOCÊ traduz o modelo para um teste vermelho (xUnit).
Red
// RecomendadorItemBasedTests.cs using Xunit; public class RecomendadorItemBasedTests { [Fact] public void DeveRetornarTopNItensQuandoUsuarioTemHistorico() { var sut = new RecomendadorItemBased(); var historico = new List<Interacao> { /* ... */ }; var top3 = sut.Recomendar(historico, n: 3); Assert.NotNull(top3); Assert.Equal(3, top3.Count); } }
$ dotnet test
RecomendadorItemBasedTests.cs(8,29): error CS0246: The type or namespace 'RecomendadorItemBased' could not be found. Build FAILED. ❌

✅ Isso é o Guardrail: A classe nem existe ainda, mas a regra já está travada. O compilador (e os testes falhando) ditam exatamente o que o AIOX terá que fazer na etapa Green.

🤖 IA Focada

Como o escopo foi fechado pelo teste, a IA não tenta reinventar a roda ou mudar a arquitetura.

🔌 Inversão de dep.

O AIOX lê que o teste usou mock e já sabe que deve injetar via construtor.

🟢 Barra verde

O ciclo só para quando o dotnet test roda limpo e a IA finaliza o trabalho sozinha.

📚 Red → AIOX → Green: A refatoração contínua é delegada, mas os contratos e arquitetura são definidos por VOCÊ.

Mocks · Stubs · Fakes
Os 3 dublês de teste · em C#: Moq e NSubstitute resolvem 95% dos casos
🎭 Mock

Espia se um método foi chamado e como. Verifica comportamento (interação).

Use quando: side-effect importa (envio de email, log).
📥 Stub

Devolve resposta pré-fabricada. Alimenta o SUT sem verificar nada.

Use quando: você só precisa de um dado de entrada.
🪛 Fake

Implementação real, mas leve. Ex.: repositório em memória que substitui o banco.

Use quando: integração precisa de comportamento, não só retorno fixo.
Moq
using Moq; [Fact] public void DeveBuscarHistoricoNoRepositorio() { // Arrange · cria mock e configura retorno var repo = new Mock<IRecomendacaoRepository>(); repo.Setup(r => r.PorUsuario(It.IsAny<int>())) .Returns(new List<Interacao> { new(Tipo.Like, 42) }); var sut = new RecomendadorItemBased(repo.Object); // Act sut.RecomendarPara(usuarioId: 7); // Assert · verifica QUE foi chamado repo.Verify(r => r.PorUsuario(7), Times.Once()); }

TAREFA
🎯

Classe CalculadoraScoreUsuario

Recebe lista de Interacao (Like, View, Compra com pesos diferentes) e devolve um score em float. Pesos: View=1, Like=3, Compra=10.

TESTES
🧪

≥ 5 testes xUnit

Cobrindo: histórico vazio · só views · só likes · mix dos 3 · peso de compra > outros. Cada [Fact] = 1 guardrail.

AIOX
🤖

Humano Red, AIOX Green

A dupla foca 100% em criar a suíte de testes perfeita (Contrato). A skill dev do AIOX deve rodar para passar o teste.

📋 Restrição: a IA não pode tocar nas classes existentes do projeto. Crie testes que forcem injeção de dependência via interfaces se precisar.

📦 Entrega: Merge Request no repositório do grupo. Deve conter os logs da interação com o AIOX QA e AIOX Dev no descritivo.

⭐ Excelência: cobertura > 90% medida com coverlet, screenshot anexado no MR. Bônus para par que entregar com [Theory] + [InlineData] bem usado.

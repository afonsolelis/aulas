POO em 1 slide · 4 pilares
Revisão relâmpago — base para entender SOLID e por que TDD pede objetos pequenos
🔒
Encapsulamento
Esconde estado interno; exposição só por métodos públicos. Em C#: private + public properties.
🧬
Herança
Classe filha reaproveita base. Use com parcimônia — composição costuma ser melhor que herança profunda.
🎭
Polimorfismo
Mesmo método, comportamentos diferentes. virtual/override e interfaces — chave para mocks.
🪞
Abstração
Esconde o "como". Interfaces e classes abstratas expõem contratos — testa-se contra o contrato, não a implementação.

💡 Ponte para TDD: abstração e polimorfismo permitem injetar dependências por interface — e isso é o que torna possível substituir o mundo real por mocks nos testes unitários.

SOLID rapidamente · 5 princípios
1 frase + 1 mini-exemplo C# por princípio · objetos pequenos = testes pequenos
S
Single Responsibility
Uma classe, uma razão para mudar. Se ela faz 3 coisas, vira 3 classes.
CalculadoraScore só calcula score · não lê do banco
O
Open/Closed
Aberto para extensão, fechado para modificação. Acrescente novas regras sem reabrir código testado.
interface IRegraScore → novas regras viram novas classes
L
Liskov Substitution
Filhos honram contratos do pai — não joguem exceção onde a base não esperaria.
RegraCompra : IRegraScore deve respeitar Aplicar(historico)
I
Interface Segregation
Muitas interfaces específicas valem mais que uma gigante. Ninguém implementa o que não usa.
ILeitor + IEscritor > IRepositorio com 20 métodos
D
Dependency Inversion
Dependa de abstração, não de implementação. Recebe a interface no construtor — fica trivial mockar.
Calculadora(IRepositorio repo) — Moq.Setup() fácil

🧠 SOLID em 1 frase: classes pequenas, interfaces pequenas e dependências por construtor — esse é o atalho para TDD que não machuca.

🔴 Red · escreva o teste primeiro

Teste deve falhar. Se passa de cara, é teste tautológico ou código já existia. Failure message tem que ser claro.

🟢 Green · código mínimo para passar

Use o caminho mais ridículo possível. Hard-code, return fixo, "fake it till you make it" — tudo permitido. Só não pode quebrar a barra verde.

🔵 Refactor · limpe sem quebrar

Renomear, extrair método, remover duplicação. dotnet test deve continuar verde após cada mudança micro.

📐 Regra dos 3 (Beck)

Tolere duplicação até a 3ª ocorrência — aí extraia. Antes disso, é cedo demais para abstrair.

xUnit · Anatomia de um teste · AAA
Arrange · Act · Assert — 1 cenário por teste, nome que vira documentação
Test.cs
using Xunit; using System.Collections.Generic; public class CalculadoraScoreTests { [Fact] public void DeveRetornarZeroQuandoUsuarioNuncaInteragiu() { // Arrange — monta o SUT (system under test) e os dados var sut = new CalculadoraScore(); var historico = new List<Interacao>(); // Act — chama exatamente UMA ação var score = sut.Calcular(historico); // Assert — verifica UMA propriedade do resultado Assert.Equal(0, score); } }

🎯 [Fact]

Cenário fixo. Para cenários parametrizados use [Theory] + [InlineData].

📛 Nome = documentação

Padrão: Deve<esperado>Quando<condição>. Lê-se o nome e sabe-se o que é.

⚙️ dotnet test

CLI roda tudo. --filter roda 1 teste. coverlet.collector mede cobertura.

🔴 Red · escreva o teste antes do código
Cenário do projeto: classe RecomendadorItemBased — teste vermelho que nem compila
Red
// RecomendadorItemBasedTests.cs using Xunit; public class RecomendadorItemBasedTests { [Fact] public void DeveRetornarTopNItensQuandoUsuarioTemHistorico() { var sut = new RecomendadorItemBased(); var historico = new List<Interacao> { /* ... */ }; var top3 = sut.Recomendar(historico, n: 3); Assert.NotNull(top3); Assert.Equal(3, top3.Count); } }
$ dotnet test
RecomendadorItemBasedTests.cs(8,29): error CS0246: The type or namespace 'RecomendadorItemBased' could not be found. Build FAILED. ❌

✅ Isso é Red bom: a classe nem existe ainda. O compilador é seu primeiro test runner. Próximo passo: criar o mínimo para fazer compilar e quebrar com mensagem clara.

🟢 Green · código mínimo para passar
"Fake it till you make it" — sem otimização, sem feature extra. Só passa o teste.
Green
// RecomendadorItemBased.cs · primeira implementação public class RecomendadorItemBased { public List<Item> Recomendar(List<Interacao> historico, int n) { // Sim, isto é proposital. O teste só checa Count == 3. return new List<Item> { new Item(1), new Item(2), new Item(3) }; } }
$ dotnet test
Passed! - Failed: 0, Passed: 1, Skipped: 0 Total time: 1.2s ✅

📐 Por que esse código "feio" é certo: o próximo teste vai exigir ranking real, e aí o hard-code quebra. Você só implementa o que algum teste exige. Triangulação leva o código ao genérico.

🔵 Refactor · limpe sem quebrar a barra verde
Após 3+ testes verdes: extrair método, renomear, eliminar duplicação · Beck + Fowler
Refactor
// depois de 4 testes verdes — extrai estratégia public class RecomendadorItemBased { private readonly ICalculadoraSimilaridade _sim; public RecomendadorItemBased(ICalculadoraSimilaridade sim) => _sim = sim; // DI · pronto para Moq public List<Item> Recomendar(List<Interacao> historico, int n) { var candidatos = _sim.Ranquear(historico); return candidatos.Take(n).ToList(); } }

✂️ Extract method

Lógica de ranking saiu para ICalculadoraSimilaridade. Cada classe tem 1 razão de mudar (SRP).

🔌 Inversão de dep.

Construtor recebe interface — Moq consegue substituir em 1 linha no teste.

🟢 Barra verde

dotnet test verde a cada micro-mudança. Se quebrar, git reset --hard e tenta menor.

📚 Beck + Fowler: "Refactoring is the process of changing a software system in such a way that it does not alter the external behavior of the code yet improves its internal structure." — testes verdes são a rede que torna isso possível.

Mocks · Stubs · Fakes
Os 3 dublês de teste · em C#: Moq e NSubstitute resolvem 95% dos casos
🎭 Mock

Espia se um método foi chamado e como. Verifica comportamento (interação).

Use quando: side-effect importa (envio de email, log).
📥 Stub

Devolve resposta pré-fabricada. Alimenta o SUT sem verificar nada.

Use quando: você só precisa de um dado de entrada.
🪛 Fake

Implementação real, mas leve. Ex.: repositório em memória que substitui o banco.

Use quando: integração precisa de comportamento, não só retorno fixo.
Moq
using Moq; [Fact] public void DeveBuscarHistoricoNoRepositorio() { // Arrange · cria mock e configura retorno var repo = new Mock<IRecomendacaoRepository>(); repo.Setup(r => r.PorUsuario(It.IsAny<int>())) .Returns(new List<Interacao> { new(Tipo.Like, 42) }); var sut = new RecomendadorItemBased(repo.Object); // Act sut.RecomendarPara(usuarioId: 7); // Assert · verifica QUE foi chamado repo.Verify(r => r.PorUsuario(7), Times.Once); }

TAREFA

Classe CalculadoraScoreUsuario

Recebe lista de Interacao (Like, View, Compra com pesos diferentes) e devolve um score em float. Pesos: View=1, Like=3, Compra=10.

TESTES
🧪

≥ 5 testes xUnit

Cobrindo: histórico vazio · só views · só likes · mix dos 3 · peso de compra > outros. Cada [Fact] = 1 cenário claro.

CICLO
🔁

Commits Red/Green/Refactor

Cada commit = 1 passo do ciclo. git log deve mostrar a sequência: red: ... → green: ... → refactor: ...

📋 Restrição: a implementação não pode tocar nas classes existentes do projeto. Use injeção de dependência caso precise consultar outros componentes — receba interfaces no construtor.

📦 Entrega: Merge Request no repositório do grupo, com 5+ commits visíveis seguindo Red/Green/Refactor + README curto na pasta com instrução de dotnet test.

⭐ Excelência: cobertura > 90% medida com coverlet, screenshot anexado no MR. Bônus para par que entregar com [Theory] + [InlineData] bem usado.

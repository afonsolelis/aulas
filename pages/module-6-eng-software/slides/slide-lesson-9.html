♻️ Idempotência · porque toda chamada vai falhar e ser reenviada
Em rede, tudo eventualmente falha · retry é obrigatório · idempotência impede duplicar pagamento, reserva, e-mail

😱 Cenário típico de produção

Cliente envia POST /pagamentos, servidor processa, mas a resposta nunca chega (timeout). Cliente faz retry. Resultado: 2 cobranças.

Lição: não dá para confiar em "tentei só uma vez". Toda operação de escrita precisa ser idempotente.

🔑 A solução: Idempotency-Key

Cliente gera UUID único, manda no header. Servidor guarda (key → resposta). Retry com a mesma key devolve a resposta original, sem reprocessar.

Bonus: em SQL, INSERT ... ON CONFLICT DO UPDATE (UPSERT) é idempotência no banco.

HTTP
# 1ª tentativa — pagamento processado POST /pagamentos HTTP/1.1 Host: api.aventura.com Content-Type: application/json Idempotency-Key: "a3f1c2d4-7b8e-4f1a-9c0d-2e5b1a8f6c7d" { "valorCentavos": 15000, "metodo": "pix" } # Timeout no cliente — retry com a MESMA key # Server vê key já processada → devolve a resposta original (HTTP 200) # Resultado: 1 cobrança, não 2 ✅
⚡ Resiliência · 4 padrões essenciais
Circuit Breaker · Retry · Timeout · Bulkhead · falham por design para o sistema sobreviver
Closed
OK · passa
Chamadas fluem normal. Conta falhas.
→
Open
FAIL FAST
5 falhas → bloqueia. Devolve erro em 1ms.
→
Half-Open
TESTE
Após 30s testa 1 chamada. Sucesso → Closed.
C#
// Circuit Breaker com Polly var policy = Policy .Handle<HttpRequestException>() .CircuitBreakerAsync( handledEventsAllowedBeforeBreaking: 5, durationOfBreak: TimeSpan.FromSeconds(30)); await policy.ExecuteAsync(async () => await http.GetAsync("/recommendations"));

⏱️ Timeout

Toda chamada de rede tem timeout explícito. Sem timeout, 1 serviço lento congela todos os threads do chamador.

🔁 Retry com backoff

Falhou? Espera 2^n segundos com jitter. Idempotência é pré-requisito (slide 6).

🛟 Bulkhead

Pool de threads/conexões separado por dependência. Falha em 1 não afoga as outras (analogia: compartimentos do navio).

🧠 Combinar os 4

Polly compõe: Timeout + Retry + CircuitBreaker + Bulkhead em 1 pipeline.

🐳 Containers e orquestração
Docker resolve "funciona na minha máquina" · Kubernetes resolve "tem que rodar 24/7 com 50 réplicas"

🐳 Por que Docker

  • Paridade dev/prod (12-factor X)
  • Distribuição · 1 imagem, todos os ambientes
  • Isolamento · processo + filesystem + rede
  • Imutabilidade · não muda em produção

☸️ O que Kubernetes resolve

  • Scheduling · onde cada pod roda
  • Self-healing · pod morto → reinicia
  • Autoscaling · HPA por CPU/RAM/custom
  • Service discovery · DNS interno + load balance
DOCKER
# Multi-stage Dockerfile · API .NET 8 FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS base WORKDIR /app EXPOSE 8080 FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build WORKDIR /src COPY . . RUN dotnet publish -c Release -o /out FROM base AS final COPY --from=build /out . ENTRYPOINT ["dotnet","Recommendation.Api.dll"]

💡 Multi-stage: imagem final só com runtime (~200 MB) — sem SDK, sem compilador, sem código-fonte. Menor superfície de ataque, deploy mais rápido.

📈 Autoscaling e load balancing
Crescer e encolher conforme a demanda · pagar pelo que usa · sobreviver à Black Friday

📊 HPA · Horizontal Pod Autoscaler

K8s adiciona/remove pods baseado em CPU, RAM ou métrica customizada (RPS, queue depth).

Min: 2 réplicas · Max: 50 · Target: 70% CPU

📨 Scale by queue depth

Padrão maduro: app lê de fila SQS/Service Bus, autoscaler observa profundidade.

100 mensagens = 1 worker · 10k = 50 workers. Pico absorvido sem perder request.

🌐 Load Balancer L4 vs L7

NLB (TCP) — barato, rápido. ALB/Ingress (HTTP) — roteia por path/host, TLS termination, WAF.

YAML
# HPA Kubernetes · escala 2 → 20 réplicas mantendo CPU em 70% apiVersion: autoscaling/v2 kind: HorizontalPodAutoscaler metadata: name: recommendation-api spec: scaleTargetRef: apiVersion: apps/v1 kind: Deployment name: recommendation-api minReplicas: 2 maxReplicas: 20 metrics: - type: Resource resource: name: cpu target: { type: Utilization, averageUtilization: 70 }
🚪 API Gateway · 🕸️ Service Mesh
Gateway cuida da borda (cliente → backend) · Mesh cuida do meio (serviço ↔ serviço)

🚪 API Gateway · Edge / North-South

EXTERNO

Porta única para clientes externos. 1 hop entre app mobile e o ecossistema de serviços.

  • AuthN/AuthZ — JWT, OAuth, API key
  • Rate limit — 100 req/min por usuário
  • Roteamento — /api/users/* → user-svc
  • TLS termination · agregação de respostas
  • Cache de respostas idempotentes

Implementações: AWS API Gateway, Azure API Management, Kong, NGINX, Tyk.

🕸️ Service Mesh · East-West

INTERNO

Sidecar proxy em cada pod. Comunicação serviço ↔ serviço dentro do cluster.

  • mTLS automático entre pods
  • Retry / Circuit Breaker sem mudar código
  • Observabilidade · trace distribuído out-of-the-box
  • Traffic shifting · canary, blue-green
  • Policy · quem pode chamar quem

Implementações: Istio, Linkerd, AWS App Mesh, Consul Connect.

🤝 Convivem: Gateway na borda + Mesh interno = arquitetura SOA cloud-native completa. Gateway protege o cliente. Mesh protege os serviços entre si.

🔐 Configurações sensíveis · NUNCA no código
Vault / Secrets Manager / Key Vault · injetados como variável de ambiente · rotação automática

🚫 Antipadrões

  • appsettings.json com senha commitado
  • .env em repo público
  • Senha hardcoded em string
  • Compartilhada em chat/e-mail

✅ Cofres gerenciados

  • AWS Secrets Manager
  • Azure Key Vault
  • GCP Secret Manager
  • HashiCorp Vault (multi-cloud)

🔄 Rotação automática

Secret expira a cada 30/60/90 dias. Cofre gera nova senha, atualiza no banco/serviço, app pega via API. Zero intervenção humana.

YAML
# Kubernetes · injetando secret do AWS Secrets Manager via External Secrets apiVersion: external-secrets.io/v1beta1 kind: ExternalSecret metadata: { name: db-credentials } spec: refreshInterval: "1h" # pega valor novo a cada hora secretStoreRef: { name: aws-vault, kind: SecretStore } target: { name: db-creds } data: - secretKey: ConnectionStrings__Default remoteRef: { key: prod/aventura/db, property: connection_string } # No container, vira: env var ConnectionStrings__Default ✅
💰 Custos cloud · FinOps básico
Cloud é barata até a primeira fatura · entender pricing model evita surpresa de R$ 50k no fim do mês

💵 3 dimensões de custo

  • Compute — CPU/RAM/hora (EC2, App Service)
  • Storage — GB/mês (S3, Blob, EBS)
  • Egress — saída de dados ⚠️ caríssimo

💸 Spot / Reserved / Savings

  • Spot · até 90% off · pode ser desligada
  • Reserved 1y/3y · 30-60% off
  • Savings Plan · flexível, comprometimento de gasto

🚨 Alertas e governança

  • Budget alerts em 50/80/100% do orçamento
  • Tagging obrigatório (env, team, project)
  • Auto-shutdown de dev fora do horário

🔍 Ferramentas de observabilidade de custo

AWS Cost Explorer · Azure Cost Management · GCP Billing Reports · todas com breakdown por tag, recurso, conta. Dashboard semanal é hábito de equipe madura.

⚖️ FinOps = engenharia + finanças

Cultura: dev pensa em custo no design (não só em runtime). RNF de custo entra junto com latência e disponibilidade. "Custo é métrica de qualidade."

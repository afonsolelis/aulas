🟢 Base · Unit Tests

Centenas de testes rodando em milissegundos. Garantem que cada peça funciona isolada. Falhas de unit tests não pegam wiring.

🟡 Meio · Integration Tests

Dezenas, não centenas. Sobem a aplicação de verdade para validar wiring entre camadas. Foco do dia.

🔴 Topo · End-to-End

Poucos casos críticos de fluxo (login → checkout). Caros de manter, frágeis a mudanças de UI. Não substituem integração.

Por que Testes de Integração? 🔌
O que o teste unitário sozinho NÃO consegue capturar

🔗 Wiring entre camadas

Controller chama Service que chama Repository. Cada peça passa nos seus unit tests, mas a injeção de dependência ou o middleware podem estar quebrados.

🗃️ Contratos com o BD real

EF Core gera SQL diferente de mocks. Migrations, índices, tipos NUMERIC × DECIMAL: tudo só aparece quando você roda contra um banco de verdade.

🔐 Auth, cache, middleware

JWT, claims, políticas de autorização, headers de cache. Tudo isso só executa quando o pipeline HTTP roda inteiro — algo que unit tests pulam.

🧱 Mapeamento ORM

Relations, lazy loading, navigation properties: erros típicos de configuração só aparecem quando uma query real roda — não em mocks.

1 bug de integração custa 10× o que custa um bug unitário — porque ele só aparece quando várias camadas conversam, e isolar a causa exige rodar tudo junto. — Princípio de custo de defeito · adaptado de Boehm e do "shift left"
WebApplicationFactory · ASP.NET Core
Sobe a app inteira em-process · troca BD por SQLite InMemory · sem servidor real
C#
public class CustomWebAppFactory : WebApplicationFactory<Program> { protected override void ConfigureWebHost(IWebHostBuilder builder) { builder.ConfigureServices(services => { // troca BD real por SQLite InMemory var desc = services.Single(d => d.ServiceType == typeof(DbContextOptions<AppDb>)); services.Remove(desc); services.AddDbContext<AppDb>(o => o.UseSqlite("DataSource=:memory:")); }); } }

🏗️ Em-process

Roda dentro do processo do teste. Não precisa abrir porta nem subir dotnet run.

🔁 Override de DI

Substitui qualquer serviço registrado: BD, e-mail, cache. Mantém o resto idêntico.

⚡ HttpClient pronto

factory.CreateClient() devolve um cliente que fala com a app real, sem rede.

Cenário de Sucesso ✅
Teste integrado de GET · valida fluxo feliz da API de recomendações
C#
public class RecomendacaoEndpointsTests : IClassFixture<CustomWebAppFactory> { private readonly HttpClient _client; public RecomendacaoEndpointsTests(CustomWebAppFactory f) { _client = f.CreateClient(); } [Fact] public async Task DeveRetornarTopRecomendacoesParaUsuario() { var r = await _client.GetAsync("/api/recomendacoes/42?k=5"); r.EnsureSuccessStatusCode(); var lista = await r.Content.ReadFromJsonAsync<List<ItemDto>>(); Assert.NotNull(lista); Assert.Equal(5, lista!.Count); } }
PASS
Passed RecomendacaoEndpointsTests.DeveRetornarTopRecomendacoesParaUsuario · 312 ms · ✅ 5 itens retornados, contrato OK
Cenário de Erro / Exceção ⚠️
Teste integrado de POST com payload inválido · espera 400 + body de erro
C#
[Fact] public async Task DeveRetornar400QuandoPayloadInvalido() { // k negativo viola a regra de negócio var payload = new { usuarioId = 42, k = -3 }; var resp = await _client.PostAsJsonAsync( "/api/recomendacoes", payload); Assert.Equal(HttpStatusCode.BadRequest, resp.StatusCode); var erro = await resp.Content.ReadFromJsonAsync<ProblemDetails>(); Assert.NotNull(erro); Assert.Contains("k deve ser positivo", erro!.Detail); }

🎯 Status code primeiro

Antes de inspecionar o body, garanta o StatusCode. Se vier 500, o problema é outro (a regra nem chegou a ser checada).

📨 Mensagem de erro

ProblemDetails é o padrão (RFC 7807). O texto do Detail deve refletir a regra que foi violada — algo que o front pode mostrar.

Banco de Dados em Testes 🗃️
Estratégias para isolar dados · do mais leve ao mais fiel

🪶 SQLite InMemory

Mais leve. DataSource=:memory:. Roda em ms, mas não fala dialetos avançados (JSONB, CTE recursiva). Ótimo para validar wiring de queries simples.

🐳 Testcontainers (Docker)

Sobe um PostgreSQL real em container por suite. Fiel ao prod. Mais lento (~5–10s startup), mas pega bugs específicos do dialeto.

🧹 Respawn

Reseta o BD entre testes sem precisar dropar e recriar. Mais rápido que migrations a cada teste. Garante isolamento.

🌱 Seed determinístico + Faker / Bogus

Use Bogus com seed fixo para gerar dados realistas e reprodutíveis. Teste que falha às vezes é teste que ninguém confia.

📌 Regra prática: SQLite InMemory para a maioria · Testcontainers para os fluxos que dependem de PostgreSQL real (full-text, JSONB, transações isoladas).

CI/CD · Integração no Pipeline 🚦
Testes que rodam em todo PR · bloqueiam merge quando falham
YML
# .gitlab-ci.yml — exemplo enxuto test: stage: test image: mcr.microsoft.com/dotnet/sdk:8.0 services: - postgres:16 script: - dotnet restore - dotnet test --logger "trx;LogFileName=tests.trx" rules: - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'

🔒 Bloqueio de merge

Configurar "all checks must pass" nas branch protection rules. Falhou? Não merge.

⏱️ Tempo < 5 min

Suite que demora demais é suite que vira flaky e que ninguém roda local. Otimize.

03
📋

Documentação do RF (Regra de Negócio)

Descrição do MR cita qual RF (do artefato de requisitos do grupo) o teste valida.

⭐ Excelência: rastreabilidade exata (ex.: "RF05") + por que previne falha crítica.

04
🎚️

Documentação do RNF (ISO 25010)

Indicar e justificar se o fluxo atende/impacta um eixo: eficiência, segurança, confiabilidade.

⭐ Excelência: citar código exato ("RNF02") do artefato.

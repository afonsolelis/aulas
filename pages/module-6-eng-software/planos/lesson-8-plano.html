Resumo

A aula 8 sobe um nível na escada de testes. Apresentamos a pirâmide (unitário → integração → E2E) e por que ela é piramidal e não invertida. Trabalhamos testes de integração com banco real usando Testcontainers (banco efêmero por suíte), testes de API com Supertest e o conceito de contract testing entre serviços SOA.

O fechamento aborda os anti-padrões frequentes: testes que não testam (mock everything), testes flaky por dependência de tempo/ordem, testes lentos que ninguém roda.

Objetivos de aprendizagem

  • OA-1Justificar a pirâmide de testes em vez do "ice cream cone".
  • OA-2Escrever teste de integração API + banco com Supertest.
  • OA-3Configurar banco efêmero por suíte com Testcontainers.
  • OA-4Implementar contract test entre dois serviços SOA.
  • OA-5Identificar testes flaky e lentos, propondo refatoração.
  • OA-6Decidir quando integração real vale a pena vs. doubles.

Pré-requisitos

  • Aula 7 concluída: classes de domínio com testes unitários verdes.
  • Docker funcionando localmente (para Testcontainers).
  • API e banco do projeto na develop.

Estratégias de metodologia ativa

  • Pirâmide como contrato — toda nova funcionalidade entra com unit + integração; E2E é exceção.
  • Banco real, descartável — Testcontainers tira o medo de "vou sujar meu banco" e dá confiança real.
  • Flaky como pauta — testes intermitentes são tratados em sala como bug, não como inconveniente.
Princípio orientador

Mock pesado é cheiro de design ruim do código sob teste. Se você precisa mockar 7 dependências para testar 1 método, esse método tem responsabilidades demais.

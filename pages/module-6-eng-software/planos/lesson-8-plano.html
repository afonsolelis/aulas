Resumo

A aula 8 traz o tema Testes de Integração aplicado à API .NET que cada grupo está construindo para o parceiro de mercado. A exposição (30 min) cobre a pirâmide de testes, por que integração captura defeitos que o unitário não captura, a anatomia de uma WebApplicationFactory e os blocos típicos de cenários de sucesso e de erro.

Os 90 minutos seguintes são dedicados a uma atividade ponderada individual: cada aluno escolhe um fluxo dos services do projeto do grupo, escreve testes integrados (1 sucesso + 1 erro), abre um Merge Request no repositório oficial, preenche a descrição com rastreabilidade a um RF e a um eixo de RNF da ISO 25010, e cola o link do MR na plataforma da disciplina.

Objetivos de aprendizagem

  • OA-1Posicionar testes de integração na pirâmide e justificar sua razão de existir frente a unit tests e E2E.
  • OA-2Configurar uma WebApplicationFactory<Program> para subir a API em-process e substituir dependências (BD, e-mail, cache).
  • OA-3Escrever um teste de cenário de sucesso e um teste de cenário de erro/exceção para um fluxo dos services.
  • OA-4Decidir a estratégia de banco de dados em testes (SQLite InMemory, Testcontainers, Respawn) adequada ao caso.
  • OA-5Entregar o trabalho via Merge Request com pipeline verde, descrição rastreando RF e justificando RNF (ISO 25010).
  • OA-6Justificar como o teste implementado previne uma falha crítica no negócio do parceiro.

Pré-requisitos

  • API .NET do grupo já com pelo menos um fluxo de services implementado e endpoint correspondente exposto.
  • Repositório do grupo configurado, com pipeline executando dotnet test em todo MR.
  • Artefato de requisitos do grupo (RF, RNF) versionado — com cada item identificado por código (ex.: RF05, RNF02).
  • Material da aula 8 lido + 1 fluxo da API mapeado no caderno de bordo.

Estratégias de metodologia ativa

  • MR como entrega real — a avaliação é feita lendo o link de um Merge Request no repositório do grupo, não um arquivo solto. O artefato é o mesmo que será revisado em produção.
  • Rastreabilidade obrigatória — toda descrição de MR cita um RF do artefato de requisitos e justifica um eixo de RNF da ISO 25010. Não é opcional: o teste só faz sentido se conectado à regra de negócio.
  • Pipeline verde como gate — não basta o código compilar; dotnet test precisa estar verde no pipeline do repositório. Falha técnica não é confusão de processo: é defeito.
Princípio orientador

Teste de integração não é luxo — é o contrato técnico que protege as regras de negócio. Sem ele, qualquer alteração no código vira aposta.

Verificação de aprendizagem · Critérios da Ponderada

  • CR-1Implementação do fluxo de teste: usando xUnit/NUnit/MSTest, há 1 cenário de sucesso e 1 cenário de erro/exceção para um fluxo dos services. dotnet test verde no pipeline.
  • CR-2Entrega via Merge Request: submetido como MR no repositório oficial do grupo (Open ou Merged/Closed), em branch test/integration-{flow}.
  • CR-3Documentação do RF: descrição do MR cita qual regra de negócio (RF do artefato de requisitos do grupo) o teste valida. ⭐ Excelência: rastreabilidade exata (ex.: "RF05") + por que o teste previne uma falha crítica.
  • CR-4Documentação do RNF: indica e justifica se o fluxo atende ou impacta um eixo de RNF da ISO 25010 (eficiência de desempenho, segurança, confiabilidade). ⭐ Excelência: citar o código exato do RNF do artefato (ex.: "RNF02").
  • CR-5Submissão: link direto do MR colado na plataforma da disciplina antes do fim dos 90 min.
Critério mínimo

Aluno que sair às 16h15 sem MR aberto + descrição com RF e RNF + pipeline verde está em débito técnico para a aula 9 (boas práticas SOA na nuvem).

Conexão com a aula 9

A aula 9 sobe um nível: do teste local para as boas práticas SOA na nuvem. Os testes de integração que rodam hoje no laptop do aluno e no pipeline do MR vão virar smoke tests executados após o deploy, garantindo que o serviço continua respondendo aos contratos mesmo depois que a infra muda.

  • O WebApplicationFactory de hoje vira o template para os testes de contrato entre serviços.
  • O eixo de RNF justificado hoje (eficiência, segurança, confiabilidade) volta como SLO medido pelo monitoramento na nuvem.
Para o autoestudo da aula 9

Antes das 10h: ler material da aula 9, listar 3 práticas de SOA na nuvem aplicáveis ao MVP do parceiro e estudar a relação entre testes de integração e smoke tests pós-deploy.

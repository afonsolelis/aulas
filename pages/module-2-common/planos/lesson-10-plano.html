Resumo

A aula 10 introduz qualidade automatizada. Trabalhamos a distinção entre White Box (testes que conhecem os internos) e Black Box (testes que enxergam só a interface), refactoring seguro, identificação de code smells, testes E2E com Playwright, integração de API com Supertest, cobertura de código, mocks/stubs e finalmente CI/CD com GitHub Actions.

O fechamento prepara a tarde, em que cada equipe terá seu pipeline rodando todos os testes a cada push e bloqueando merges com falhas.

Objetivos de aprendizagem

  • OA-1Diferenciar testes White Box e Black Box e escolher a abordagem certa para cada caso.
  • OA-2Identificar code smells e aplicar receitas de refactoring que preservam comportamento.
  • OA-3Escrever testes E2E com Playwright simulando o usuário no browser.
  • OA-4Testar a API com Supertest sem precisar subir o servidor manualmente.
  • OA-5Medir cobertura de código e interpretar o que ela revela (e o que ela esconde).
  • OA-6Configurar pipeline CI/CD com GitHub Actions rodando os testes a cada push.

Pré-requisitos

  • Aula 9 concluída: frontend responsivo do projeto operacional na develop.
  • Playwright instalado localmente: npx playwright install.
  • Lista de 5 fluxos críticos do projeto que precisam de teste E2E.

Estratégias de metodologia ativa

  • Refactoring com rede de segurança — só refatoramos sobre teste verde; o teste é o critério de "não quebrei nada".
  • Crítica deliberada da cobertura — confronto direto com a falsa segurança do "100% coverage".
  • CI como contrato de equipe — pipeline configurado em sala para que ninguém possa "esquecer" de rodar testes.
Princípio orientador

Teste que nunca quebrou em CI provavelmente não testa nada importante.

Critério mínimo

Aluno que sair às 16h sem pipeline CI verde no projeto está em débito técnico para a aula 11 (resiliência e fechamento).

Conexão com a aula 11

A aula 11 fecha o módulo aprofundando a infraestrutura: TCP/IP, DNS, HTTPS, CDN, load balancer e padrões de resiliência. Os testes de hoje viram a base para regressão durante essa última iteração.

  • O CI configurado deve barrar qualquer mudança da próxima aula que quebre comportamento existente.
  • A cobertura atual é o baseline a manter na última iteração.
Para o autoestudo da aula 11

Antes das 8h: ler material da aula 11, observar o cabeçalho HTTPS de um site que você usa diariamente e pensar em 2 falhas que seu projeto sofreria com 100 usuários simultâneos.

🌐 JavaScript fora do browser

Node.js usa o motor V8 do Chrome para executar JavaScript no servidor, sem precisar de navegador.

⚡ Event-driven, non-blocking I/O

Nunca fica parado esperando: enquanto aguarda resposta do banco, já atende outra requisição.

📅 Criado por Ryan Dahl em 2009

Open-source, mantido pela OpenJS Foundation. Versão LTS atual: Node 20+.

🏢 Usado por Netflix, Uber, LinkedIn

Alta performance em APIs com muitas conexões simultâneas e baixa latência.

O Event Loop
Como Node.js processa tudo sem travar
📚
Call Stack
Onde o código JavaScript executa, linha por linha. Apenas uma operação por vez.
🌐
Web APIs / I/O
Operações assíncronas: leitura de arquivo, consulta ao banco, chamadas de rede.
📬
Event Queue
Fila de callbacks prontos para executar. O Event Loop os busca aqui quando a Call Stack esvazia.
🔄
Event Loop
O maestro: verifica se a Call Stack está vazia e move callbacks da fila para execução.

💡 Regra de ouro

Node processa 1 coisa por vez, mas nunca bloqueia esperando I/O — por isso consegue atender milhares de conexões simultâneas com um único processo.

Padrão MVC — Model View Controller
Separação de responsabilidades que organiza e escala qualquer sistema
🎨
View — O que o usuário vê
HTML, CSS, JavaScript no browser
Interface visual que consome a API — no nosso projeto, o frontend é separado do backend.
⬇️ HTTP Request / Response
🎮
Controller — A lógica da requisição
Recebe req, valida dados, chama o Model, envia res
Orquestra o fluxo: autenticação, tratamento de erros, serialização da resposta.
⬇️ Chamadas de função
📦
Model — O acesso aos dados
Consultas SQL, regras de negócio, validações
Única camada que fala com o banco de dados — isola a lógica de persistência.

Sem View no backend — o frontend (HTML/JS) é uma aplicação separada. O backend expõe apenas endpoints REST (JSON).

Estrutura de Pastas MVC
Cada arquivo tem um lugar certo — convenção que todo time entende
meu-projeto/ — árvore de arquivos

🏗️ Por que essa estrutura?

  • config/ — conexão com banco, variáveis de ambiente
  • models/ — todo SQL fica aqui, isolado
  • controllers/ — lógica HTTP, sem SQL
  • routes/ — mapeia URLs para controllers
  • server.js — ponto de entrada da aplicação

🧪 Spec-Driven Development
RM-ODP → Business Drivers → Spec → Tests → Code

1️⃣ RM-ODP: Enterprise → Business Driver

Enterprise viewpoint: "Usuários precisam se cadastrar para acessar recursos protegidos."
Computational: POST /users deve criar usuário se email não existe.

2️⃣ Spec — defina o contrato antes do código

POST /users { nome, email } válidos → 201 { id, nome, email }
POST /users email duplicado → 409 { error: "já existe" }

3️⃣ 🔴 Test — spec vira código executável

it('cria usuário com email único', async () => {
  const res = await request(app)
    .post('/users')
    .send({nome:'Ana',email:'ana@t.com'});
  expect(res.status).toBe(201);
  expect(res.body).toHaveProperty('id');
}); // ❌ FALHA — rota não existe

4️⃣ 🟢 Code → 🔵 Refactor

Implemente o mínimo para passar. Refatore com segurança. O teste garante que o business driver continua satisfeito.

🔍 White Box Testing — você conhece os internos

Você escreveu o código → conhece a lógica interna → testa cada unidade isolada. Na aula 10: Black Box — testamos comportamento externo sem conhecer os internos.

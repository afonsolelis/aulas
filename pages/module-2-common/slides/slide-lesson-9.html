🎯 Objetivo da Aula 9

Auditar o código gerado por IA: gerar o relatório de cobertura, medir as métricas de qualidade no terminal e refatorar os pontos vermelhos até os números entrarem na régua.

A virada: a IA escreve, você audita
Seu trabalho deixou de ser digitar — passou a ser ler os números e decidir

🤖 Quem digita agora é a IA

Copilot, ChatGPT e Claude geram grande parte do código. O custo de produzir caiu; o de confiar não. O dev migrou de autor para revisor.

⚠️ Passar no teste ≠ código bom

A IA otimiza para "funciona", raramente para "sustentável". O código roda, o teste fica verde — e mesmo assim está duplicado, complexo ou inseguro.

🔍 O papel do dev mudou

Auditar é ler o raio-x: cobertura, complexidade, duplicação, dependências, segurança e tipos. Sem números, "código bom" vira opinião.

📐 Os RNF entram em cena

Os testes garantem que o código faz a coisa certa. As métricas garantem o quanto disso está protegido e se o código é bom. Testes são a rede; métricas são o raio-x.

RNF mensuráveis — ISO/IEC 25010
8 características de qualidade · foco em 4 que medimos hoje no terminal
MANT
Manutenibilidade
Quão fácil é mudar o código sem quebrar. Complexidade, duplicação e MI medem isto.
SEG
Segurança
Padrões inseguros e CVEs em dependências. npm audit + eslint-plugin-security.
CONF
Confiabilidade
O quanto está protegido por testes. Cobertura de statements e branches.
DES
Eficiência
Custo de execução do algoritmo. Ler o Big-O: O(n²) da IA vira O(n).

📚 As 8 características

Adequação Funcional · Eficiência de Desempenho · Compatibilidade · Usabilidade · Confiabilidade · Segurança · Manutenibilidade · Portabilidade.

🎯 RNF viram número

"Código bom" é subjetivo. RNF medido é objetivo: cobertura ≥ 80%, ciclomática ≤ 10, duplicação < 5%, 0 CVE alto. O painel local transforma adjetivo em medição.

Cobertura de testes — o entregável central
npm test -- --coverage · gere e anexe o relatório ao MR
npm test -- --coverage

📊 As 4 dimensões

  • % Statements — instruções executadas
  • % Branches — galhos de if/else testados (a mais reveladora)
  • % Functions — funções chamadas
  • % Lines — linhas físicas tocadas

🗂️ O relatório HTML

Abra coverage/lcov-report/index.html — mostra linha-a-linha o que falta (vermelho = não coberto). Cobertura é bússola para achar buracos, não meta cega de 100%. Gere e anexe o relatório ao MR.

Complexidade ciclomática — ESLint
Caminhos independentes pela função · cada decisão soma +1
npx eslint src

🧮 O que a ciclomática mede

Número de caminhos independentes pela função — o mínimo de testes para cobrir todos os galhos. Mais decisões (if, &&, case), maior o número.

📏 As regras no ESLint

  • complexity: [error, 10]
  • max-depth: [error, 4]
  • max-params: [error, 4]
  • max-lines-per-function: [error, 50]
Complexidade cognitiva — ts-complex
Não é caminho de execução: é esforço de leitura · aninhamento penaliza
npx ts-complex src/freight.ts

🧠 Ciclomática × Cognitiva

Ciclomática = caminhos independentes (testabilidade). Cognitiva = esforço de leitura humana; cada nível de aninhamento de if pesa mais que o anterior.

🔬 calcFrete medida

ANTES: ciclo 14 cog 22 — três blocos aninhados de if/else if. Difícil de ler, difícil de cobrir. Candidata número 1 a refatorar.

Manutenibilidade — o dashboard com plato
npx plato -r -d report src · relatório HTML navegável

📈 Maintainability Index (0–100)

Quanto maior, melhor. Faixas:

< 65 ruim 65–85 moderado > 85 bom

📊 O que o relatório mostra

  • MI e ciclomática média por arquivo
  • Halstead — volume, dificuldade, esforço
  • SLOC — linhas de código reais
  • Erros de lint agregados

🖥️ Gerar o dashboard

Um comando produz report/index.html — gráficos navegáveis por arquivo, ótimos para anexar como evidência no MR.

$ npx plato -r -d report src

Leitura: MI médio do projeto 78 (bom). Pior arquivo order.service.ts com MI 61 — alvo de refatoração.

Duplicação — jscpd
Copy/paste que a IA adora gerar · limiar < 5%
npx jscpd src

♻️ DRY — Don't Repeat Yourself

Bloco repetido é bug repetido: corrige num lugar, esquece nos outros. A IA gera variações quase idênticas porque não vê o arquivo inteiro.

📏 Limiar de duplicação

Meta: < 5% de linhas duplicadas. No exemplo, 4.7% (12 clones) está OK — acima disso, extrair função e remover o copy/paste.

Dependências circulares — madge
npx madge --circular src · mede acoplamento entre módulos
npx madge --circular src

🔄 Por que ciclos são ruins

order → payment → order: nenhum dos dois compila/testa isolado, mudar um quebra o outro. Acoplamento alto = manutenção cara.

🗺️ Visualizar o grafo

Para ver o desenho completo das dependências e o acoplamento:

$ npx madge --image graph.svg src

Meta: 0 dependências circulares.

Segurança — dependências e código
npm audit (CVE nas libs) + eslint-plugin-security (padrões inseguros)

🔐 npm audit — CVE nas bibliotecas

$ npm audit --audit-level=high

Saída: found 3 vulnerabilities (1 critical, 2 high). Correção automática quando possível:

$ npm audit fix

Segurança das dependências que a IA importou sem checar.

🧹 eslint-plugin-security — código

Padrões inseguros que a IA costuma gerar:

  • detect-non-literal-regexp — RegExp de input do usuário
  • detect-child-process · detect-eval-with-expression
  • detect-non-literal-fs-filename — fs com path externo
  • detect-object-injection

Clássico da IA: SQL por concatenação (injection), eval, RegExp de entrada não sanitizada.

Saídas e tipos — type-coverage
npx type-coverage --at-least 95 · any esconde bug · auditar o Big-O
npx type-coverage --detail --at-least 95

🧷 any esconde bugs

type-coverage mede o % de código coberto por tipos (não-any). Tipos garantem o shape do retorno — o contrato de saída da função. Meta: > 95%.

📤 Auditar saídas e algoritmo

  • Contrato de saída: tipo de retorno e edge cases (null, undefined, array vazio)
  • Big-O: a IA gera O(n²) (.find/.includes em loop) — trocar por Set O(n)
Refatoração guiada pelos números — calcFrete
Substituir Condicional por Tabela · saídas idênticas, métricas despencam
freight.ts (gerado por IA) · ciclo 14 / cog 22
freight.ts (refatorado) · ciclo 3 / cog 4

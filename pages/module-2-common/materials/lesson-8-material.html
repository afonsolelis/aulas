1. Por que JavaScript é assíncrono?

JavaScript foi criado em 1995 para rodar dentro do navegador — um ambiente com uma restrição crítica: há apenas uma thread de execução. Diferente de linguagens de servidor como Java ou Python, onde múltiplas threads podem rodar em paralelo, o JavaScript do navegador opera em um único fio de execução. Isso significa que, se o código travar esperando uma resposta de rede, a tela congela, eventos do usuário não são processados e a experiência se torna inaceitável. A assincronia é, portanto, uma necessidade de sobrevivência do ambiente, não uma escolha de design.

O mecanismo que torna isso possível é o event loop, composto por três estruturas principais. A call stack (pilha de chamadas) é onde funções são empilhadas e executadas, uma de cada vez, de forma síncrona. A task queue (fila de tarefas, também chamada de macrotask queue) armazena callbacks de operações assíncronas como setTimeout, eventos DOM e callbacks de I/O. A microtask queue tem prioridade sobre a task queue e é esvaziada completamente após cada tarefa — é aqui que vivem as Promises resolvidas e callbacks de queueMicrotask. O event loop tem uma função simples: quando a call stack está vazia, ele verifica as filas e empurra a próxima tarefa para execução.

Uma analogia útil: imagine um restaurante com um único garçom (a thread JS). Ele anota seu pedido e o leva à cozinha — mas não fica parado na janela esperando a comida ficar pronta. Ele volta para as mesas, atende outros clientes, pega mais pedidos. Quando a cozinha sinaliza que um prato está pronto (event), o garçom o pega e serve. A cozinha é a rede, o disco, os timers — operações que acontecem fora da thread principal. O garçom nunca para; é o evento que o chama de volta.

Por que isso importa na prática?

Quando você chama fetch('/api/dados'), o JavaScript não espera a resposta. Ele registra um callback, continua executando o restante do código e, quando a resposta chega (um evento de rede), empurra o callback para a microtask queue. Isso explica por que console.log(resultado) logo após um fetch sem await imprime undefined — o resultado ainda não chegou quando a linha é executada. Entender o event loop é pré-requisito para não escrever código assíncrono quebrado.

Uma consequência prática imediata é que operações de longa duração dentro da call stack — um loop for processando um array enorme, por exemplo — bloqueiam o event loop inteiro. Durante esse tempo, nenhum clique, nenhum scroll e nenhuma resposta de rede são processados. A UI trava. Para esse tipo de trabalho pesado de CPU, a solução são Web Workers, que rodam em threads separadas sem bloquear a thread principal. Para operações de I/O (rede, disco), a solução é a assincronia nativa que estudamos aqui.

Conceito: Concorrência sem paralelismo

JavaScript é single-threaded mas concorrente. "Concorrência" aqui significa que múltiplas operações assíncronas podem estar em andamento simultaneamente — esperando respostas de rede, timers etc. — mas apenas uma executa código JavaScript por vez. Web Workers permitem paralelismo real, mas são threads isoladas que não compartilham estado com a thread principal e comunicam-se apenas via mensagens.

2. Callbacks, Promises e async/await

A história da assincronia em JavaScript é uma evolução de legibilidade. Nos primórdios do Node.js e das APIs de browser, toda operação assíncrona aceitava um callback — uma função passada como argumento que seria chamada quando a operação completasse. Para uma única operação, isso funcionava bem. O problema aparecia quando operações dependiam umas das outras: carregar um usuário, depois carregar seus pedidos, depois carregar os itens de cada pedido resultava em funções aninhadas formando o famoso callback hell (ou "pyramid of doom") — um triângulo de indentação crescente que tornava o código impossível de ler, manter e depurar. O tratamento de erros era ainda pior: cada nível precisava de seu próprio callback de erro.

As Promises, introduzidas nativamente no ES6 (2015), resolveram o aninhamento com encadeamento. Uma Promise é um objeto que representa um valor que pode estar disponível agora, no futuro ou nunca. Ela existe em um de três estados: pending (operação em andamento), fulfilled (completou com sucesso, tem um valor) ou rejected (falhou, tem um motivo de erro). Uma vez que uma Promise sai do estado pending, ela é imutável — não pode voltar a pending nem mudar de fulfilled para rejected. Ao invés de aninhar callbacks, você encadeia .then() para sucesso e .catch() para falha. O encadeamento é possível porque cada .then() retorna uma nova Promise, permitindo que a cadeia continue de forma linear, de cima para baixo, sem pirâmide.

O async/await, introduzido no ES2017, é açúcar sintático sobre Promises — não é uma nova forma de assincronia, mas sim uma nova forma de escrever código baseado em Promises com aparência de código síncrono. Uma função marcada com async sempre retorna uma Promise. Dentro dela, await pausa a execução da função (sem bloquear a thread principal — o event loop continua rodando) até a Promise resolver, então extrai o valor resolvido. O resultado é código que pode ser lido de cima para baixo, com variáveis recebendo valores de operações assíncronas como se fossem síncronas, e com tratamento de erros via try/catch como no código síncrono convencional.

javascript — evolução histórica: callback hell, Promises, async/await 
// CALLBACK HELL — pyramid of doom
carregarUsuario(1, function(usuario) {
  carregarPedidos(usuario.id, function(pedidos) {
    carregarItens(pedidos[0].id, function(itens) {
      // 3 níveis de indentação — e poderia ter mais
    }, function(err) { console.error(err); });
  }, function(err) { console.error(err); });
}, function(err) { console.error(err); });

// PROMISES — encadeamento linear, um .catch() para tudo
carregarUsuario(1)
  .then(usuario => carregarPedidos(usuario.id))
  .then(pedidos => carregarItens(pedidos[0].id))
  .then(itens => console.log(itens))
  .catch(err => console.error(err));

// ASYNC/AWAIT — parece código síncrono, é código assíncrono
async function carregarTudo() {
  try {
    const usuario = await carregarUsuario(1);
    const pedidos = await carregarPedidos(usuario.id);
    const itens   = await carregarItens(pedidos[0].id);
    console.log(itens);
  } catch (err) {
    console.error(err);
  }
}

// PARALELO com Promise.all — mais rápido que sequencial
async function carregarParalelo() {
  const [usuario, config] = await Promise.all([
    carregarUsuario(1),
    carregarConfig()    // independente do usuário — pode rodar junto
  ]);
}
Boa prática: paralelo vs sequencial

Prefira async/await para fluxos sequenciais onde cada passo depende do anterior. Use Promise.all() para operações independentes que podem rodar em paralelo — isso reduz o tempo total de espera. Usar await sequencial em operações independentes é um bug de desempenho silencioso.

Aprofundamento: Promise.all, Promise.race, Promise.allSettled e Promise.any

▼

Promise.all(array) — aguarda todas as Promises resolverem. Se qualquer uma rejeitar, toda a expressão rejeita imediatamente ("fail fast"). Use quando todas as operações são obrigatórias e uma falha invalida o resto.

Promise.allSettled(array) — aguarda todas as Promises terminarem, independente de sucesso ou falha. Retorna um array de objetos { status, value } ou { status, reason }. Use quando você precisa tratar cada resultado individualmente, mesmo que alguns falhem.

Promise.race(array) — resolve ou rejeita assim que a primeira Promise terminar (sucesso ou falha). Útil para implementar timeouts: Promise.race([fetchDados(), rejectAfter(5000)]).

Promise.any(array) — resolve quando a primeira Promise tiver sucesso. Só rejeita se todas falharem (com um AggregateError). Útil para redundância: tentar múltiplos endpoints e usar o mais rápido que responder com sucesso.

3. fetch() API

A fetch() API é a forma nativa e moderna de fazer requisições HTTP no navegador. Introduzida como padrão W3C e disponível em todos os browsers modernos desde 2015, ela substitui o antigo XMLHttpRequest (XHR), que tinha uma API verbosa e baseada em eventos. fetch() retorna uma Promise que resolve para um objeto Response quando o servidor responde — não quando os dados estão prontos para uso, mas quando os primeiros bytes da resposta chegam e os metadados estão disponíveis.

Isso explica por que são necessários dois awaits para obter dados JSON. O primeiro await fetch(url) espera o servidor responder e retorna o objeto Response. Neste ponto, você tem acesso a metadados como res.status, res.headers e res.ok, mas o corpo da resposta ainda está sendo transmitido. O segundo await res.json() espera o corpo completo ser recebido e então o parseia como JSON. Chamar res.json() sem await retorna mais uma Promise — sem o await você ficará com um objeto Promise não resolvido, não os dados.

A propriedade res.ok é true quando o status HTTP está entre 200 e 299, e false para qualquer outro código — incluindo 400, 404, 500 e todos os outros erros. Esta propriedade é um atalho conveniente; se precisar verificar o código exato, use res.status. Conhecer essa propriedade é fundamental para o próximo tópico: a armadilha mais comum ao usar fetch().

Armadilha Crítica — fetch() NÃO rejeita em erros HTTP 4xx/5xx

fetch() só rejeita a Promise em caso de falha de rede — sem conexão, servidor inacessível ou CORS bloqueado. Respostas com status 400, 401, 403, 404, 500 ou qualquer outro código de erro HTTP são tratadas como sucesso pelo fetch — a Promise resolve normalmente. Se você não checar res.ok, seu código vai tentar parsear a resposta de erro como se fossem dados válidos e vai falhar silenciosamente, ou pior: vai renderizar a mensagem de erro do servidor como se fossem dados do usuário.

javascript — a armadilha e o wrapper correto 
// ERRADO — 404 e 500 NÃO caem no catch
const res   = await fetch('/api/produtos/999');
const dados = await res.json(); // pode ser {"error":"Not found"} — tratado como sucesso!

// CORRETO — verificar res.ok antes de parsear o corpo
const res = await fetch('/api/produtos/999');
if (!res.ok) throw new Error(`HTTP ${res.status}`);
const dados = await res.json();

// MELHOR — wrapper reutilizável que unifica o tratamento
async function fetchJson(url, options = {}) {
  const res = await fetch(url, options);
  if (!res.ok) {
    const erro = await res.json().catch(() => ({})); // body pode não ser JSON
    throw new Error(erro.message || `HTTP ${res.status}`);
  }
  return res.json();
}

Quando fetch() rejeita de verdade?

A Promise de fetch() é rejeitada apenas em falhas de rede genuínas: o computador está sem conexão, o DNS não resolve o hostname, o servidor recusou a conexão TCP, ou a requisição foi bloqueada por CORS (Cross-Origin Resource Sharing). Em todos esses casos, nem mesmo uma resposta HTTP chega — não há status code para verificar. Esta distinção é essencial para o tratamento de erros robusto, pois falhas de rede e erros HTTP requerem mensagens diferentes para o usuário.

4. Os 3 estados da UI assíncrona

Qualquer operação assíncrona que afeta a interface do usuário precisa tratar exatamente três estados: Loading, Success e Error. Ignorar qualquer um desses estados não é apenas uma má prática — é um bug de UX que afeta diretamente a percepção de qualidade e confiabilidade do sistema. Um sistema que simplesmente não mostra nada enquanto carrega, ou que silencia erros sem explicar o que aconteceu, frustra o usuário e destrói a confiança que ele deposita na ferramenta.

O estado de Loading comunica que algo está acontecendo. Sem feedback visual de carregamento, o usuário não sabe se o sistema está processando ou travado — e vai clicar no botão novamente, disparando múltiplas requisições simultâneas. Spinners e skeleton screens são as duas estratégias principais (detalhadas na seção 9). O estado de Success é onde o fluxo feliz vive: renderizar os dados recebidos, atualizar a interface, possivelmente mostrar uma mensagem de confirmação transitória e então deixar o usuário trabalhar. O estado de Error precisa comunicar o que deu errado em linguagem humana, nunca expor mensagens técnicas, stack traces ou códigos de status ao usuário final, e sempre oferecer uma saída — geralmente um botão "Tentar novamente".

Uma implementação correta dos três estados protege o usuário de decisões equivocadas baseadas em informações incompletas. Se você renderiza uma lista vazia durante o loading, o usuário pode concluir erroneamente que não há dados. Se você mantém a última versão dos dados em tela durante um erro silencioso, o usuário pode agir sobre informações desatualizadas sem saber. Cada estado exige uma representação visual deliberada — não acidental — que comunica claramente o que o sistema está fazendo.

⏳
Loading
Spinner, skeleton screen ou progress bar. Comunica que a operação está em andamento e que o sistema está respondendo.
✅
Success
Renderizar os dados obtidos. Pode incluir toast de confirmação para ações destrutivas ("Salvo com sucesso"). Substitui o loading.
⚠
Error
Mensagem em linguagem humana. Nunca stack traces. Sempre uma ação de recuperação: retry, voltar, ou contato com suporte.
javascript — os 3 estados implementados na UI 
async function carregarDados() {
  const container = document.getElementById('container');

  // Estado 1: Loading
  container.innerHTML = `
    <div class="loading-state">
      <div class="spinner"></div>
      <p>Carregando produtos...</p>
    </div>
  `;

  try {
    const dados = await fetchJson('/api/produtos');

    // Estado 2: Success
    renderizar(dados);

  } catch (err) {
    // Estado 3: Error — mensagem humana + botão de retry
    container.innerHTML = `
      <div class="error-state">
        <p>Não foi possível carregar os produtos.</p>
        <p class="detail">${err.message}</p>
        <button onclick="carregarDados()">Tentar novamente</button>
      </div>
    `;
  }
}

Exemplo visual: spinner + skeleton screen

O skeleton screen imita o layout do conteúdo que virá, reduzindo a percepção de espera comparado a uma área vazia com apenas um spinner.

5. Tratamento de erros robusto

O tratamento de erros em código assíncrono tem três camadas distintas que precisam ser consideradas separadamente. A primeira são os erros de rede — quando o fetch() rejeita sua Promise porque não foi possível sequer estabelecer uma conexão HTTP. A segunda são os erros HTTP — quando o servidor respondeu, mas com um status de erro (4xx ou 5xx), indicando que a requisição foi inválida, não autorizada, não encontrada ou que houve um problema no servidor. A terceira são os erros de parsing JSON — quando a resposta chegou com status 200, mas o corpo não é um JSON válido (por exemplo, uma página HTML de erro gerada por um proxy ou pelo servidor antes da aplicação tomar controle).

Cada tipo de erro tem causas distintas, mensagens distintas para o usuário e estratégias de recuperação distintas. Um erro de rede sugere verificar a conexão. Um erro 404 sugere que o recurso não existe mais e talvez o usuário precise navegar para outro lugar. Um erro 401 indica sessão expirada e exige reautenticação. Um erro 500 indica problema no servidor e o usuário pode tentar novamente mais tarde. Tratar todos os erros com a mesma mensagem genérica ("Algo deu errado") é melhor do que não tratar, mas ainda assim é uma experiência inferior à que um sistema bem projetado pode oferecer.

As mensagens de erro para o usuário merecem atenção especial. Nunca exiba mensagens técnicas como TypeError: Failed to fetch ou stack traces — além de confusas, podem revelar informações sobre a arquitetura interna do sistema. Traduza erros para linguagem humana contextualizada: "Não foi possível carregar os produtos. Verifique sua conexão e tente novamente." é infinitamente mais útil do que a mensagem do runtime. Para aplicações de produção, registre o erro técnico completo em um serviço de monitoramento (Sentry, Datadog) enquanto exibe a mensagem amigável ao usuário.

javascript — wrapper com tratamento em 3 camadas 
async function fetchJson(url, options = {}) {
  let res;
  try {
    res = await fetch(url, options);        // Camada 1: erro de rede
  } catch (networkErr) {
    throw new Error('Sem conexão com o servidor. Verifique sua internet.');
  }

  if (!res.ok) {                             // Camada 2: erro HTTP
    const corpo = await res.json().catch(() => ({}));
    throw new Error(corpo.message || `Erro ${res.status}: ${res.statusText}`);
  }

  try {
    return await res.json();                // Camada 3: erro de parsing
  } catch {
    throw new Error('Resposta do servidor não é um JSON válido.');
  }
}

// No componente: log técnico separado da mensagem ao usuário
try {
  const dados = await fetchJson('/api/produtos');
  renderizar(dados);
} catch (err) {
  logParaSentry(err);                        // log técnico completo para debugging
  mostrarErroNaUI(err.message);              // mensagem amigável para o usuário
}
Erro silencioso é o pior tipo de erro

Evite .catch(() => {}) ou catch (err) {} sem qualquer ação dentro. "Engolir" erros silenciosamente torna bugs impossíveis de depurar — o código parece funcionar mas dados nunca chegam na tela, sem nenhuma pista do que aconteceu. Se você não sabe como tratar um erro em determinado ponto, pelo menos faça um console.error(err) temporário e mostre o estado de erro na UI.

6. Headers, métodos e corpo da requisição

Por padrão, fetch(url) faz uma requisição GET sem corpo e sem headers customizados. Para qualquer outra operação — criar, atualizar, deletar recursos ou enviar dados — você precisa passar um objeto de opções como segundo argumento. As três propriedades mais importantes são method (string com o verbo HTTP), headers (objeto com os headers da requisição) e body (string com o corpo serializado). Para enviar JSON, o padrão é: serializar o objeto com JSON.stringify(), definir o Content-Type como application/json e usar o método HTTP correto conforme a semântica da operação — POST para criar, PUT para substituir completamente, PATCH para atualização parcial, DELETE para remover.

O header Content-Type: application/json é obrigatório ao enviar JSON — ele informa ao servidor como o corpo deve ser interpretado. Sem ele, muitos frameworks de servidor (como Express com o middleware express.json()) não vão parsear o corpo corretamente e a requisição chegará com req.body vazio ou com o valor undefined. Da mesma forma, o header Accept: application/json indica que o cliente espera uma resposta JSON — alguns servidores podem retornar HTML ou XML por padrão se não especificado.

A autenticação geralmente é feita via o header Authorization com o esquema Bearer: Authorization: Bearer <token>. O token é tipicamente um JWT (JSON Web Token) obtido no login e armazenado pelo cliente. Em uma aplicação real, você criaria uma função auxiliar que adiciona automaticamente o token de autenticação a todas as requisições privadas, centralizando a lógica e evitando a repetição do mesmo header em cada chamada individual.

javascript — POST, PATCH com JSON e autenticação Bearer 
// POST — criar recurso
async function criar(dados) {
  return fetchJson('/api/produtos', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${localStorage.getItem('token')}`
    },
    body: JSON.stringify(dados)
  });
}

// PATCH — atualizar campos específicos
async function atualizarPreco(id, novoPreco) {
  return fetchJson(`/api/produtos/${id}`, {
    method: 'PATCH',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ preco: novoPreco })
  });
}

// Helper que injeta autenticação em todas as requisições privadas
function authFetch(url, options = {}) {
  const token = localStorage.getItem('authToken');
  return fetchJson(url, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { 'Authorization': `Bearer ${token}` } : {}),
      ...options.headers  // permite sobrescrever por chamada
    }
  });
}

7. localStorage e sessionStorage

O navegador oferece duas APIs de armazenamento no lado do cliente que complementam o trabalho com fetch(): localStorage e sessionStorage. A diferença fundamental entre elas é a persistência. O localStorage persiste os dados indefinidamente — eles sobrevivem ao fechamento da aba, ao fechamento do navegador e à reinicialização do computador, até que sejam explicitamente removidos pelo código ou pelo usuário nas configurações do navegador. O sessionStorage persiste apenas durante a sessão da aba atual — é apagado quando a aba ou o navegador é fechado, e não é compartilhado entre abas mesmo do mesmo site.

A API é simples e idêntica entre os dois: setItem(chave, valor) para gravar, getItem(chave) para ler (retorna null se a chave não existir), removeItem(chave) para remover uma chave específica e clear() para apagar todo o conteúdo do storage. Como apenas strings são suportadas, objetos e arrays precisam ser serializados com JSON.stringify() antes de gravar e desserializados com JSON.parse() ao ler. Um erro comum é esquecer a serialização — o storage vai chamar .toString() automaticamente, resultando em "[object Object]" gravado ao invés do objeto real.

Os casos de uso típicos são: localStorage para preferências do usuário (tema escuro/claro, idioma, tamanho de fonte), tokens de autenticação de longa duração, e cache leve de dados estáticos. sessionStorage para estado temporário de formulários multi-passo, filtros de busca que não precisam persistir entre sessões, e dados de navegação dentro de um fluxo específico. O limite de armazenamento é tipicamente 5 MB por origem — inadequado para grandes volumes de dados binários ou listas extensas.

Limitação de segurança — nunca guarde dados sensíveis

localStorage e sessionStorage são acessíveis a qualquer JavaScript rodando na mesma origem. Um ataque XSS (Cross-Site Scripting) bem-sucedido pode extrair todo o conteúdo do storage com uma única linha de código. Nunca guarde senhas, dados de cartão de crédito ou informações médicas. Para tokens de autenticação em aplicações de alta segurança, prefira cookies com a flag HttpOnly — eles não são acessíveis ao JavaScript. Use o Web Storage para conveniência e preferências, não como cofre de dados sensíveis.

javascript — helpers para serialização e padrões de uso 
// Wrapper com serialização automática e proteção contra parse errors
const store = {
  set(chave, valor) {
    localStorage.setItem(chave, JSON.stringify(valor));
  },
  get(chave, padrao = null) {
    try {
      const item = localStorage.getItem(chave);
      return item !== null ? JSON.parse(item) : padrao;
    } catch { return padrao; } // valor corrompido → retorna padrão
  },
  remove(chave) { localStorage.removeItem(chave); }
};

// Uso: preferências e token de sessão
store.set('preferencias', { tema: 'escuro', idioma: 'pt-BR', fontePequena: false });
store.set('authToken', resposta.token);

const prefs = store.get('preferencias', { tema: 'claro' });
aplicarTema(prefs.tema);

// Logout: limpar dados do usuário do storage
function logout() {
  store.remove('authToken');
  store.remove('usuario');
  window.location.assign('/login.html');
}

8. AbortController — cancelar requisições

Imagine que o usuário abre uma tela que dispara um fetch() para carregar dados. Antes da resposta chegar, ele navega para outra tela. O fetch() continua em execução em background e, quando a resposta finalmente chega, o callback vai tentar atualizar um componente que já não existe mais na tela — causando erros, comportamentos inesperados ou, em frameworks como React, o aviso "Can't perform a state update on an unmounted component". A solução para esse problema clássico é o AbortController, uma API nativa do navegador que permite cancelar requisições fetch() em andamento.

O padrão é direto: você cria uma instância de AbortController, passa seu atributo signal para o objeto de opções do fetch(), e chama controller.abort() quando quiser cancelar. Quando a requisição é abortada, a Promise do fetch() é rejeitada com um erro especial chamado AbortError — um DOMException com name === 'AbortError'. É essencial verificar o nome do erro no bloco catch e ignorar (ou tratar de forma diferente) os AbortError, pois eles representam cancelamentos intencionais, não falhas reais que o usuário precisa ver.

O AbortController resolve também o problema de race conditions em buscas em tempo real. Sem cancelamento, se o usuário digita "pro" e depois "prod", podem chegar duas respostas fora de ordem — a de "prod" pode chegar antes da de "pro", e então a resposta de "pro" sobrescreve os resultados corretos. Com AbortController, cada nova busca cancela a anterior antes de iniciar, eliminando o risco de resultados desatualizados sobrescreverem os corretos.

javascript — AbortController para cancelar buscas e prevenir race conditions 
// Padrão: cancelar requisição anterior antes de iniciar nova
function criarBusca() {
  let controller;

  return async function(termo) {
    if (controller) controller.abort();    // cancela requisição anterior
    controller = new AbortController();

    try {
      const dados = await fetchJson(
        `/api/busca?q=${encodeURIComponent(termo)}`,
        { signal: controller.signal }
      );
      renderizar(dados);
    } catch (err) {
      if (err.name !== 'AbortError') throw err; // ignora cancelamentos intencionais
    }
  };
}

const buscar = criarBusca();
input.addEventListener('input', e => buscar(e.target.value));

// Timeout: abortar automaticamente após N milissegundos
function fetchComTimeout(url, options = {}, ms = 10000) {
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), ms);
  return fetchJson(url, { ...options, signal: controller.signal })
    .finally(() => clearTimeout(timer));
}

9. Estratégias de loading UX

A escolha entre diferentes estratégias de feedback de loading afeta diretamente a percepção de velocidade e qualidade da aplicação. Spinners são simples de implementar e adequados para ações discretas como enviar formulário ou clicar em um botão de salvar. Porém, se a espera for longa (mais de 3-4 segundos), o spinner gera ansiedade porque o usuário não tem nenhuma referência de quanto tempo falta. Skeleton screens são esqueletos visuais que imitam a estrutura do conteúdo que está carregando, preenchidos com um efeito shimmer animado. Pesquisas de UX mostram que skeleton screens reduzem a percepção subjetiva de espera porque o usuário processa informação estrutural mesmo sem os dados reais, tornando a transição menos abrupta.

A UI otimista (optimistic UI) é uma estratégia avançada onde a interface assume que a operação será bem-sucedida e atualiza imediatamente, revertendo apenas se o servidor retornar erro. Por exemplo: ao curtir uma publicação, o contador incrementa instantaneamente na tela sem esperar a confirmação do servidor. Isso torna a interface extremamente responsiva e elimina o ciclo perceptível de espera para ações simples. A desvantagem é a complexidade de implementar o rollback (reversão) quando a operação falha, especialmente se o usuário já interagiu com o estado otimista entre o envio e o erro.

O debounce em campos de busca é praticamente obrigatório para qualquer campo que dispara requisições ao digitar. Sem debounce, cada tecla pressionada dispara um fetch(), resultando em dezenas de requisições por segundo — sobrecarrega o servidor, consome largura de banda desnecessária e pode resultar em race conditions onde respostas chegam fora de ordem. O debounce agrupa eventos em uma janela de tempo e dispara apenas quando há uma pausa na digitação. Um valor típico de 300-500ms é imperceptível para o usuário mas reduz em mais de 90% o número de requisições disparadas durante uma sessão de busca.

javascript — debounce + busca com cancelamento e 3 estados 
// Debounce genérico — retarda a execução até o usuário parar
function debounce(fn, ms) {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), ms);
  };
}

const buscar = criarBusca(); // AbortController da seção anterior

const buscarComDebounce = debounce(async (termo) => {
  if (!termo.trim()) { limparResultados(); return; }

  mostrarLoading();
  try {
    const resultados = await buscar(termo);
    renderizarResultados(resultados);
  } catch (err) {
    if (err.name !== 'AbortError') mostrarErro(err.message);
  }
}, 400); // 400ms de pausa antes de disparar

document.getElementById('busca').addEventListener('input', e => {
  buscarComDebounce(e.target.value);
});

// Skeleton screen gerado dinamicamente
function renderizarSkeleton(qtd = 3) {
  return Array.from({ length: qtd }, () => `
    <div class="skeleton-card">
      <div class="skeleton" style="height:140px;margin-bottom:12px;"></div>
      <div class="skeleton" style="width:70%;height:18px;margin-bottom:8px;"></div>
      <div class="skeleton" style="width:40%;height:14px;"></div>
    </div>
  `).join('');
}

10. RM-ODP — fetch como interface de serviço distribuído

O framework RM-ODP (ISO/IEC 10746) oferece cinco visões complementares para descrever sistemas distribuídos. Para a aula de JavaScript assíncrono e fetch(), as visões mais diretamente relevantes são a Computational e a Engineering. A visão Computational descreve a decomposição funcional do sistema em componentes com interfaces bem definidas e contratos explícitos. A visão Engineering descreve como esses componentes se comunicam — os canais de comunicação, os protocolos, a latência, as falhas possíveis e os mecanismos de binding que tornam a comunicação possível.

Sob a visão Computational, cada chamada fetch(url, options) é a invocação de uma interface de serviço exposta por um componente remoto — a API REST rodando no servidor. O contrato desta interface inclui: o endpoint (URL + método HTTP), o formato dos parâmetros de entrada (query string, body JSON, headers de autenticação), o formato dos dados de saída (estrutura do JSON da resposta), e os possíveis estados de erro (status codes e seus significados documentados). A função fetchJson() wrapper que construímos é um proxy de interface: ela encapsula os detalhes de transporte HTTP e expõe apenas o contrato funcional — "dado uma URL, retorno os dados ou lanço um erro com mensagem amigável".

Sob a visão Engineering, o fetch() usa a rede como canal de comunicação — um canal com características reais e não ideais que moldam cada decisão de design. A latência (50–500ms típico) justifica o estado de Loading. A possibilidade de falha justifica o tratamento de erros e os retries com backoff exponencial. As requisições fora de ordem (race conditions) justificam o AbortController. A repetição excessiva de requisições justifica o debounce. Cada ferramenta que estudamos nesta aula é uma resposta de engenharia a uma característica real do canal de comunicação distribuído.

🏢
Enterprise — por que buscamos dados remotos?
O sistema precisa apresentar dados atualizados de um servidor centralizado para garantir consistência entre usuários diferentes. A regra de negócio exige que nenhum dado crítico exista apenas no cliente, onde pode estar desatualizado ou ser manipulado.
RF: O sistema exibe dados atualizados buscados da API a cada acesso
⚙
Computational — fetch como invocação de interface
O componente front-end invoca a interface GET /api/produtos do componente back-end. Contrato: aceita Bearer token no header Authorization, retorna array de produtos em JSON com status 200, ou objeto de erro com status 4xx/5xx. fetchJson() é o proxy desta interface.
fetchJson() encapsula o transporte e expõe apenas o contrato
🔧
Engineering — rede como canal com características reais
O canal HTTP tem latência variável (50–500ms), pode falhar (timeout, CORS, sem rede) e pode retornar respostas fora de ordem (race conditions). AbortController cancela requisições stale. Debounce limita a frequência de invocação. Retry com backoff lida com falhas transitórias.

11. RF, RN e RNF — front-end assíncrono

Os Requisitos Funcionais (RF) descrevem o que o sistema deve fazer. No contexto de integração front-end com API, os RFs tipicamente descrevem o fluxo completo de uma operação: buscar dados, exibir dados, enviar formulário, atualizar estado na tela. Um RF bem escrito menciona o endpoint envolvido, o formato esperado e o comportamento resultante: "O sistema deve permitir que o usuário liste todos os produtos da sua loja, buscando os dados em GET /api/produtos?page=1&limit=20 com autenticação Bearer; o resultado deve ser exibido em cards com nome, preço e estoque."

Os Requisitos de Negócio (RN) estabelecem restrições que emergem de regras do domínio, independentes de tecnologia. "A lista de produtos deve sempre refletir o estoque em tempo real — dados com mais de 5 minutos de cache não podem ser exibidos a usuários compradores" é uma RN que impõe restrições sobre estratégias de cache. "Usuários não autenticados não podem visualizar preços de atacado" é uma RN que se manifesta tanto no servidor (autorização no endpoint) quanto no front-end (o componente não renderiza o campo de preço de atacado sem token válido).

Os Requisitos Não-Funcionais (RNF) definem as qualidades do sistema. Para o front-end assíncrono, três eixos são especialmente relevantes: Usabilidade (o usuário sempre sabe o que está acontecendo — estados de loading e error explícitos), Confiabilidade (o sistema se recupera de falhas transitórias sem perda de dados) e Desempenho (a interface não sobrecarrega o servidor e responde rapidamente às interações). Cada um desses eixos deve ter métricas mensuráveis associadas, não apenas intenções vagas.

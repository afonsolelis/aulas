1. HTML Semântico

HTML é muito mais do que uma linguagem de marcação para "fazer coisas aparecerem no navegador". Cada elemento HTML carrega um significado — uma semântica — que descreve o propósito daquele trecho de conteúdo. Um <header> diz "este é o cabeçalho da página ou seção". Um <nav> diz "estes são links de navegação". Um <article> diz "este é um conteúdo autossuficiente". Usar esses elementos corretamente transforma o HTML em um documento estruturado, compreensível tanto para humanos quanto para máquinas.

A semântica importa por três razões fundamentais que afetam diretamente o sucesso de um produto digital. A primeira é a acessibilidade: leitores de tela usados por pessoas cegas ou com baixa visão navegam pelo HTML semântico para anunciar a estrutura da página — "você está no rodapé", "este é um formulário de login", "há três links de navegação". Sem semântica, o leitor de tela lê uma sopa de elementos <div> sem nenhum contexto. A segunda é o SEO: mecanismos de busca como o Google interpretam a hierarquia semântica para entender sobre o que a página fala e qual é a informação principal — o conteúdo dentro de um <h1> tem muito mais peso do que o mesmo texto dentro de um <div>. A terceira é a manutenibilidade: um time que lê HTML semântico entende a estrutura da página em segundos, sem precisar rastrear classes CSS para descobrir o que cada bloco representa.

Existe uma diferença fundamental entre os elementos genéricos <div> e <span> e os elementos semânticos. <div> e <span> são contêineres sem significado — servem apenas para agrupamento e estilização quando nenhum elemento semântico se aplica. Usá-los para estrutura de página onde existem elementos semânticos equivalentes é um erro de engenharia, não de estética. A regra prática: antes de escrever um <div>, pergunte-se se existe um elemento semântico que descreve o que esse bloco representa no documento.

Elementos estruturais de página

A estrutura de uma página HTML5 bem organizada usa um conjunto pequeno e coerente de elementos. O <header> contém o cabeçalho da página ou de uma seção — logotipo, título principal, tagline. O <nav> envolve links de navegação primária ou secundária. O <main> delimita o conteúdo principal da página — deve aparecer apenas uma vez por documento e não deve conter conteúdo repetido entre páginas (como barras laterais de navegação globais). O <section> agrupa conteúdo temático relacionado e deve ter um título próprio. O <article> envolve conteúdo autossuficiente que poderia ser distribuído independentemente — um post de blog, uma notícia, um card de produto. O <aside> marca conteúdo tangencialmente relacionado ao conteúdo principal — barras laterais, anúncios, links relacionados. O <footer> fecha a página ou seção com informações de rodapé — créditos, links legais, contato.

<header>

Hierarquia de títulos e elementos de conteúdo

Os títulos <h1> a <h6> formam uma hierarquia de importância — o <h1> é o título principal da página (deve aparecer apenas uma vez), os demais formam a árvore de seções e subseções. Não pule níveis por razões estéticas; use CSS para ajustar a aparência visual, mas mantenha a hierarquia lógica intacta. O <p> marca parágrafos de texto. <ul> e <ol> marcam listas não-ordenadas e ordenadas, respectivamente, com itens <li>. O elemento <a> com atributo href cria links. O <img> exige obrigatoriamente o atributo alt — uma descrição textual da imagem para acessibilidade; se a imagem for puramente decorativa, use alt="". O <table> com <thead>, <tbody>, <tr>, <th> e <td> estrutura dados tabulares — mas nunca deve ser usado para criar layouts de página, papel que pertence ao CSS Grid e ao Flexbox.

Boas práticas

Use o validador oficial do W3C (validator.w3.org) para checar se seu HTML está semanticamente correto. Um HTML válido garante comportamento previsível entre navegadores e é requisito básico para acessibilidade.

2. Formulários HTML

Formulários são a interface primária de coleta de dados do usuário na Web. Todo formulário começa com o elemento <form>, que define o contêiner semântico e os atributos de submissão. O atributo action especifica para onde os dados vão; sem JavaScript, o navegador faz uma requisição HTTP para aquela URL. O atributo method define se a submissão usa GET (parâmetros na URL, adequado para buscas) ou POST (dados no corpo da requisição, adequado para criação e alteração). Na prática moderna com SPAs e fetch, você frequentemente intercepta o evento submit e processa os dados manualmente, mas entender o comportamento nativo é essencial.

O elemento <input> é o mais versátil dos campos de formulário. Seu comportamento muda radicalmente de acordo com o atributo type: text é um campo de texto livre; email valida automaticamente o formato de e-mail e abre teclado adequado em dispositivos móveis; number restringe entrada numérica e habilita os atributos min, max e step; password oculta os caracteres; checkbox cria uma caixa de marcação booleana; radio cria botões de opção mutuamente exclusivos dentro do mesmo name; file abre o seletor de arquivos do sistema operacional. Cada tipo de input aciona comportamentos específicos do navegador — aproveite isso antes de reescrever em JavaScript.

A validação nativa do browser é subestimada por desenvolvedores iniciantes. Atributos como required, minlength, maxlength, min, max, pattern e type ativam validações automáticas que o navegador executa antes de permitir a submissão — sem uma linha de JavaScript. Essas validações também exibem mensagens de erro acessíveis, localizadas no idioma do sistema do usuário, e são comunicadas a leitores de tela via ARIA. A regra: use validação nativa para tudo que ela cobre; adicione JavaScript apenas para regras que o HTML não consegue expressar (como "senha e confirmação devem ser iguais" ou "nome de usuário já existe no banco").

Acessibilidade em formulários

Acessibilidade em formulários começa com o <label>. Cada campo de input deve ter um label associado via atributo for contendo o mesmo valor que o id do campo. Essa associação serve para que clicar no texto do label ative o campo (melhora a usabilidade) e para que o leitor de tela anuncie o nome do campo ao focar nele. Nunca use apenas o atributo de dica (hint text) como substituto ao label — esse atributo desaparece quando o usuário começa a digitar e não é anunciado por todos os leitores de tela.

  <label for="categoria">Categoria</label>
  <select id="categoria" name="categoria">
    <option value="">Selecione...</option>
    <option value="livros">Livros</option>

Validação no lado do servidor

Validação HTML e JavaScript no cliente é UX — ela agiliza o feedback ao usuário. Mas nunca é suficiente para segurança. Um usuário malicioso pode enviar dados diretamente para sua API, ignorando completamente o formulário. Toda validação crítica de negócio e segurança precisa existir também no servidor.

3. O DOM — Document Object Model

Quando um navegador carrega um arquivo HTML, ele não exibe o texto bruto — ele o analisa (faz o parsing) e constrói uma representação em árvore de todos os elementos, atributos e textos do documento. Essa árvore é chamada de DOM (Document Object Model). O DOM é o que existe na memória do navegador; o arquivo HTML é apenas a "semente" a partir da qual o DOM é construído. Depois do parsing, JavaScript pode modificar o DOM sem tocar no arquivo original, e o navegador reflete essas mudanças imediatamente na tela.

A relação entre HTML, DOM e JavaScript é a tríade central do desenvolvimento front-end. O HTML declara a estrutura inicial. O DOM é a representação viva dessa estrutura, manipulável em tempo real. O JavaScript é a linguagem que interage com o DOM através da API exposta pelo objeto global document. Quando você escreve document.getElementById('meu-id'), está pedindo ao navegador que localize, na sua árvore DOM, o nó com aquele ID e retorne uma referência ao objeto JavaScript que o representa. A partir dessa referência, você pode ler e modificar tudo sobre aquele elemento — conteúdo, atributos, estilos, eventos.

O browser constrói o DOM em fases bem definidas. Primeiro, o parser HTML lê os tokens do documento de cima para baixo, criando nós no DOM à medida que encontra abertura e fechamento de tags. Quando encontra um <script> sem atributos defer ou async, ele pausa o parsing para executar o JavaScript — por isso, scripts que manipulam o DOM devem ser colocados antes de </body> ou usar o atributo defer. Após o parsing completo, o browser dispara o evento DOMContentLoaded, sinalizando que o DOM está pronto. Imagens e outros recursos externos podem ainda estar carregando — o evento load dispara quando tudo termina.

document
  └─ <html>
      ├─ <head>
      │    └─ <title> "Loja"
      └─ <body>
           └─ <section id="lista-produtos">
                 ├─ <h2> "Nossos Produtos"
                 └─ <ul id="produtos-list"> [ itens dinâmicos ]

Representação em árvore do DOM — cada nó é um objeto JavaScript acessível

DOM e HTML sao diferentes

O DOM pode divergir do HTML original após modificações por JavaScript. Se você usar DevTools para inspecionar o DOM, verá o estado atual dos elementos — não o HTML que veio do servidor. Isso é importante ao debugar: o que você vê no "Inspecionar Elemento" reflete o estado vivo do DOM, não o arquivo em disco.

4. Selecionando Elementos

A seleção de elementos é o primeiro passo para qualquer manipulação do DOM. O objeto document (e qualquer elemento DOM) expõe vários métodos de busca, cada um com características distintas. Entender essas diferenças evita bugs sutis que consomem horas de debug. O método mais antigo é getElementById — recebe uma string e retorna o único elemento com aquele ID, ou null se não existir. É o método mais rápido para busca por ID, pois o browser mantém um índice interno de IDs.

O método querySelector é o mais versátil e o preferido em código moderno. Recebe qualquer seletor CSS válido — classe (.classe), ID (#id), tipo (div), combinador (section > h2), pseudo-classes (:first-child) — e retorna o primeiro elemento que corresponde, ou null. Já querySelectorAll retorna todos os elementos correspondentes como uma NodeList estática. A diferença entre NodeList (retornada por querySelectorAll) e HTMLCollection (retornada por getElementsByClassName e getElementsByTagName) é importante: a NodeList é estática — reflete o DOM no momento da chamada — enquanto a HTMLCollection é live, ou seja, atualiza automaticamente quando o DOM muda, o que pode causar comportamentos inesperados em loops.

A recomendação de boas práticas é simples: prefira querySelector e querySelectorAll para todo código novo. Eles são expressivos, usam a sintaxe CSS que você já conhece, e são suficientemente rápidos para a grande maioria dos casos. Reserve getElementById para situações onde você sabe que precisa do único elemento com um ID específico e quer comunicar essa intenção explicitamente. Evite getElementsByClassName e getElementsByTagName em código novo — o comportamento live da HTMLCollection raramente é o que você quer.

5. Manipulando o DOM

Com um elemento selecionado, você pode modificar qualquer aspecto dele: seu conteúdo de texto, atributos, classes CSS e estilos inline. A forma mais básica de alterar o conteúdo de um elemento é pela propriedade textContent, que define ou lê o texto puro de um nó, ignorando qualquer HTML interno. Já innerHTML define ou lê o conteúdo incluindo tags HTML — o navegador faz o parsing do HTML e atualiza o DOM de acordo. innerHTML é conveniente, mas perigoso quando o conteúdo vem de entrada do usuário ou de fontes externas, pois cria uma brecha de Cross-Site Scripting (XSS).

Perigo de XSS com innerHTML

Nunca insira dados do usuário diretamente em innerHTML. Um usuário pode enviar o texto <img src=x onerror="roubarCookies()"> — e se você colocar isso em innerHTML, o navegador vai executar o script. Use sempre textContent para dados de usuário, ou crie nós DOM com createElement e appendChild, que nunca interpretam HTML.

A forma segura e idiomática de criar novo conteúdo no DOM é com document.createElement seguido de appendChild ou append. Você cria um nó vazio, configura seus atributos e texto, e depois o insere no DOM. Essa abordagem nunca interpreta HTML — o conteúdo é sempre tratado como texto puro, eliminando o risco de XSS por construção. Para manipular classes CSS, a propriedade classList é muito superior a alterar className como string: classList.add('ativo') adiciona sem remover outras classes, classList.remove('ativo') remove apenas aquela, classList.toggle('aberto') alterna e classList.contains('erro') verifica. Manipular estilos inline via element.style.color = 'red' deve ser reservado para valores dinâmicos calculados em runtime — para o restante, altere classes CSS.

Inserir um elemento recém-criado no DOM requer especificar onde ele vai. parent.appendChild(filho) adiciona o filho como último filho do pai. parent.prepend(filho) adiciona como primeiro filho. ref.before(novoEl) e ref.after(novoEl) inserem como irmão antes ou depois de ref. Para remover um elemento, use el.remove(). Para clonar, el.cloneNode(true) cria uma cópia profunda incluindo descendentes. Quando você precisa adicionar vários elementos, use um DocumentFragment para minimizar repaints — adicione todos os nós ao fragment e depois insira o fragment de uma vez.

JavaScript 
const lista = document.querySelector('#produtos-list');

// Forma SEGURA: createElement + textContent
function renderizarProdutos(produtos) {
  lista.innerHTML = ''; // limpar lista antes de repopular
  produtos.forEach(p => {
    const li = document.createElement('li');
    li.textContent = `${p.nome} — R$ ${p.preco.toFixed(2)}`;
    li.dataset.id = p.id;              // atributo data-*
    li.classList.add('produto-item');
    lista.appendChild(li);
  });
}

6. Eventos no DOM

Eventos são o mecanismo pelo qual o browser comunica ao JavaScript o que aconteceu na página: o usuário clicou em um botão, enviou um formulário, moveu o mouse, digitou uma tecla, a página terminou de carregar. O padrão moderno para capturar eventos é element.addEventListener(tipo, handler). O primeiro argumento é uma string com o nome do evento ('click', 'submit', 'input', 'keydown', 'mouseover'). O segundo é uma função — o handler — que será chamada cada vez que o evento ocorrer naquele elemento ou borbulhar (bubble) até ele.

Quando o handler é chamado, ele recebe automaticamente um objeto Event (convencionalmente chamado de e ou event) com informações sobre o que aconteceu. e.target aponta para o elemento que originou o evento — útil em event delegation. e.currentTarget aponta para o elemento no qual o listener está registrado. e.preventDefault() cancela o comportamento padrão do navegador para aquele evento — essencial em formulários para evitar o recarregamento da página. e.stopPropagation() impede que o evento continue subindo na árvore DOM (bubbling), o que deve ser usado com cuidado pois pode quebrar listeners em elementos pai.

Event delegation é uma técnica poderosa e eficiente: em vez de adicionar um listener em cada elemento filho, você adiciona um único listener no elemento pai e usa e.target para identificar qual filho foi clicado. Isso funciona porque eventos borbulham (bubble) do elemento de origem até a raiz do DOM. Event delegation é especialmente valioso quando os filhos são criados dinamicamente — um listener adicionado antes dos filhos existirem não os captura, mas um listener no pai captura filhos futuros também. Isso reduz significativamente o uso de memória em listas longas.

Demonstração interativa — clique no botão e observe o log

const lista = document.querySelector('#produtos-list');

7. Template Literals e Interpolação

Template literals (ou template strings) são strings delimitadas por crases (`) em vez de aspas. A diferença principal é que permitem interpolação de expressões com a sintaxe ${expressao} — qualquer expressão JavaScript válida é avaliada e seu resultado convertido em string e inserido na posição correspondente. Isso torna a criação dinâmica de strings muito mais legível do que concatenação com o operador +. Template literals também preservam quebras de linha, permitindo strings multilinhas sem caracteres de escape.

Na prática do front-end, template literals são frequentemente usados para gerar strings de HTML que serão inseridas via innerHTML. Isso é conveniente para estruturas simples, mas carrega o risco de XSS mencionado anteriormente. Se qualquer parte da string HTML vem de dados externos — banco de dados, API, input do usuário — o conteúdo deve ser sanitizado antes de ser inserido como HTML. A alternativa segura é usar template literals apenas para partes controladas (como classes e IDs estruturais) e definir o conteúdo textual dos elementos via textContent.

Uma técnica híbrida comum é criar o HTML estático via template literal e depois usar querySelector no elemento recém-criado para definir textos dinâmicos com textContent. Essa abordagem é muito mais segura do que embutir dados do usuário diretamente no template HTML, e mais ergonômica do que criar cada elemento individualmente com createElement. O importante é nunca misturar dados não confiáveis diretamente em strings HTML — sempre trate os dados como texto puro.

JavaScript 
const p = { nome: 'Caderno', preco: 29.90, id: 42 };

// Partes controladas no template, dados em textContent
function criarCardSeguro(produto) {
  const div = document.createElement('div');
  div.className = 'card-produto';
  div.dataset.id = produto.id; // id numérico — seguro
  div.innerHTML = `
    <div class="card-nome"></div>
    <div class="card-preco"></div>
  `;
  // Define texto dinamicamente — sem risco de XSS
  div.querySelector('.card-nome').textContent  = produto.nome;
  div.querySelector('.card-preco').textContent = `R$ ${produto.preco.toFixed(2)}`;
  return div;
}

// PERIGOSO: dados do usuário em innerHTML
// container.innerHTML = `<p>${inputUsuario.value}</p>`; // XSS!

// SEGURO: usar textContent
const msg = document.createElement('p');
msg.textContent = nomeDigitadoPeloUsuario; // texto puro, nunca executado como HTML
Regra de ouro

Se o dado veio do usuário ou de um banco de dados, use textContent ou setAttribute. Se o HTML é literalmente escrito por você (estrutura estática), innerHTML com template literal é aceitável. A distinção fundamental é: quem controla os dados?

8. Ciclo Completo: Fetch + Renderização

A combinação de fetch com manipulação de DOM é o padrão central do front-end moderno. O fluxo segue etapas bem definidas: você dispara uma requisição HTTP para uma API, aguarda a resposta (que é uma Promise), faz o parsing do JSON, itera sobre os dados e cria elementos DOM para cada item, e finalmente os insere na página. Ao mesmo tempo, você precisa gerenciar estados intermediários — mostrar um indicador de carregamento enquanto aguarda, e exibir uma mensagem de erro se algo der errado.

O estado de carregamento é frequentemente subestimado, mas é crucial para a experiência do usuário. Uma página que não dá nenhum feedback enquanto carrega parece travada. A solução mais simples é exibir um texto ou spinner antes de iniciar o fetch e removê-lo após a resposta. O atributo aria-live pode ser usado para que leitores de tela anunciem as mudanças de estado. O tratamento de erros deve ser sempre explícito — nunca deixe o catch vazio ou com apenas um console.log; mostre ao usuário uma mensagem acionável.

Uma preocupação de performance frequentemente ignorada por iniciantes é a recriação de elementos DOM a cada atualização. Se a lista de produtos é atualizada frequentemente, limpar o innerHTML e recriar tudo a cada fetch gera flicker (piscar) e é potencialmente lento para listas longas. Estratégias mais sofisticadas envolvem comparar o novo estado com o atual e atualizar apenas o que mudou — é exatamente o que frameworks como React fazem com o Virtual DOM. Para listas estáticas ou de baixa frequência de atualização, a abordagem de limpar e recriar é perfeitamente aceitável.

1
Mostrar loading — exibe spinner ou texto "Carregando..." no container de resultados antes de qualquer requisição
2
fetch(url) — dispara a requisição HTTP; retorna uma Promise. Aguarde com await
3
res.json() — faz o parsing da resposta como JSON; retorna outra Promise. Aguarde com await
4

6

  // 1. Mostrar loading
  lista.innerHTML = '<li aria-live="polite">Carregando...</li>';

  try {
    // 2 e 3. Fetch + JSON parsing
    const res      = await fetch('/api/produtos');
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    const produtos = await res.json();

    // 4 e 5. Renderizar no DOM
    renderizarProdutos(produtos);

  } catch (err) {
    // 6. Exibir erro ao usuário
    lista.innerHTML = '';
    const li = document.createElement('li');
    li.textContent = 'Erro ao carregar produtos. Tente novamente.';
    lista.appendChild(li);
    console.error(err);
  }
}

carregarProdutos();

9. Debugging no Browser

As DevTools do browser (abertas com F12 ou Cmd+Option+I no Mac) são a ferramenta de debugging mais importante do desenvolvedor front-end. Ao contrário de um backend que só pode ser inspecionado por logs e debuggers externos, o front-end tem tudo exposto em tempo real no browser. Aprender a usar as DevTools com proficiência é uma das habilidades que mais diferenciam desenvolvedores experientes de iniciantes.

A aba Elements exibe a árvore DOM atual — não o HTML estático, mas o estado vivo após todas as modificações de JavaScript. Você pode selecionar qualquer elemento, inspecionar suas propriedades CSS computadas, editar atributos em tempo real e até deletar ou mover nós para testar mudanças sem editar o código-fonte. A aba Console é onde aparecem erros de JavaScript, warnings e outputs de console.log. Erros de JavaScript aparecem em vermelho com o arquivo e a linha de origem — sempre leia esses erros antes de tentar adivinhar o problema. A aba Network exibe todas as requisições HTTP feitas pela página — você pode inspecionar o status code, os headers de request e response, o corpo da resposta e o tempo de cada requisição, incluindo as chamadas fetch para sua API.

A aba Sources é onde fica o poder real do debugger. Você pode abrir qualquer arquivo JavaScript carregado pela página e adicionar breakpoints — pontos de parada onde a execução é suspensa e você pode inspecionar o valor de qualquer variável no escopo, navegar pela call stack e executar passo a passo. Breakpoints são muito mais poderosos do que encher o código de console.log — eles deixam o código mais limpo e dão visibilidade completa do estado do programa no momento exato da parada. Uma dica: a instrução debugger; inserida no código JavaScript funciona como um breakpoint programático e ativa automaticamente o painel de Sources quando as DevTools estão abertas.

10. RM-ODP — Front-End nas Visões do Sistema

O RM-ODP oferece cinco visões complementares de qualquer sistema distribuído. Nesta aula, o front-end ocupa um papel central em duas delas. Na visão Computacional, cada elemento HTML visível é a interface de uma funcionalidade — o <form> é a interface da operação "criar produto", a <ul> é a interface da operação "listar produtos". O JavaScript que conecta eventos de usuário às chamadas de API é a camada lógica desta visão: cada handler de evento implementa um contrato funcional. Quando documentamos o front-end na visão computacional, descrevemos as operações que o componente expõe (renderizar, atualizar, capturar input) e seus contratos (pré-condições, pós-condições).

Na visão de Engenharia, o browser é o runtime que executa o front-end, e o DOM é a camada de comunicação entre o código JavaScript e o motor de renderização. O browser não é uma caixa preta — ele é um nó no sistema distribuído com responsabilidades específicas: fazer o parsing do HTML, construir o DOM, executar JavaScript, renderizar a página e gerenciar as conexões de rede para as requisições fetch. Entender o browser como um nó de engenharia ajuda a tomar decisões como onde colocar os scripts (antes de </body> vs defer), como minimizar repaints e como otimizar requisições.

As demais visões também se aplicam ao front-end, mesmo que indiretamente. A visão Enterprise define que o usuário final precisa visualizar, criar e gerenciar produtos — o front-end é o ponto de contato entre esse usuário e o sistema. A visão de Informação descreve como os dados de produto trafegam como JSON da API até o DOM, onde são exibidos como texto. A visão de Tecnologia registra as escolhas concretas: HTML5 semântico, CSS via Bootstrap, JavaScript ES2022, Fetch API nativa, DevTools para debugging.

11. RF, RN e RNF no Contexto do Front-End

Todo elemento visual e interativo da interface é a materialização de um Requisito Funcional. O formulário de cadastro existe porque existe um RF que diz "o usuário autenticado pode cadastrar novos produtos com nome e preço". A lista de produtos existe porque existe um RF que diz "o sistema deve exibir os produtos cadastrados ordenados por nome". Ao projetar o front-end, você está traduzindo RF em elementos HTML, eventos JavaScript e chamadas de API. Se o RF não existe ou está mal escrito, o front-end será igualmente ambíguo.

As Regras de Negócio também se manifestam no front-end, embora não exclusivamente. Uma RN que diz "produto com estoque zero não pode ser adicionado ao carrinho" se traduz em uma verificação JavaScript que desabilita o botão "Adicionar ao carrinho" quando o estoque é zero — e no back-end, em uma validação que rejeita a requisição mesmo que o front seja contornado. A regra no front melhora a UX; a regra no back garante integridade. Ambas são necessárias.

Os Requisitos Não Funcionais do front-end se concentram em três eixos principais nesta aula. O eixo de Usabilidade é atendido pelo HTML semântico e pelo uso correto de labels — a página deve ser navegável por teclado e por leitores de tela, e os formulários devem ter mensagens de validação claras. O eixo de Segurança exige que dados de usuários nunca sejam inseridos via innerHTML sem sanitização, prevenindo XSS. O eixo de Desempenho é favorecido pelo event delegation — um único listener para uma lista de 1.000 itens usa a mesma quantidade de memória que para 1 item, ao contrário de 1.000 listeners individuais.

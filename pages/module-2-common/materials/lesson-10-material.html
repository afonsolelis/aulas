1. Por que testar?

Existe uma metáfora poderosa para os testes de software: o paraquedas. Você só vai descobrir se o paraquedas funciona quando precisar dele — e nesse momento já é tarde demais para consertar. Da mesma forma, um sistema sem testes automatizados pode parecer saudável durante meses e falhar de forma catastrófica justamente no momento mais crítico: o lançamento de uma funcionalidade importante, uma madrugada de alta demanda, ou logo após uma refatoração que parecia inofensiva. Testes existem exatamente para garantir que, quando o momento crítico chegar, você possa confiar no seu sistema.

O custo de um bug aumenta exponencialmente conforme avança no ciclo de vida do software. Um bug descoberto durante o desenvolvimento pelo próprio programador custa minutos para corrigir. O mesmo bug descoberto num code review custa horas. Descoberto em produção, pode custar dias de trabalho, reputação do produto e, em sistemas críticos, dinheiro real de usuários. Estudos clássicos da IBM e outros publicados no SWEBOK apontam que o custo médio de um bug em produção é de 15 a 100 vezes maior do que o custo de detectá-lo com testes automatizados na fase de desenvolvimento. Escrever testes não é overhead — é o investimento que evita um custo muito maior mais tarde.

Testes automatizados cumprem três papéis distintos e complementares. O primeiro é a rede de segurança: ao modificar qualquer parte do código, você roda os testes e descobre imediatamente se quebrou algo que funcionava antes — essa detecção de regressão é o que dá confiança para refatorar. O segundo papel é a documentação viva: um conjunto de testes bem escrito descreve o comportamento esperado do sistema de forma executável e verificável. Diferente de um documento Word que envelhece, os testes falham quando o comportamento muda, tornando a documentação automaticamente inválida quando desatualizada. O terceiro papel é o design force: código difícil de testar é quase sempre código mal projetado. Funções com muitas responsabilidades, acoplamento excessivo, dependências ocultas — tudo isso emerge como dor na hora de escrever testes, incentivando designs mais limpos.

Realidade do mercado

Em equipes de produto maduras, pull requests sem testes adequados são rejeitados no code review — não como capricho, mas como norma de qualidade. Saber escrever testes é hoje tão fundamental quanto saber escrever o código de produção em si.

Existe ainda o argumento da velocidade. Times sem testes muitas vezes alegam que testes "atrasam o desenvolvimento". Isso é verdade no curtíssimo prazo — e completamente falso ao longo de meses. O tempo "economizado" ao não escrever testes é invariavelmente gasto três vezes mais em debugging manual, investigação de regressões e deploys com medo. Times com boa cobertura de testes entregam mais rápido no médio prazo porque podem mover código com confiança, fazer deploys frequentes e detectar problemas antes que o usuário veja.

2. A Pirâmide de Testes

Mike Cohn popularizou a Pirâmide de Testes como um modelo para pensar na distribuição ideal de testes em um sistema. A ideia central é que você deve ter muitos testes unitários (base larga), um número moderado de testes de integração (meio), e poucos testes ponta-a-ponta (topo estreito). O formato de pirâmide não é arbitrário: ele reflete o custo, a velocidade e a amplitude de cada tipo de teste.

E2E / Ponta-a-ponta Poucos · Lentos · Amplos · Caros
Testes de Integração Médio número · Verificam comunicação entre partes
Testes Unitários Muitos · Rápidos · Baratos · Focados

Role a página para animar a pirâmide

Os testes unitários ficam na base porque são os mais numerosos, rápidos e baratos de criar e manter. Um teste unitário verifica uma única unidade de comportamento isoladamente — uma função, um método, um módulo — sem depender de banco de dados, rede ou sistema de arquivos. Quando rodam em milissegundos, você pode executar centenas deles a cada save no editor, sem impacto no fluxo de desenvolvimento. Isso os torna a primeira linha de defesa contra bugs lógicos.

Os testes de integração ficam no meio. Eles verificam que componentes distintos do sistema funcionam corretamente quando conectados. Um teste de integração pode verificar que uma rota Express, ao receber uma requisição válida, persiste corretamente um registro no banco de dados e retorna o status correto. São mais lentos porque envolvem I/O real — banco de dados, sistema de arquivos, rede interna. O número ideal é menor que os unitários, mas suficiente para cobrir os pontos de integração críticos.

Os testes E2E ficam no topo porque são os mais caros em todos os sentidos: mais lentos para executar, mais difíceis de manter, mais frágeis quando a interface muda, mais complexos de configurar. Um teste E2E simula exatamente o que um usuário real faria — abre o navegador, navega pelas páginas, preenche formulários, verifica o resultado. Eles são insubstituíveis para garantir que o fluxo completo funciona end-to-end, mas há um custo real em tê-los em grande quantidade.

O anti-padrão: o sorvete invertido

Quando uma equipe confia principalmente em testes manuais ou E2E e tem poucos testes unitários, a pirâmide se inverte — virando um sorvete. O resultado é uma suíte de testes cara, lenta e frágil que os desenvolvedores evitam rodar. Equipes que chegam aqui muitas vezes simplesmente param de testar, porque o custo de manter os testes supera o benefício percebido.

TipoVelocidadeCusto de manutençãoO que verificaFerramenta (stack Node)
UnitárioMilissegundosBaixoLógica isolada: funções, regras de negócio, validaçõesJest
IntegraçãoSegundosMédioRotas + banco de dados, serviços externos internosJest + Supertest
E2EDezenas de segundosAltoFluxos completos no navegador como usuário realPlaywright, Cypress

3. TDD — Test-Driven Development

O Test-Driven Development (Desenvolvimento Guiado por Testes) é uma prática de desenvolvimento proposta por Kent Beck no contexto do Extreme Programming. A ideia central é contraintuitiva para quem nunca fez: você escreve o teste antes de escrever o código de produção. O ciclo é capturado em três palavras: Red — Green — Refactor.

🔴 Red Escreva o teste.
Veja-o falhar.
→
🟢 Green Escreva o mínimo
para passar.
→
🔵 Refactor Melhore o código
sem quebrar testes.

Role para animar o ciclo

A fase Red consiste em escrever um teste que descreve o comportamento desejado antes de esse comportamento existir. O teste vai falhar — e isso é esperado e necessário. Ver o teste falhar é a confirmação de que o teste está testando algo real. Se o teste passar imediatamente sem código de produção, há algo errado: ou o teste não está exercitando o que você pensa, ou o comportamento já existe por acidente.

A fase Green consiste em escrever o código de produção mais simples possível que faça o teste passar — e nada mais. Isso parece forçado, mas tem um propósito: impede que você escreva código que nenhum teste justifica. Cada linha de código de produção deve existir porque algum teste exige sua existência. Essa restrição força um foco no que o sistema realmente precisa fazer.

A fase Refactor é a que diferencia TDD de simples teste-primeiro. Com o teste verde como rede de segurança, você pode melhorar livremente a estrutura interna do código — extrair funções, renomear variáveis, eliminar duplicação — sabendo que qualquer regressão vai ser detectada imediatamente. Sem testes, refatoração é um exercício de coragem. Com testes, é rotina.

TDD e Design de Software

O benefício mais subestimado do TDD não é a cobertura de testes — é o design melhorado. Quando você escreve o teste antes, é forçado a pensar na interface pública da sua função ou classe antes de pensar na implementação. Isso invariavelmente leva a APIs mais limpas, menos acoplamento e funções com responsabilidade única. Código difícil de testar é, quase sempre, código mal estruturado.

Quando TDD ajuda — e quando é overhead

TDD funciona especialmente bem para lógica de negócio complexa, algoritmos, validações e qualquer código em que os requisitos são claros e estáveis. Nesses casos, o ciclo Red-Green-Refactor é natural e produtivo. Por outro lado, TDD pode ser overhead para protótipos exploratórios (onde você ainda não sabe o que quer construir), para código de infraestrutura que depende de ambiente externo, ou para interfaces de usuário onde o comportamento visual é difícil de especificar em testes antecipadamente.

A resposta pragmática: use TDD como ferramenta, não como dogma. O importante é que, ao final, o código de produção tenha testes que documentam e verificam seu comportamento. Se você chegou lá primeiro escrevendo o teste ou primeiro escrevendo o código é, em muitos casos, secundário ao resultado.

4. Jest — Framework de Testes JavaScript

O Jest é o framework de testes mais adotado no ecossistema JavaScript e Node.js. Criado pelo Facebook (Meta), ele se destaca por uma configuração mínima, integração nativa com coverage, mocks embutidos e uma API de assertions expressiva. Para projetos Node.js com Express, Jest é a escolha padrão do mercado.

Instalação e configuração básica 
# Instalar Jest como dependência de desenvolvimento
npm install --save-dev jest

# Para projetos com módulos ES6 ou TypeScript:
npm install --save-dev jest @types/jest babel-jest @babel/core @babel/preset-env

# Adicionar scripts no package.json:
# "test":          "jest"
# "test:watch":    "jest --watch"
# "test:coverage": "jest --coverage"
jest.config.js — configuração recomendada 
module.exports = {
  // ambiente de execução
  testEnvironment: 'node',

  // padrões de arquivos que o Jest reconhece como testes
  testMatch: [
    '**/__tests__/**/*.js',
    '**/*.test.js',
    '**/*.spec.js'
  ],

  // arquivos/pastas a ignorar
  testPathIgnorePatterns: ['/node_modules/'],

  // limiares de coverage que reprovam o CI se não atingidos
  coverageThreshold: {
    global: {
      branches: 70,
      functions: 70,
      lines: 80,
      statements: 80
    }
  },

  // exibir nome de cada teste no terminal
  verbose: true
};

Estrutura de um arquivo de teste

Jest usa as funções describe, it (ou test), beforeAll, afterAll, beforeEach e afterEach para organizar os testes. A convenção é ter um arquivo *.test.js ou *.spec.js para cada módulo que você quer testar, mantendo os testes próximos do código que testam.

Matchers essenciais

Os matchers são os métodos de assertion que verificam se o valor produzido corresponde ao esperado. Jest oferece dezenas deles, mas um conjunto pequeno cobre a maioria dos casos:

.toBe(valor)
Igualdade estrita com Object.is. Use para primitivos: números, strings, booleans.
expect(soma(2,3)).toBe(5)
.toEqual(objeto)
Igualdade profunda para objetos e arrays. Compara cada propriedade recursivamente.
expect(user).toEqual({id:1, nome:'Ana'})
.toContain(item)
Verifica se array contém item ou string contém substring.
expect(lista).toContain('produto-A')
.toThrow(msg)
Verifica se função lança um erro (com mensagem opcional).
expect(() => fn()).toThrow('inválido')
.toBeTruthy() / .toBeFalsy()
Verifica se valor é truthy ou falsy no sentido do JavaScript.
expect(resultado).toBeTruthy()
.toBeGreaterThan(n)
Comparações numéricas: também existe .toBeLessThan, .toBeCloseTo para floats.
expect(res.body.length).toBeGreaterThan(0)
.toHaveBeenCalledWith(...)
Verifica que uma função mock foi chamada com argumentos específicos.
expect(mockFn).toHaveBeenCalledWith(42)
.resolves / .rejects
Para Promises: encadeia qualquer matcher com valores assíncronos.
await expect(p).resolves.toBe('ok')

5. Testes Unitários na Prática

Um teste unitário ideal é rápido, isolado, determinístico e focado. A palavra-chave é isolado: o teste não deve depender de banco de dados, de rede, de sistema de arquivos ou de qualquer serviço externo. Quando um teste falha, você deve conseguir identificar a causa sem precisar levantar serviços. Essa propriedade é o que torna os testes unitários valiosos como feedback imediato durante o desenvolvimento.

O que vale a pena testar com testes unitários? Funções puras são os candidatos ideais: recebem argumentos, retornam um valor, sem efeitos colaterais. Regras de negócio: cálculo de desconto, validação de CPF, aplicação de regra fiscal, ordenação de resultados. Validações e transformações: sanitização de input, formatação de dados, parsing de strings. Por outro lado, evite testar getters e setters triviais, código de framework que você não controla, ou comportamentos que só fazem sentido com dependências externas — esses casos são cobertos por testes de integração.

O padrão AAA — Arrange, Act, Assert

O padrão AAA é uma convenção de estruturação interna de cada caso de teste. Arrange (preparar): configure os dados de entrada, crie os objetos necessários. Act (agir): execute a função ou operação que está sendo testada. Assert (verificar): confirme que o resultado corresponde ao esperado. Essa separação torna os testes mais legíveis e mais fáceis de diagnosticar quando falham.

utils/calculos.js — função pura 
// Função pura: sem efeitos colaterais, fácil de testar
function calcularDesconto(preco, percentual) {
  if (preco < 0 || percentual < 0 || percentual > 100) {
    throw new Error('Parâmetros inválidos');
  }
  return preco * (1 - percentual / 100);
}

module.exports = { calcularDesconto };
utils/calculos.test.js — testes unitários com padrão AAA 
const { calcularDesconto } = require('./calculos');

describe('calcularDesconto', () => {

  it('aplica 10% de desconto corretamente', () => {
    // Arrange
    const preco = 100;
    const percentual = 10;

    // Act
    const resultado = calcularDesconto(preco, percentual);

    // Assert
    expect(resultado).toBe(90);
  });

  it('aplica 0% sem alterar o preço', () => {
    expect(calcularDesconto(250, 0)).toBe(250);
  });

  it('lança erro para percentual negativo', () => {
    expect(() => calcularDesconto(100, -5)).toThrow('Parâmetros inválidos');
  });

  it('lança erro para percentual maior que 100', () => {
    expect(() => calcularDesconto(100, 150)).toThrow('Parâmetros inválidos');
  });

  it('lança erro para preço negativo', () => {
    expect(() => calcularDesconto(-50, 10)).toThrow('Parâmetros inválidos');
  });
});
Boas práticas de nomenclatura

O nome de cada teste deve ser uma frase legível que descreva o comportamento esperado. "aplica 10% de desconto corretamente" é mais informativo que "teste 1". Quando o teste falha, o nome aparece no relatório — ele deve ser suficiente para entender o que quebrou sem precisar ler o código do teste.

6. Mocks e Stubs

Um dos desafios de testes unitários é lidar com dependências. Código real raramente é uma função pura — ele chama o banco de dados, faz requisições HTTP para APIs externas, envia emails, grava logs em disco. Trazer todas essas dependências para um teste unitário quebra o princípio da isolação: o teste se torna lento, frágil (falha quando a API externa está fora do ar) e não-determinístico (pode produzir resultados diferentes em cada execução).

A solução são os mocks e stubs. Um stub substitui uma dependência por uma implementação falsa que retorna valores pré-determinados. Um mock vai além: além de substituir a dependência, registra como ela foi chamada, permitindo que você verifique não apenas o resultado, mas também o comportamento — quantas vezes a função foi chamada, com quais argumentos, etc. Jest integra ambos na mesma API, acessível pelo objeto jest.

O principal risco dos mocks é testar os próprios mocks em vez do código de produção. Se você cria um mock que retorna exatamente o que o teste espera e nunca valida a interação real com a dependência em um teste de integração, pode ter 100% de cobertura em testes que não detectam bugs reais. Mocks devem ser usados para isolar, não para contornar a complexidade.

Mocks com Jest — jest.fn() e jest.mock() 
// Mockando um model inteiro (todas as funções exportadas)
jest.mock('../models/produtoModel');
const produtoModel = require('../models/produtoModel');

it('retorna 404 quando produto não existe', async () => {
  // Configura o mock para resolver com null (produto não encontrado)
  produtoModel.findById.mockResolvedValue(null);

  const res = await request(app).get('/api/produtos/999');

  expect(res.status).toBe(404);
  expect(produtoModel.findById).toHaveBeenCalledWith('999');
});

// jest.spyOn — mocka apenas um método de um objeto existente
const emailService = require('../services/emailService');
const spy = jest.spyOn(emailService, 'enviarConfirmacao')
  .mockResolvedValue({ sucesso: true });

// Após o teste, restaurar o comportamento original
afterEach(() => spy.mockRestore());
Quando não usar mocks

Não mocke o que você não possui. Evite mockar funções nativas do Node.js ou do sistema operacional a não ser que seja absolutamente necessário. E nunca mocke o sistema que você está testando — se o teste do controller mocka o próprio controller, você não está testando nada.

Aprofundamento: Diferença entre Mock, Stub, Spy, Fake e Dummy

▼

Martin Fowler define esses termos com precisão em seu artigo clássico Mocks Aren't Stubs. Na prática cotidiana, "mock" é usado como termo genérico para qualquer substituto de dependência, mas há distinções úteis:

  • Dummy: objeto passado apenas para preencher a assinatura da função, nunca usado de fato.
  • Stub: fornece respostas pré-programadas para chamadas durante o teste, sem verificar como foi chamado.
  • Spy: como stub, mas também registra chamadas, permitindo verificar depois quantas vezes e com quais argumentos a dependência foi invocada.
  • Mock: substituto com expectativas pré-definidas sobre como deve ser chamado. Falha o teste se essas expectativas não forem atendidas.
  • Fake: implementação funcional alternativa, mais simples que a real. Exemplo: banco em memória ao invés de PostgreSQL.

Jest combina stub e spy em jest.fn() e jest.spyOn(), e o conceito de fake pode ser emulado com jest.mock() fornecendo uma implementação alternativa.

7. Testes de Integração com Banco de Dados

Testes de integração verificam que os componentes do sistema funcionam corretamente quando conectados. No contexto de uma API Express com PostgreSQL, o cenário mais comum é testar rotas que fazem operações no banco: criar um registro, buscar dados, atualizar, deletar. Esses testes precisam de um banco de dados real, o que introduz complexidade na configuração — mas essa complexidade é necessária para garantir que o sistema inteiro funciona como esperado.

O padrão recomendado é ter um banco de dados dedicado para testes, separado do banco de desenvolvimento e completamente isolado do de produção. Esse banco de teste é populado antes dos testes com dados conhecidos e limpo após cada execução. Nunca execute testes de integração contra o banco de produção — o risco de dados corrompidos ou apagados é real.

Jest oferece hooks de ciclo de vida que permitem configurar esse ambiente de forma limpa. beforeAll roda uma única vez antes de todos os testes do describe, ideal para abrir conexão com o banco. afterAll roda ao final, ideal para fechar a conexão. beforeEach e afterEach rodam antes e depois de cada teste individual, ideal para inserir e remover dados de fixture, garantindo que cada teste começa com um estado conhecido.

Teste de integração com Supertest + PostgreSQL 
const request = require('supertest');
const app     = require('../src/app');
const pool    = require('../src/config/database');

describe('GET /api/produtos', () => {

  beforeAll(async () => {
    // Insere fixture de teste no banco
    await pool.query(
      "INSERT INTO produtos (nome, preco) VALUES ('Teste', 99.90)"
    );
  });

  afterAll(async () => {
    // Limpa os dados inseridos pelo teste
    await pool.query("DELETE FROM produtos WHERE nome = 'Teste'");
    // Fecha a conexão para Jest não ficar pendente
    await pool.end();
  });

  it('retorna lista de produtos com status 200', async () => {
    const res = await request(app).get('/api/produtos');
    expect(res.status).toBe(200);
    expect(Array.isArray(res.body)).toBe(true);
    expect(res.body.length).toBeGreaterThan(0);
  });

  it('retorna 404 para produto inexistente', async () => {
    const res = await request(app).get('/api/produtos/99999');
    expect(res.status).toBe(404);
  });

  it('cria produto com POST e retorna 201', async () => {
    const res = await request(app)
      .post('/api/produtos')
      .send({ nome: 'Caneta', preco: 3.50 });
    expect(res.status).toBe(201);
    expect(res.body).toHaveProperty('id');
    expect(res.body.nome).toBe('Caneta');
  });
});

Supertest é uma biblioteca que integra com Express para fazer requisições HTTP sem precisar de um servidor escutando em uma porta real. Ele cria o servidor na memória para cada teste, tornando os testes de integração muito mais rápidos e simples de configurar do que subindo um servidor HTTP separado.

Estratégia com transações

Uma técnica avançada para testes de integração é envolver cada teste em uma transação de banco de dados que é revertida ao final (ROLLBACK), ao invés de inserir e deletar dados manualmente. Isso garante isolamento perfeito entre testes e é especialmente eficaz quando a suíte tem muitos testes de integração.

8. Cobertura de Testes

A cobertura de testes (ou code coverage) mede qual porcentagem do código de produção é exercitada pelos testes. Jest gera relatórios de cobertura com o flag --coverage, usando o Istanbul internamente. O relatório decompõe a cobertura em quatro dimensões: Statements (linhas de código executadas), Branches (caminhos de condicional — if/else, switch, operadores ternários), Functions (funções chamadas ao menos uma vez) e Lines (linhas físicas cobertas).

Gerando o relatório de cobertura 
# Gera relatório no terminal e em /coverage/
npx jest --coverage

# Relatório HTML navegável em coverage/lcov-report/index.html
# Abre no navegador para ver linha a linha quais caminhos não foram cobertos

# Exemplo de saída resumida no terminal:
# File          | % Stmts | % Branch | % Funcs | % Lines
# calculos.js   |   100   |   100    |   100   |   100
# controller.js |    82   |    70    |    90   |    82

A meta de 80% de cobertura de linhas é um ponto de referência razoável e amplamente adotado pela indústria. Abaixo disso, há partes significativas do sistema que não foram verificadas por nenhum teste automatizado. Mas 80% não é um número mágico — contexto importa. Um sistema de pagamentos pode exigir 95%; um script de migração pontual pode justificar menos.

A falácia dos 100% de cobertura é importante entender. Cobertura mede execução de código, não qualidade dos testes. É trivialmente fácil atingir 100% de cobertura com testes que não verificam nada relevante — basta chamar todas as funções sem nenhum expect significativo. Por isso, cobertura alta é condição necessária mas não suficiente para qualidade. O que importa é que as verificações (assertions) capturam comportamentos reais e relevantes do sistema.

O relatório de cobertura HTML mostra linha a linha quais caminhos não foram exercitados. Use isso como guia: se um branch de tratamento de erro nunca foi testado, pergunte-se se esse caminho pode acontecer em produção. Se sim, escreva o teste. Se não, talvez o código possa ser simplificado.

9. Playwright — Testes E2E

Testes E2E (ponta-a-ponta) simulam um usuário real usando o sistema: abrem um navegador real, navegam entre páginas, interagem com elementos, preenchem formulários e verificam resultados visíveis. Eles são a camada mais alta da pirâmide e a que mais se aproxima da perspectiva do usuário final. Um teste E2E bem escrito captura cenários que os testes unitários e de integração não conseguem verificar — como interações de formulário, redirecionamentos, fluxos multipage e comportamentos específicos do navegador.

Playwright é a ferramenta criada pela Microsoft para automação de browsers, e tornou-se a principal alternativa ao Selenium e Cypress no mercado. Suas vantagens incluem suporte nativo a Chromium, Firefox e WebKit, uma API moderna baseada em Promises, execução paralela automática de testes, capturas de screenshot e vídeo em falhas, e modos de depuração interativa. Para projetos Node.js, a integração é direta via @playwright/test.

Este próprio projeto educacional usa Playwright para garantir que os links do index.html apontam para arquivos que existem em disco, que as cores dos cards de módulo seguem o design system, e que as convenções de nomenclatura de arquivos são respeitadas. Os testes em tests/index.spec.ts e tests/specs-compliance.spec.ts rodam automaticamente como pre-commit hook via Husky — nenhum commit passa sem que esses testes estejam verdes.

Exemplo de teste E2E com Playwright 
import { test, expect } from '@playwright/test';

// Teste que verifica fluxo de navegação completo
test('usuário consegue adicionar produto ao carrinho', async ({ page }) => {
  // Navega para a página inicial
  await page.goto('http://localhost:3000');

  // Verifica que a lista de produtos está visível
  await expect(page.locator('.produto-card')).toHaveCount(3);

  // Clica no botão "Adicionar" do primeiro produto
  await page.locator('.produto-card').first()
    .getByRole('button', { name: 'Adicionar' })
    .click();

  // Verifica que o contador do carrinho foi atualizado
  await expect(page.locator('#carrinho-contador'))
    .toHaveText('1');
});

// Verificação de convenção (estilo deste projeto)
test('link do módulo aponta para arquivo existente', async ({ page }) => {
  await page.goto('file:///caminho/para/index.html');
  const href = await page.locator('.module-card a').first()
    .getAttribute('href');
  // Garante kebab-case e prefixo home- conforme CLAUDE.md
  expect(href).toMatch(/^pages\/home-/);
});
Playwright no CI

Em pipelines de CI/CD, Playwright roda em modo headless (sem janela visível) e pode capturar screenshots e vídeos automaticamente em caso de falha. Configure com use: { screenshot: 'only-on-failure', video: 'on-first-retry' } no playwright.config.ts para facilitar debugging de falhas que só acontecem no ambiente de integração contínua.

10. RM-ODP e Testes Automatizados

O RM-ODP (Reference Model for Open Distributed Processing) oferece cinco visões complementares para descrever sistemas distribuídos. Nesta aula, os testes automatizados se conectam especialmente a dois viewpoints: o Engineering e o Computational.

🔧
Engineering (Engenharia)
Esta visão descreve como os componentes se distribuem e comunicam: processos, canais, protocolos. Os testes automatizados habitam essa camada como a infraestrutura de qualidade do CI/CD. Um pipeline de integração contínua que roda Jest e Playwright a cada push é um artefato da visão Engineering: garante que as interações entre componentes (rota, banco, cliente) se mantêm corretas conforme o sistema evolui.
Testes de integração, CI/CD, pre-commit hooks
⚙
Computational (Computacional)
Esta visão descreve a decomposição funcional do sistema em componentes com interfaces bem definidas. Os testes unitários e de integração são, nessa perspectiva, a especificação executável das interfaces: cada caso de teste descreve o contrato de uma função ou rota — quais inputs são aceitos, quais outputs são produzidos, quais erros são lançados. Isso torna os testes uma documentação viva e verificável dos contratos computacionais do sistema.
Testes unitários, mocks de interfaces, contratos de API

Pensar em testes pela lente do RM-ODP ajuda a entender por que testes em diferentes níveis da pirâmide respondem perguntas diferentes. Os testes unitários verificam os contratos computacionais de cada componente individualmente. Os testes de integração verificam os canais de comunicação entre componentes na visão Engineering. Os testes E2E verificam os fluxos empresariais da visão Enterprise — o que o usuário final precisa que funcione.

11. RF, RN e RNF — Testes como Verificação de Requisitos

Os testes automatizados não existem no vácuo — eles são a forma mais concreta de verificar se os requisitos do sistema estão sendo atendidos. Cada tipo de requisito tem um tipo de teste correspondente.

Os Requisitos Funcionais (RF) descrevem o que o sistema deve fazer. Os testes de integração e E2E são a principal ferramenta de verificação: um teste que faz POST /api/pedidos e verifica que o status 201 é retornado com o pedido persistido está verificando diretamente um RF. Quando um RF muda — o fluxo de checkout é modificado, uma nova validação é adicionada — os testes correspondentes devem falhar, forçando uma atualização explícita.

As Regras de Negócio (RN) são as restrições que limitam o comportamento do sistema dentro do domínio. Testes unitários são os mais eficazes para verificá-las, pois as RNs tipicamente existem em funções puras de lógica de domínio. A função calcularDesconto dos exemplos anteriores implementa uma RN — e seus testes verificam explicitamente que as restrições do domínio (desconto entre 0 e 100, preço não negativo) são respeitadas.

CONF
Confiabilidade
Testes de regressão detectam bugs antes do usuário. CI impede que código quebrado chegue à produção.
"Suíte de 200 testes unitários roda em menos de 30s. Qualquer falha bloqueia o merge."
SUP
Suportabilidade
Testes automatizados são a rede de segurança que torna refatorações seguras e mudanças de design viáveis.
"Coverage maior que 80%. Refatoração do model de produtos sem quebrar nenhum teste."
CAP
Capacidade
Testes de integração com fixtures de volume verificam que o banco e as rotas comportam carga esperada sem degradação.
"Teste de rota com 1000 produtos no banco: resposta em menos de 200ms."
RNF de Testes bem escritos

Testes também podem ser alvos de RNF. Exemplos bem formados:

CONF: A suíte de testes unitários deve executar completamente em menos de 60 segundos em qualquer máquina de desenvolvimento, validado por medição na pipeline de CI.

SUP: Qualquer nova funcionalidade deve ser acompanhada de testes com cobertura de branches superior a 80%, evidenciada pelo relatório Jest do pull request.

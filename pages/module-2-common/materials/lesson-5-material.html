1. O que é Node.js?

Durante décadas, JavaScript viveu exclusivamente dentro do navegador. Tudo que você podia fazer com JavaScript era manipular o DOM, responder a eventos do usuário e, eventualmente, fazer requisições AJAX. O servidor era território exclusivo de linguagens como Java, PHP, Ruby e Python. Em 2009, um engenheiro chamado Ryan Dahl mudou esse cenário ao criar o Node.js — um ambiente de execução JavaScript fora do navegador, construído sobre o motor V8 do Google Chrome.

A motivação original de Dahl era resolver um problema que ele enxergava no Apache HTTP Server: ao lidar com muitas conexões simultâneas, servidores tradicionais bloqueavam uma thread por requisição. Se a requisição precisasse ler um arquivo do disco ou esperar uma resposta do banco de dados, a thread ficava parada, desperdiçando recursos. A solução de Dahl foi levar para o servidor o modelo que o navegador já usava há anos: event-driven non-blocking I/O. Em vez de bloquear uma thread esperando o banco responder, o Node.js registra um callback e continua processando outras requisições. Quando o banco responde, o evento é adicionado à fila e processado.

Esse modelo torna o Node.js excepcionalmente eficiente para aplicações I/O bound — aquelas cujo gargalo é esperar operações de entrada e saída (rede, disco, banco de dados) — em contraste com aplicações CPU bound, onde o gargalo é processamento intensivo. Para APIs REST que leem e escrevem no banco, o Node.js é uma escolha excelente. Empresas como Netflix, Uber, LinkedIn e PayPal adotaram Node.js exatamente por esse motivo: alta concorrência com baixo consumo de memória. O LinkedIn relatou redução de 10x no número de servidores após migrar de Ruby on Rails para Node.js.

Definição: Event Loop

O Node.js tem uma única thread principal que executa código JavaScript. Operações demoradas (I/O) são delegadas ao sistema operacional; quando terminam, um callback é colocado na fila de eventos e executado assim que a thread estiver livre. Isso permite que um único processo Node.js sirva milhares de conexões simultâneas.

JavaScript no servidor: vantagens práticas

Além do desempenho, há um benefício prático enorme em usar JavaScript tanto no front-end quanto no back-end: compartilhamento de código e contexto mental. Um time full-stack pode reutilizar funções de validação, tipos de dados e lógica de negócio entre cliente e servidor. Isso reduz duplicação de código e elimina a fricção de alternar entre duas linguagens com paradigmas diferentes.

2. npm e o Ecossistema de Pacotes

O npm (Node Package Manager) é o gerenciador de pacotes padrão do Node.js e também o maior repositório de software open-source do mundo, com mais de 2 milhões de pacotes publicados. Quando você instala o Node.js, o npm vem junto. Com ele, você pode adicionar dependências ao seu projeto com um único comando — sem baixar arquivos ZIP, sem configurar classpath, sem copiar JARs manualmente.

O coração de qualquer projeto Node.js é o arquivo package.json. Ele descreve o projeto: nome, versão, scripts que podem ser executados, dependências de produção (dependencies) e dependências apenas para desenvolvimento (devDependencies). A distinção importa: em produção, você não quer instalar ferramentas de teste ou formatadores de código. Quando executa npm install --production, apenas as dependencies são instaladas. Quando executa npm install em ambiente de desenvolvimento, ambas são instaladas.

O arquivo package-lock.json é gerado automaticamente e nunca deve ser editado manualmente. Ele registra a versão exata de cada pacote — e de cada dependência transitiva — que foi instalada. Isso garante que todos no time e os ambientes de CI/CD instalem exatamente as mesmas versões, eliminando o clássico problema "funciona na minha máquina". Já a pasta node_modules, onde os pacotes são instalados fisicamente, nunca deve ser versionada no Git — ela pode facilmente ter centenas de megabytes.

Aviso: Nunca versione node_modules

Adicione node_modules/ ao .gitignore imediatamente. Qualquer pessoa que clonar o repositório pode recuperar os pacotes rodando npm install, que lerá o package-lock.json e instalará as versões exatas.

Semântica de versões

O npm usa Semantic Versioning (SemVer): MAJOR.MINOR.PATCH. O prefixo no package.json controla quais atualizações são aceitas automaticamente. ^1.2.3 aceita atualizações que não quebrem compatibilidade (minor e patch), fixando o major. ~1.2.3 é mais conservador: aceita apenas atualizações de patch, fixando major e minor. Para dependências críticas em produção, considere fixar a versão exata (1.2.3 sem prefixo) para máximo controle.

3. Express.js

O Node.js puro fornece módulos nativos para criar servidores HTTP, mas o código resultante é verboso e difícil de manter. O Express.js é um micro-framework minimalista que torna a criação de servidores HTTP em Node.js muito mais ergonômica. "Micro-framework" significa que ele faz apenas o essencial — roteamento e middleware — sem impor uma estrutura completa. Você tem liberdade para organizar seu projeto como quiser, e adicionar apenas os pacotes que precisa.

O conceito central do Express é o middleware. Um middleware é uma função com a assinatura (req, res, next) que tem acesso ao objeto de requisição, ao objeto de resposta e à função next que passa o controle para o próximo middleware na pilha. Quando uma requisição chega ao servidor, ela percorre sequencialmente todos os middlewares registrados até que um deles envie uma resposta com res.json(), res.send() ou similar. Isso cria uma pipeline composável: você pode adicionar logging, autenticação, validação e parsing de JSON como middlewares independentes, cada um com uma única responsabilidade.

O roteamento no Express mapeia métodos HTTP e caminhos de URL para funções handler. app.get('/api/usuarios', handler) define que requisições GET para aquela rota chamam aquela função. Os objetos req e res encapsulam respectivamente a requisição recebida (com propriedades como req.body, req.params, req.query, req.headers) e a resposta que será enviada (res.json(), res.status(), res.send()).

server.js — entrada da aplicação
const express = require('express');
const app = express();

// Middleware global: parseia corpo JSON automaticamente
app.use(express.json());

// Middleware de logging simples
app.use((req, res, next) => {
  console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`);
  next(); // passa para o próximo middleware
});

// Rotas (definidas em módulos separados)
const usuariosRoutes = require('./routes/usuarios');
app.use('/api/usuarios', usuariosRoutes);

// Middleware de erro global (4 parâmetros = tratador de erro)
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).json({ erro: 'Erro interno do servidor' });
});

const PORT = process.env.PORT || 3000;
app.listen(PORT, () => console.log(`Servidor rodando na porta ${PORT}`));
Dica: A ordem dos middlewares importa

O Express processa middlewares na ordem em que são registrados com app.use(). O middleware de parsing (express.json()) deve vir antes das rotas que precisam ler req.body. O middleware de erro deve ser o último, com 4 parâmetros (err, req, res, next).

Roteamento com Express Router

Para organizar as rotas, o Express oferece express.Router(), que cria um mini-aplicativo com seu próprio conjunto de rotas. Isso permite separar as rotas de usuários, produtos, pedidos etc. em arquivos distintos, e montar todos no arquivo principal com app.use(). O resultado é um servidor fácil de navegar e manter.

routes/usuarios.js
const express = require('express');
const router = express.Router();
const usuarioController = require('../controllers/usuarioController');

// GET /api/usuarios — lista todos
router.get('/', usuarioController.listar);

// GET /api/usuarios/:id — busca por id
router.get('/:id', usuarioController.buscarPorId);

// POST /api/usuarios — cria novo
router.post('/', usuarioController.criar);

// PUT /api/usuarios/:id — atualiza por completo
router.put('/:id', usuarioController.atualizar);

// DELETE /api/usuarios/:id — remove
router.delete('/:id', usuarioController.remover);

module.exports = router;

4. Arquitetura MVC no Contexto de APIs

Conforme um projeto cresce, a tentação de misturar tudo — query SQL, validação de input, lógica de negócio e formatação de resposta — em um único arquivo ou função torna-se cada vez mais problemática. O padrão arquitetural MVC (Model-View-Controller), criado nos anos 1970 no contexto de interfaces gráficas, foi adaptado ao mundo de APIs REST para resolver exatamente esse problema: separação de responsabilidades.

Em uma API REST, o papel de cada componente é bem definido. O Model é a camada que fala com o banco de dados. Ele encapsula as queries SQL, a lógica de persistência e os relacionamentos entre entidades. Ele não sabe nada sobre HTTP, status codes ou o formato da resposta — só sabe executar operações sobre os dados. O Controller é o intermediário: recebe a requisição HTTP, extrai e valida os dados recebidos, chama os métodos do Model que precisa, e formata e retorna a resposta. A View, em APIs REST, não é um template HTML — é o próprio JSON retornado. Às vezes nem existe como componente separado; o controller simplesmente chama res.json(dados).

Os benefícios dessa separação são concretos e imediatos. Testabilidade: você pode testar o Model isolado (mock do banco) e o Controller isolado (mock do Model) sem precisar subir um servidor real. Manutenibilidade: quando precisa mudar uma query SQL, você abre apenas o Model — o Controller nem sabe que query foi modificada. Reusabilidade: o mesmo Model pode ser usado por diferentes Controllers (um controller HTTP e um worker de background queue).

Aprofundamento: MVC vs outras arquiteturas

MVC é um ponto de partida excelente. Projetos maiores podem evoluir para arquiteturas em camadas com Services (lógica de negócio complexa entre Controller e Model), Repositories (abstração da camada de persistência) ou Clean Architecture. Por enquanto, MVC oferece a separação necessária sem complexidade desnecessária.

5. Estrutura de Pastas do Projeto

A organização física dos arquivos é a primeira coisa que outro desenvolvedor vê ao clonar um repositório. Uma estrutura bem definida comunica a arquitetura do projeto antes de qualquer leitura de código. Para projetos Node.js com Express e arquitetura MVC, existe uma convenção amplamente adotada que separa as responsabilidades por diretório.

O arquivo server.js (ou app.js) na raiz é o ponto de entrada da aplicação. Ele é responsável por criar a instância do Express, registrar middlewares globais, montar as rotas e iniciar o servidor na porta configurada. Dentro da pasta src/, o código é dividido em três diretórios principais: models/ contém as funções que fazem queries no banco; controllers/ contém os handlers das rotas; routes/ define o mapeamento URL para controller. Separar rotas de controllers é importante: a rota apenas mapeia URLs; o controller contém a lógica.

Arquivos de configuração ficam em config/ — como a configuração do pool de conexão com o banco. Na raiz, além do server.js, ficam .env (nunca versionado), .env.example (versionado, com valores fictícios como documentação), package.json, package-lock.json e .gitignore. A convenção de nomenclatura para arquivos JavaScript é camelCase: usuarioModel.js, usuarioController.js, database.js.

meu-projeto/ server.js — ponto de entrada, inicializa Express .env — variáveis de ambiente (NAO versionar) .env.example — template com chaves sem valores reais .gitignore — inclui node_modules/ e .env package.json package-lock.json src/ config/ database.js — configura pool de conexao pg models/ usuarioModel.js — queries SQL de usuarios produtoModel.js controllers/ usuarioController.js produtoController.js routes/ usuarios.js — define router, importa controller produtos.js
Dica: Por que separar routes/ de controllers/?

A rota responde à pergunta "qual URL aciona qual função?". O controller responde a "o que fazer quando essa função é chamada?". Se você colocar os dois juntos, mistura configuração de infraestrutura HTTP com lógica de negócio — mais difícil de testar e manter. Separar os dois permite testar o controller chamando a função diretamente, sem precisar simular uma requisição HTTP.

6. Variáveis de Ambiente (.env)

Imagine que você hardcoda a senha do banco de dados no código: const senha = 'minha_senha_super_secreta'. Ao subir para o GitHub, essa senha fica exposta para qualquer pessoa que acesse o repositório. Bots automatizados varrem repositórios públicos buscando credenciais expostas e as exploram em minutos. Isso não é hipotético — acontece com frequência. A solução é nunca hardcodar credenciais e sempre usar variáveis de ambiente.

O pacote dotenv permite carregar variáveis de um arquivo .env na raiz do projeto para o objeto process.env do Node.js. Basta chamar require('dotenv').config() no início do server.js, antes de qualquer código que use variáveis de ambiente. O arquivo .env nunca deve ser versionado — adicione-o imediatamente ao .gitignore. Em contrapartida, o arquivo .env.example deve ser versionado: ele documenta quais variáveis o projeto precisa, com valores fictícios ou vazios, para que qualquer desenvolvedor que clonar o repositório saiba o que precisa configurar.

Em ambientes de produção (serviços de cloud como Railway, Render, Heroku, AWS), as variáveis de ambiente são configuradas diretamente no painel do serviço, sem arquivo .env. O código não muda — ele sempre lê de process.env. Essa abstração é o que permite o mesmo código rodar identicamente no ambiente local, em staging e em produção, alterando apenas as configurações de ambiente.

Segurança: .env nunca no Git

Se por acidente você subir um .env com credenciais reais para o GitHub, considere aquelas credenciais comprometidas imediatamente. Mesmo que você delete o commit logo em seguida, o histórico do Git é permanente e bots provavelmente já escanearam o repositório. A ação correta é revogar e regenerar todas as credenciais expostas imediatamente.

7. Models — A Camada de Dados

O Model é o componente responsável por toda comunicação com o banco de dados. Isso significa que as queries SQL vivem exclusivamente nos arquivos de model — não devem aparecer em controllers, rotas ou qualquer outro lugar. Essa concentração torna fácil revisar todas as operações de banco do sistema, otimizar queries, adicionar índices ou migrar de banco de dados sem precisar caçar SQL espalhado por todo o código.

A conexão com o PostgreSQL é feita através de um pool de conexões gerenciado pelo pacote pg. Um pool mantém um conjunto de conexões abertas com o banco, reutilizando-as entre as requisições. Sem um pool, cada requisição abriria e fecharia uma conexão TCP com o banco — operação cara, com latência de dezenas de milissegundos. Com o pool, a conexão já está estabelecida e pronta. Isso é fundamental para o RNF de Desempenho: o pool é configurado uma vez em config/database.js e importado por todos os models.

Todas as funções do model são assíncronas, usando async/await. Operações de banco de dados são I/O — podem demorar de microsegundos a segundos. Usando await no método pool.query(), o Node.js libera a thread principal para processar outras requisições enquanto espera o banco responder. O objeto retornado por pool.query() contém uma propriedade rows com os registros retornados. Para queries que retornam um único registro (busca por ID, INSERT com RETURNING), acessa-se rows[0].

src/models/usuarioModel.js
const pool = require('../config/database');

// Retorna todos os usuários
async function findAll() {
  const { rows } = await pool.query('SELECT * FROM usuarios ORDER BY nome');
  return rows;
}

// Retorna usuário por ID (null se não existir)
async function findById(id) {
  const { rows } = await pool.query(
    'SELECT * FROM usuarios WHERE id = $1', [id]
  );
  return rows[0] || null;
}

// Cria novo usuário e retorna o registro criado
async function create(dados) {
  const { nome, email } = dados;
  const { rows } = await pool.query(
    'INSERT INTO usuarios (nome, email) VALUES ($1, $2) RETURNING *',
    [nome, email]
  );
  return rows[0];
}

// Atualiza usuário e retorna o registro atualizado
async function update(id, dados) {
  const { nome, email } = dados;
  const { rows } = await pool.query(
    'UPDATE usuarios SET nome=$1, email=$2 WHERE id=$3 RETURNING *',
    [nome, email, id]
  );
  return rows[0] || null;
}

// Remove usuário; retorna true se existia
async function remove(id) {
  const { rowCount } = await pool.query(
    'DELETE FROM usuarios WHERE id = $1', [id]
  );
  return rowCount > 0;
}

module.exports = { findAll, findById, create, update, remove };
Queries parametrizadas previnem SQL Injection

Nunca concatene valores do usuário diretamente na string SQL: 'SELECT * FROM usuarios WHERE id = ' + id. Sempre use parâmetros posicionais ($1, $2...). O driver pg escapa os valores automaticamente, tornando SQL Injection impossível nessa query.

8. Controllers — A Camada de Lógica

O Controller é o ponto de contato entre o mundo HTTP e a lógica da aplicação. Ele recebe o objeto req (requisição), extrai os dados necessários, os valida minimamente, delega o trabalho ao Model, e usa o objeto res para enviar a resposta apropriada. Um Controller bem escrito não tem SQL, não acessa o banco diretamente, e não tem lógica de rendering — ele orquestra.

A responsabilidade de validação no controller é importante: antes de chamar o Model, verifique se os campos obrigatórios estão presentes e têm o formato correto. Se req.body.email está ausente, responda imediatamente com 400 Bad Request — não propague dados inválidos até o banco. Isso economiza uma round-trip ao banco e retorna um erro mais descritivo ao cliente.

Todo controller deve usar try/catch para tratar erros inesperados. Se o Model lançar uma exceção (falha de conexão, violação de constraint, timeout), o catch a intercepta e retorna 500 Internal Server Error sem expor detalhes internos ao cliente. Internamente, o erro deve ser logado para diagnóstico. A separação entre "o que responder ao cliente" (mensagem amigável, sem detalhes técnicos) e "o que logar no servidor" (stack trace completo) é uma boa prática de segurança.

src/controllers/usuarioController.js
const usuarioModel = require('../models/usuarioModel');

// GET /api/usuarios — lista todos
async function listar(req, res) {
  try {
    const usuarios = await usuarioModel.findAll();
    res.json(usuarios);               // 200 OK implícito
  } catch (err) {
    console.error(err);
    res.status(500).json({ erro: 'Erro ao buscar usuários' });
  }
}

// GET /api/usuarios/:id — busca por id
async function buscarPorId(req, res) {
  try {
    const usuario = await usuarioModel.findById(req.params.id);
    if (!usuario) return res.status(404).json({ erro: 'Usuário não encontrado' });
    res.json(usuario);
  } catch (err) {
    console.error(err);
    res.status(500).json({ erro: 'Erro interno' });
  }
}

// POST /api/usuarios — cria novo
async function criar(req, res) {
  const { nome, email } = req.body;

  // Validação de campos obrigatórios
  if (!nome || !email) {
    return res.status(400).json({ erro: 'nome e email são obrigatórios' });
  }

  try {
    const novoUsuario = await usuarioModel.create({ nome, email });
    res.status(201).json(novoUsuario); // 201 Created
  } catch (err) {
    console.error(err);
    res.status(500).json({ erro: 'Erro ao criar usuário' });
  }
}

module.exports = { listar, buscarPorId, criar };

9. Async/Await e Promises

JavaScript nasceu com um modelo de programação assíncrona baseado em callbacks: funções passadas como argumento que seriam chamadas quando uma operação terminasse. Para operações simples, callbacks funcionam bem. Mas para sequências de operações dependentes (buscar usuário, buscar pedidos, buscar produtos de cada pedido), os callbacks se aninham infinitamente, criando o famoso callback hell — código ilegível, difícil de manter e de tratar erros.

As Promises, introduzidas no ES6, melhoraram a situação ao representar o resultado futuro de uma operação assíncrona como um objeto. Com Promises, operações sequenciais podiam ser encadeadas com .then() e erros tratados com .catch(). O código ficou mais legível, mas ainda tinha uma diferença sintática marcante em relação ao código síncrono.

O async/await, introduzido no ES2017, é açúcar sintático sobre Promises que permite escrever código assíncrono com aparência de código síncrono. Uma função marcada com async sempre retorna uma Promise. Dentro dela, await pausa a execução daquela função (sem bloquear a thread principal) até que a Promise resolvida resolva. Para tratar erros, usa-se o familiar try/catch. O resultado é código assíncrono que lê de cima para baixo, como código síncrono, tornando o raciocínio sobre o fluxo muito mais direto.

1
Requisição chega. Controller é chamado — começa a executar a função async listar(req, res).
2
await usuarioModel.findAll() — o controller pausa aqui. A thread fica livre para processar outras requisições.
3
O Node.js delega a query ao driver pg, que envia SQL ao PostgreSQL pela rede. Zero bloqueio de thread.
4
PostgreSQL processa, retorna os dados. O driver pg resolve a Promise.
5
O controller retoma a execução. A variável usuarios agora contém os dados. res.json(usuarios) envia a resposta.
Callbacks vs Promises vs Async/Await
// Callback hell (evitar)
db.query('SELECT * FROM usuarios', (err, result) => {
  if (err) { console.error(err); return; }
  db.query('SELECT * FROM pedidos WHERE usuario_id = $1', [result.rows[0].id], (err2, result2) => {
    // ... mais níveis de aninhamento
  });
});

// Async/Await (preferido)
async function buscarUsuarioComPedidos(id) {
  const { rows: [usuario] } = await pool.query('SELECT * FROM usuarios WHERE id = $1', [id]);
  const { rows: pedidos } = await pool.query('SELECT * FROM pedidos WHERE usuario_id = $1', [id]);
  return { ...usuario, pedidos };
}

// Promise.all: operações paralelas (mais rápido)
async function buscarDadosDashboard() {
  const [usuarios, pedidos, produtos] = await Promise.all([
    usuarioModel.findAll(),
    pedidoModel.findAll(),
    produtoModel.findAll(),
  ]);
  return { usuarios, pedidos, produtos };
}
Dica: Promise.all para operações independentes

Se você precisa buscar dados de múltiplas tabelas que não dependem umas das outras, use Promise.all(). Isso dispara todas as queries em paralelo — o tempo total é o máximo entre elas, não a soma. Se você usar await sequencialmente para operações independentes, está desperdiçando tempo.

10. RM-ODP — Visões Computacional e de Engenharia

Nas aulas anteriores, introduzimos as 5 visões do RM-ODP. Nesta aula, duas delas ganham destaque concreto: a visão Computacional e a visão de Engenharia. Compreender como a arquitetura MVC se mapeia para essas visões ajuda a desenvolver o hábito de pensar o sistema a partir de múltiplas perspectivas — não apenas como código, mas como um sistema distribuído com componentes, contratos e infraestrutura.

A visão Computacional do RM-ODP descreve o sistema como uma coleção de objetos computacionais com interfaces bem definidas. Um objeto computacional não é uma classe Java ou um objeto JavaScript literal — é uma unidade funcional do sistema que expõe operações e aceita interações. Na nossa arquitetura: o Model é um objeto computacional que expõe operações como findAll(), findById(id), create(dados). O Controller é outro objeto que expõe as operações HTTP correspondentes. A interface entre eles é o conjunto de funções assíncronas que o Model exporta via module.exports.

A visão de Engenharia foca em como os componentes se comunicam na infraestrutura real: qual porta o servidor escuta, quantos processos rodam, como é feito o balanceamento de carga. O Node.js, por padrão, é um único processo em uma única porta. Em produção, ferramentas como pm2 permitem criar múltiplas instâncias do processo (cluster mode) para aproveitar todos os núcleos da CPU, com o sistema operacional distribuindo as conexões entre elas. O pool de conexões do banco também pertence à visão de Engenharia: é uma preocupação de binding entre o processo Node.js e o processo PostgreSQL.

⚙️
Visão Computacional — Foco desta Aula
Decomposição do sistema em componentes com interfaces: Routes (ponto de entrada HTTP), Controllers (processam requisições, definem contratos de entrada/saída), Models (operações sobre dados, encapsulam SQL). Cada componente tem uma interface explícita — as funções exportadas via module.exports.

🔧
Visão de Engenharia — Infraestrutura do Node.js
O servidor Node.js é um processo que escuta em uma porta TCP (ex: 3000). O pool de conexões mantém N conexões abertas com o PostgreSQL. Em produção, pm2 cria clusters de processos. Variáveis de ambiente configuram o binding sem alterar código.

Aprofundamento: o Event Loop como mecanismo de Engineering

▼

Na visão de Engenharia do RM-ODP, o mecanismo de execução é tão importante quanto os canais de comunicação. O Event Loop do Node.js é o mecanismo central: uma thread executa callbacks em fila. As fases do Event Loop incluem timers (setTimeout), I/O callbacks, setImmediate e outros. Entender que o Node.js é single-threaded mas não blocking explica por que ele pode servir milhares de conexões simultâneas: a thread principal nunca fica parada esperando I/O.

Na prática, isso significa que um loop infinito ou processamento pesado de CPU dentro de um controller bloqueia o servidor inteiro para todos os clientes enquanto executa — porque é CPU bound, não I/O bound. Para processamento intensivo de CPU, a solução é usar Worker Threads ou delegar para um serviço separado.

11. RF, RN e RNF — Mapeando para Código

Na aula 1, introduzimos os conceitos de Requisitos Funcionais (RF), Regras de Negócio (RN) e Requisitos Não Funcionais (RNF). Agora que você conhece a arquitetura MVC com Node.js, é possível fazer um mapeamento concreto: cada RF se traduz em um conjunto de endpoints, e cada endpoint tem um Controller e um Model correspondente. Esse rastreamento — do requisito até a linha de código — é o que torna o desenvolvimento rastreável e manutenível.

As Regras de Negócio têm lugar específico na arquitetura: no Controller (quando são simples, como validar formato de email) ou em uma camada de Service (quando são complexas, como calcular desconto com múltiplas condições). Nunca em SQL — queries devem ser declarativas, sem lógica de negócio embutida. RNs em SQL são difíceis de testar, difíceis de reutilizar e se tornam silenciosas ao mudar de banco.

Os Requisitos Não Funcionais também têm implementação concreta. O RNF de Desempenho é atendido em parte pelo pool de conexões (evita overhead de abertura de conexão por requisição) e pelo modelo não-bloqueante do Node.js. O RNF de Suportabilidade é atendido pela estrutura MVC: um novo desenvolvedor sabe exatamente onde encontrar as queries (models/) e a lógica (controllers/), reduzindo o tempo de onboarding. O RNF de Segurança é atendido pelas queries parametrizadas (prevenção de SQL Injection) e pelas variáveis de ambiente (proteção de credenciais).

RF-01
Listar usuários
O sistema deve retornar todos os usuários cadastrados.
GET /api/usuarios → controller.listar → model.findAll
RF-02
Buscar usuário por ID
O sistema deve retornar um usuário pelo seu identificador único.
GET /api/usuarios/:id → controller.buscarPorId → model.findById
RF-03
Criar usuário
O sistema deve registrar um novo usuário com nome e email.

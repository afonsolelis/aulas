1. A virada: a IA escreve, você audita

Algo mudou de forma silenciosa e definitiva no jeito de programar. Hoje, ainda no primeiro semestre, ferramentas como Copilot, ChatGPT e Claude escrevem boa parte do código que entra nos projetos. Você descreve a intenção em uma frase e recebe uma função inteira, pronta, com testes que passam. O trabalho do desenvolvedor não desapareceu — ele migrou. Deixou de ser, principalmente, o ato de digitar e passou a ser, principalmente, o ato de auditar.

Aqui mora a armadilha. Código que compila, passa no teste e roda em produção não é sinônimo de código bom. A IA otimiza para "funciona" — quase nunca para "sustentável". Ela entrega a primeira solução que satisfaz o exemplo, e essa solução costuma vir com condicionais aninhadas, trechos duplicados, dependências circulares, consultas inseguras montadas por concatenação de strings e algoritmos quadráticos onde caberia um Set. Tudo isso passa no teste. Nada disso aparece quando você só pergunta "rodou?".

A tese desta aula

Esta aula ensina a montar um painel de qualidade local: um conjunto de bibliotecas npm que você roda no próprio terminal, sem servidor e sem serviço externo, para medir os requisitos não-funcionais do código gerado — cobertura de testes, complexidade, manutenibilidade, duplicação, arquitetura, segurança e corretude de tipos — e então refatorar guiado por esses números. É o raio-x do código: enquanto os testes dizem que o programa faz a coisa certa, as métricas dizem o quanto disso está protegido e se o código é bom.

Testes são a rede; métricas são o raio-x

Nas aulas anteriores você aprendeu a escrever testes: a pirâmide, o AAA, os test doubles, o Jest com TypeScript. Os testes garantem que o código faz a coisa certa — são a rede de proteção contra regressão. Mas eles não respondem a duas perguntas igualmente importantes: o quanto do código está realmente protegido por essa rede (é a cobertura), e se o código que passou no teste é de fato bom de manter (são as métricas de qualidade). Esta aula fecha esse buraco. Os testes continuam sendo a rede; o painel e a cobertura são o raio-x que mostra onde a rede tem furos e onde o esqueleto do código está torto.

O novo perfil de erro

Antes, o erro típico do iniciante era escrever código que não funcionava. Agora, o erro típico é aceitar código que funciona mas é ruim: ilegível, frágil, inseguro, lento. A IA é uma estagiária brilhante e apressada — produz muito, mas não responde pela manutenção. Quem responde é você. Auditar virou a competência central.

2. Requisitos não-funcionais e a ISO/IEC 25010

Todo software tem dois tipos de requisito. Os funcionais dizem o que o sistema faz: "cadastrar usuário", "calcular frete", "emitir nota". Os não-funcionais (RNF) dizem quão bem ele faz: com que rapidez, quão seguro, quão fácil de manter, quão confiável. Os testes que você já escreveu atacam sobretudo os requisitos funcionais. O painel de qualidade ataca os RNF.

A norma internacional ISO/IEC 25010 organiza a qualidade de produto de software em oito características. Conhecê-las dá nome aos números que vamos medir:

CaracterísticaO que avaliaMede no painel?
Adequação FuncionalFaz o que deveria, completo e corretocobertura de testes
Eficiência de DesempenhoTempo e recursos consumidosBig-O dos algoritmos
CompatibilidadeCoexiste e interopera com outros sistemas—
UsabilidadeFacilidade de uso pela pessoa—
ConfiabilidadeFunciona sem falhar ao longo do tempocobertura, dependências
SegurançaProtege dados e resiste a ataquesnpm audit, eslint-security
ManutenibilidadeFacilidade de modificar sem quebrarcomplexidade, MI, duplicação
PortabilidadeMigra entre ambientes—

Nesta aula o foco recai sobre quatro delas, justamente as que mais sofrem com código gerado às pressas: Manutenibilidade (o código será lido e alterado dezenas de vezes), Segurança (a IA adora concatenar SQL e usar eval), Confiabilidade (cobertura e ausência de ciclos) e Eficiência (o algoritmo quadrático escondido). Cada ferramenta do painel coloca um número onde antes só havia opinião.

Por que normatizar a qualidade

Sem uma norma, "código bom" vira discussão de gosto. A ISO/IEC 25010 transforma qualidade em vocabulário compartilhado: quando você diz "a manutenibilidade caiu", todo o time entende a mesma coisa. O painel é a régua que dá valor numérico a esse vocabulário — e número você consegue comparar antes e depois de uma refatoração.

3. Cobertura de testes — npm test -- --coverage

Esta é a peça central do entregável. Você já escreveu testes nas aulas anteriores; agora precisa gerar e anexar o relatório de cobertura ao seu Merge Request. Cobertura é a métrica que responde "do código que existe, quanto está protegido por pelo menos um teste?". Sem ela, você sabe que tem testes, mas não sabe o que não tem teste — e o que não tem teste é exatamente onde a IA esconde os bugs.

O comando

Com Jest + ts-jest, basta passar a flag --coverage. Ela imprime uma tabela no terminal e gera uma pasta coverage/ com um relatório HTML navegável.

Bash — gerar o relatório
# roda os testes E coleta cobertura
npm test -- --coverage

# o relatório navegável fica em:
# coverage/lcov-report/index.html  (abra no navegador)

Configurar no jest.config.ts

Para o relatório sair completo e consistente, configure a coleta no jest.config.ts. O collectCoverageFrom diz quais arquivos contam (e quais ignorar, como o server.ts que só sobe o servidor); o coverageReporters diz em que formatos gerar.

jest.config.ts — bloco de cobertura
import type { Config } from 'jest';

const config: Config = {
  preset: 'ts-jest',
  testEnvironment: 'node',
  collectCoverage: true,
  collectCoverageFrom: [
    'src/**/*.ts',
    '!src/**/*.d.ts',
    '!src/server.ts',
  ],
  coverageDirectory: 'coverage',
  coverageReporters: ['text', 'html', 'lcov'],
  // opcional: barra a suíte se cair abaixo do mínimo
  coverageThreshold: {
    global: { statements: 80, branches: 70, functions: 80, lines: 80 },
  },
};

export default config;

As quatro dimensões

O relatório mede cobertura em quatro recortes diferentes do mesmo código. Entender a diferença entre eles é o que separa quem lê o relatório de quem só olha o número grande.

DimensãoO que contaPor que importa
% StatementsComandos executados ao menos uma vezVisão geral; fácil de inflar
% BranchesGalhos de decisão (if/else, ?:, &&)a mais reveladora
% FunctionsFunções chamadas ao menos uma vezAponta funções nunca exercitadas
% LinesLinhas físicas executadasParecido com statements

Branches é a dimensão mais reveladora. É comum ter 90% de statements mas só 60% de branches: significa que você testou o caminho feliz de cada função, mas deixou o else, o tratamento de erro e o caso de borda sem exercício. E é justamente no galho não testado que o bug se esconde. Sempre olhe branches antes de comemorar.

Lendo a saída do terminal

Ao rodar o comando, o reporter text imprime uma tabela como esta:

Terminal — saída de npm test -- --coverage
----------------|---------|----------|---------|---------|-------------------
File            | % Stmts | % Branch | % Funcs | % Lines | Uncovered Line #s
----------------|---------|----------|---------|---------|-------------------
All files       |   87.4  |   79.1   |   91.2  |   87.0  |
 helpers/       |   100   |   100    |   100   |   100   |
 services/      |   84.0  |   71.5   |   88.0  |   83.7  | 42-55, 71
 controllers/   |   78.2  |   63.0   |   85.0  |   78.2  | 23, 30-38
----------------|---------|----------|---------|---------|-------------------

Leia da direita para a esquerda. A coluna Uncovered Line #s aponta exatamente quais linhas nenhum teste tocou. Aqui, controllers/ tem só 63% de branches e linhas 23, 30-38 sem cobertura — provavelmente os ramos de erro (status 400/409). Esse é o alvo: escreva testes para esses galhos e o número sobe de forma honesta.

O relatório HTML, linha a linha

Abra coverage/lcov-report/index.html no navegador. Você clica em um arquivo e vê o código-fonte colorido: verde = coberto, vermelho = nunca executado, e um marcador I/E nos galhos (if/else) que só rodaram por um lado. É a ferramenta mais didática do painel: você vê o buraco. É esse HTML (ou um print dele) que você anexa ao MR.

Cobertura é bússola, não meta cega

Cobertura serve para achar buracos, não para perseguir 100%. Os últimos pontos percentuais quase sempre custam testes artificiais que afirmam pouco. A meta saudável: ≥ 80% statements e ≥ 70% branches, com foco em cobrir a lógica de negócio e os ramos de erro. Mas atenção: mesmo sendo bússola, o relatório precisa ser gerado e anexado — é o coração do entregável.

4. Complexidade ciclomática

A complexidade ciclomática, criada por Thomas McCabe em 1976, conta o número de caminhos independentes que atravessam uma função. A receita é simples: comece em 1 e some 1 para cada ponto de decisão — cada if, else if, case, &&, ||, operador ternário e laço. Quanto maior o número, mais caminhos a função tem e, portanto, mais testes são necessários para exercitá-la inteira.

Essa métrica liga-se diretamente à testabilidade: uma função com complexidade 14 precisa, no limite, de 14 testes para cobrir todos os galhos. Por isso ela conversa com a cobertura da seção anterior — funções complexas são as que mais derrubam o % de branches.

O limite e como vigiá-lo com ESLint

A convenção da indústria é manter funções com complexidade ≤ 10. Acima disso, a função vira difícil de ler, testar e alterar com segurança. O ESLint já traz uma regra complexity nativa — sem plugin nenhum — que falha o lint quando alguma função estoura o teto. Junte algumas regras irmãs e você tem um guardião de complexidade no terminal:

.eslintrc — regras de complexidade
{
  "rules": {
    "complexity":               ["error", 10],
    "max-depth":                ["error", 4],
    "max-params":               ["error", 4],
    "max-lines-per-function":   ["error", 50],
    "max-nested-callbacks":     ["error", 3]
  }
}

Quando alguma função passa do limite, o terminal aponta o dedo com precisão de linha e coluna:

Terminal — npx eslint src
freight.ts
  3:1  error  Function 'calcFrete' has a complexity of 14. Maximum allowed is 10  complexity

Repare nas regras companheiras: max-depth (aninhamento de blocos), max-params (lista longa de parâmetros é um smell) e max-lines-per-function (método longo). Juntas, formam um cerco contra as funções inchadas que a IA tende a gerar.

5. Complexidade cognitiva

A ciclomática conta caminhos, mas trata todos os galhos como iguais. Na prática, um if no topo da função é fácil de ler; o mesmo if aninhado a quatro níveis de profundidade é um pesadelo. A complexidade cognitiva corrige isso: mede o esforço humano de leitura, penalizando especialmente o aninhamento. Dois trechos podem ter a mesma ciclomática e cognitivas muito diferentes.

Complexidade CiclomáticaComplexidade Cognitiva
MedeCaminhos independentesEsforço de leitura humana
AninhamentoNão penaliza profundidadePenaliza fortemente
Liga-se aTestabilidade (nº de testes)Legibilidade / manutenção
Limite usual≤ 10≤ 15

Medindo com ts-complex

A ferramenta ts-complex calcula as duas métricas, por função, direto em TypeScript:

Terminal — npx ts-complex src/freight.ts
npx ts-complex src/freight.ts

  Function       Cyclomatic   Cognitive
  calcFrete          14           22

A calcFrete gerada pela IA marca ciclomática 14 e cognitiva 22 — ambas acima do limite, e a cognitiva alta denuncia o aninhamento de condicionais. Depois de refatorada (seção 13), a mesma função cai para ciclomática 3 e cognitiva 4, com as mesmas saídas. É a prova viva de que "funciona" e "é bom" são coisas diferentes: as duas versões passam nos mesmos testes, mas só uma é agradável de manter.

Por que a cognitiva é tão útil na auditoria de IA

A IA frequentemente gera código que funciona à custa de muitos if aninhados — porque foi treinada para satisfazer o exemplo, não para minimizar carga cognitiva. A complexidade cognitiva é o sensor mais direto desse problema: número alto significa "vai doer ler isto daqui a três meses".

6. Índice de Manutenibilidade e o dashboard plato

O Índice de Manutenibilidade (MI, Maintainability Index) condensa várias métricas em um único número de 0 a 100 — quanto maior, mais fácil de manter. Ele combina três ingredientes: o volume de Halstead (tamanho e densidade de operadores/operandos), a complexidade ciclomática e as linhas de código (SLOC). Funções grandes, densas e cheias de decisões derrubam o índice.

Faixa de MIInterpretação
< 65Ruim — manutenção difícil, candidato a refatorar
65 – 85Moderado — aceitável, vigiar
> 85Bom — fácil de manter

plato: o dashboard visual

O plato varre o projeto e gera um relatório HTML navegável — o dashboard do painel. Você vê gráficos de MI ao longo do tempo, ciclomática média, métricas de Halstead (volume, dificuldade, esforço), SLOC e erros de lint, arquivo por arquivo.

Terminal — gerar o dashboard
npx plato -r -d report src

# abre o painel navegável em:
# report/index.html

A flag -r é recursiva (entra nas subpastas) e -d report define a pasta de saída. Abra report/index.html e você terá uma visão de pássaro da saúde do código: os arquivos com MI baixo aparecem em destaque, prontos para serem o alvo da próxima refatoração.

As métricas de Halstead

Maurice Halstead propôs medir software como se fosse um texto: contando operadores distintos, operandos distintos e suas ocorrências, derivam-se volume (tamanho conceitual), dificuldade (propensão a erro) e esforço (trabalho mental para entender). Não decore as fórmulas; entenda a ideia — código denso e cheio de símbolos é mentalmente caro, e o plato põe esse custo num gráfico.

7. Duplicação de código

O princípio DRY (Don't Repeat Yourself) diz que cada pedaço de conhecimento deve ter uma única representação no sistema. Código duplicado viola isso: quando a regra muda, você precisa lembrar de alterar todas as cópias — e sempre esquece uma. A IA é uma máquina de duplicar: ela gera funções parecidas em arquivos diferentes porque cada prompt é respondido de forma isolada, sem visão do todo.

Medindo com jscpd

O jscpd (copy/paste detector) varre o código procurando blocos repetidos e reporta a porcentagem de duplicação e o número de clones:

Terminal — npx jscpd src
npx jscpd src

  Found 12 clones.
  Duplicated lines: 4.7% (143 of 3024 lines)

O limiar saudável é menos de 5% de duplicação. Acima disso, é hora de extrair o trecho repetido para uma função ou módulo comum. Você fixa o limite num arquivo .jscpd.json para que o comando falhe quando a duplicação ultrapassar a régua — bom para deixar o número visível e sob controle.

Duplicação não é só copiar e colar literal

O jscpd também pega duplicação estrutural: dois blocos que fazem a mesma coisa com nomes de variáveis diferentes. Esse é o tipo que a IA mais produz — e o mais perigoso, porque não salta aos olhos numa leitura rápida.

8. Dependências e arquitetura

Um código bem arquitetado tem dependências que fluem em uma direção: controllers dependem de services, services de repositories, e ninguém volta. Quando esse fluxo se torna um ciclo — A importa B que importa A —, você tem uma dependência circular: um sintoma de acoplamento alto que dificulta testar, refatorar e até carregar os módulos na ordem certa. A IA cria ciclos com facilidade porque não enxerga o grafo completo de imports.

Detectando com madge

O madge analisa o grafo de imports do projeto e denuncia ciclos:

Terminal — npx madge --circular src
npx madge --circular src

  ✖ Found 1 circular dependency:

  1) order.service.ts > payment.service.ts > order.service.ts

A meta é simples e inegociável: 0 dependências circulares. Quando o madge aponta um ciclo, quebre-o — geralmente extraindo a parte compartilhada para um terceiro módulo do qual ambos dependem, ou invertendo a dependência com uma interface.

O madge também desenha o grafo inteiro como imagem, ótimo para enxergar o acoplamento de relance:

Terminal — gerar o grafo visual
npx madge --image graph.svg src

# abre graph.svg: um nó por módulo, setas = imports

Um grafo com poucos nós muito conectados (uma "teia") indica acoplamento alto — outro RNF de manutenibilidade que vale auditar.

9. Segurança das bibliotecas — npm audit

O código que você escreve é só a ponta do iceberg: por baixo, dezenas (às vezes centenas) de pacotes de terceiros rodam dentro da sua aplicação. Cada um deles pode ter uma vulnerabilidade conhecida — uma CVE (Common Vulnerabilities and Exposures). Quando a IA sugere "instale o pacote X", ela raramente verifica se aquela versão tem buracos de segurança. Quem verifica é você, com o npm audit.

Terminal — auditar dependências
npm audit

  found 3 vulnerabilities (1 critical, 2 high)

# falha só com severidade alta ou crítica (bom para o painel)
npm audit --audit-level=high

A flag --audit-level=high faz o comando falhar apenas quando há vulnerabilidades high ou critical — útil para o painel não travar por avisos menores. A meta: 0 vulnerabilidades high/critical.

Corrigindo

Boa parte das vulnerabilidades se resolve com um único comando, que atualiza os pacotes para versões corrigidas dentro do que o package.json permite:

Terminal — corrigir automaticamente
npm audit fix

# para correções que exigem major version (revise antes):
npm audit fix --force
Cuidado com o --force

O npm audit fix --force pode subir versões major de pacotes, o que muda a API e pode quebrar seu código. Use-o com a suíte de testes verde por perto: rode, depois rode npm test. Se quebrou, você descobre na hora — mais uma razão para a rede de testes existir.

10. Segurança no código — eslint-plugin-security

O npm audit cuida das bibliotecas; mas a vulnerabilidade pode estar no seu código — ou melhor, no código que a IA escreveu para você. O eslint-plugin-security detecta padrões inseguros direto nos arquivos-fonte, sinalizando construções clássicas que abrem brechas.

RegraO que detectaRisco que a IA introduz
detect-non-literal-regexpRegExp montado a partir de inputReDoS (negação de serviço por regex)
detect-child-processExecução de processos do sistemaInjeção de comando
detect-non-literal-fs-filenamefs com caminho dinâmicoPath traversal (ler arquivo proibido)
detect-eval-with-expressioneval com expressãoExecução de código arbitrário
detect-object-injectionAcesso a objeto por chave dinâmicaObject injection / prototype pollution

Ative o plugin no .eslintrc com extends: ["plugin:security/recommended"] e ele passa a apontar os problemas junto com o lint normal:

Terminal — eslint-plugin-security em ação
user.controller.ts
  23:14  warning  Found non-literal argument to RegExp Constructor  security/detect-non-literal-regexp

O que a IA costuma errar

Quatro padrões aparecem com frequência no código gerado, e todos são pegos por esse plugin (ou por uma leitura atenta):

  • SQL por concatenação — montar a query com 'SELECT ... WHERE id = ' + req.params.id, abrindo SQL injection. O certo é usar queries parametrizadas ($1, $2 no pg).
  • eval — usar eval para "interpretar" algo do usuário. Quase nunca é necessário e é perigosíssimo.
  • RegExp de input — new RegExp(req.query.busca) permite que o usuário derrube o servidor com uma regex catastrófica.
  • fs com path do usuário — fs.readFile(req.query.arquivo) deixa o usuário pedir ../../etc/passwd.
A IA não tem instinto de segurança

A IA otimiza para produzir um resultado que "funciona com o exemplo dado". Um exemplo benigno (id = 5) nunca dispara a brecha — ela só aparece quando um usuário malicioso manda id = 5 OR 1=1. Por isso a auditoria de segurança é sua responsabilidade: a IA não pensa em adversários, você precisa pensar.

11. Corretude das saídas e cobertura de tipos

Auditar a saída de uma função é verificar o seu contrato: qual o tipo de retorno, e o que acontece nos casos de borda — null, undefined, array vazio, string vazia. A IA tende a tratar o caminho feliz e esquecer as bordas; o retorno "funciona" para o exemplo e quebra quando alguém passa [].

Em TypeScript, o melhor guardião do contrato de saída é o sistema de tipos. Mas há um vilão: o any. Cada any é um buraco no contrato — o compilador desliga as verificações e o bug volta a ser possível. A IA usa any demais, porque é o caminho de menor resistência quando ela não sabe o tipo exato.

Medindo com type-coverage

O type-coverage calcula a porcentagem do código que está coberta por tipos reais (não-any):

Terminal — npx type-coverage
npx type-coverage --detail --at-least 95

  9821 / 10000  98.21%

# --detail lista CADA ocorrência de any (arquivo:linha:coluna)
# --at-least 95 faz o comando falhar abaixo de 95%

A meta é acima de 95%. A flag --detail lista cada any com arquivo, linha e coluna — é a lista de tarefas para tipar o que a IA deixou solto. Tipos bem definidos garantem o shape do retorno: se a função promete devolver User, o compilador impede que ela devolva undefined por engano.

Tipos como teste estático

Um tipo é um teste que roda em todo build, sem você escrever um expect. Quando você troca any por um tipo preciso, está adicionando dezenas de verificações gratuitas sobre as saídas — e o type-coverage mede o quanto dessa proteção você realmente tem.

12. Complexidade algorítmica (Big-O)

Última peça da auditoria: a eficiência, RNF de Desempenho. Código pode funcionar e passar nos testes com dados pequenos, mas ficar lento de forma catastrófica conforme a entrada cresce. A IA, focada em "produzir algo que roda", frequentemente gera soluções O(n²) — um laço dentro de outro, ou um .find()/.includes() chamado dentro de um .map(), o que esconde um laço aninhado.

O caso clássico: deduplicar

Veja como a IA costuma deduplicar uma lista — e como auditar transforma O(n²) em O(n):

TypeScript — ANTES: O(n²) gerado pela IA
function dedup(itens: number[]): number[] {
  const saida: number[] = [];
  for (const x of itens) {
    // includes percorre 'saida' inteira: laço dentro de laço = O(n²)
    if (!saida.includes(x)) saida.push(x);
  }
  return saida;
}
TypeScript — DEPOIS: O(n) com Set
function dedup(itens: number[]): number[] {
  // Set verifica pertencimento em O(1); total O(n)
  return [...new Set(itens)];
}

As duas versões passam exatamente nos mesmos testes. A diferença só aparece com 100 mil itens: a primeira faz bilhões de comparações; a segunda, cem mil. Ler o Big-O do código gerado e questionar "isso escala?" é parte da auditoria de eficiência. Não há ferramenta que meça Big-O automaticamente — aqui o sensor é o seu olho treinado: laço dentro de laço, ou método de busca dentro de laço, acende a luz amarela.

Estruturas de dados são a cura

A maioria dos O(n²) da IA vira O(n) trocando a estrutura de dados: Set para pertencimento, Map para busca por chave, ordenar uma vez em vez de buscar repetidamente. Quando ver um .find() dentro de um laço, pergunte: "dá para indexar isso num Map antes?".

13. Refatoração guiada por métricas

Medir só faz sentido se levar à ação. Refatorar é melhorar a estrutura interna do código sem mudar seu comportamento externo — as saídas continuam idênticas, os testes continuam verdes, mas os números do painel melhoram. Martin Fowler catalogou os code smells (sinais de problema) e os refactorings (movimentos para resolvê-los).

Code smell (Fowler)Refactoring que resolve
Método LongoExtrair Função

Código DuplicadoExtrair Função

O exemplo canônico: calcFrete

Lembre dos números das seções anteriores: a calcFrete gerada pela IA marcava ciclomática 14 / cognitiva 22. Ela tem todos os smells: método longo, condicional aninhada, código duplicado (as três faixas de peso repetidas) e números mágicos. Eis o estado original:

freight.ts — ANTES (ciclomática 14 / cognitiva 22)
export function calcFrete(uf: string, peso: number, expresso: boolean): number {
  let valor = 0;
  if (uf === 'SP' || uf === 'RJ' || uf === 'MG' || uf === 'ES') {
    if (peso <= 1) valor = 10;
    else if (peso <= 5) valor = 18;
    else if (peso <= 10) valor = 28;
    else valor = 40;
  } else if (uf === 'PR' || uf === 'SC' || uf === 'RS') {
    if (peso <= 1) valor = 14;
    else if (peso <= 5) valor = 22;
    else if (peso <= 10) valor = 34;
    else valor = 50;
  } else {
    if (peso <= 1) valor = 20;
    else if (peso <= 5) valor = 32;
    else if (peso <= 10) valor = 48;
    else valor = 70;
  }
  if (expresso) valor = valor * 1.6;
  return valor;
}

Aplicando três refactorings — Substituir Condicional por Tabela, Extrair Constante e Cláusulas de Guarda —, a função encolhe para ciclomática 3 / cognitiva 4, com saídas idênticas:

freight.ts — DEPOIS (ciclomática 3 / cognitiva 4)
type Regiao = 'sudeste' | 'sul' | 'outras';

const REGIAO: Record<string, Regiao> = {
  SP: 'sudeste', RJ: 'sudeste', MG: 'sudeste', ES: 'sudeste',
  PR: 'sul', SC: 'sul', RS: 'sul',
};

const TABELA: Record<Regiao, number[]> = {
  sudeste: [10, 18, 28, 40],
  sul:     [14, 22, 34, 50],
  outras:  [20, 32, 48, 70],
};

const FAIXAS = [1, 5, 10, Infinity];

export function calcFrete(uf: string, peso: number, expresso: boolean): number {
  const regiao = REGIAO[uf] ?? 'outras';
  const faixa = FAIXAS.findIndex((limite) => peso <= limite);
  const base = TABELA[regiao][faixa];
  return expresso ? base * 1.6 : base;
}

De 24 linhas cheias de galhos para 5 linhas declarativas. A regra de negócio agora mora em dados (as tabelas), não em controle de fluxo — adicionar uma região nova vira uma linha na tabela, sem tocar na lógica. É a refatoração guiada por métrica: o número apontou o problema, o catálogo de Fowler apontou o movimento, e os testes garantiram que o comportamento não mudou.

Refatore sempre com testes verdes

A condição inegociável para refatorar é ter testes cobrindo o comportamento antes de mexer. Sem rede, "refatorar" é só "reescrever e torcer". Rode os testes, veja o verde, refatore um passo pequeno, rode de novo. Se ficou vermelho, você mudou comportamento — desfaça.

14. Passo a passo: configurar e gerar os relatórios

Esta seção é o roteiro reproduzível. Siga na ordem e ao final você terá todos os relatórios do painel prontos para anexar ao MR.

Passo 1 — Instalar as devDependencies

Bash — instalação

         type-coverage eslint-plugin-security plato

Passo 2 — Configurar a cobertura

No jest.config.ts, ative a coleta (mesmo bloco da seção 3): collectCoverage: true, collectCoverageFrom, coverageDirectory: 'coverage' e coverageReporters: ['text','html','lcov']. Depois rode e abra o HTML:

}
Bash — rodar lint e complexidade
npx eslint src
npx ts-complex src

Passo 4 — Fixar os limiares de duplicação e tipos

npx type-coverage --at-least 95     # cobertura de tipos
Dica: rode tudo de uma vez

Em vez de digitar seis comandos, deixe-os encadeados num único script quality no package.json (próxima seção). Um npm run quality roda o painel inteiro; npm test -- --coverage gera a cobertura. Dois comandos, todo o raio-x.

Passo 6 — Preencher e anexar

Com os relatórios em mãos (terminal + os HTMLs em coverage/ e report/), preencha o relatório-modelo (seção 15) e anexe os prints ao Merge Request. Esse é o entregável.

O que cada seção significa

  • 1. Cobertura de testes — as quatro dimensões com a meta de cada uma; aponte o pior arquivo e referencie o HTML. É o coração do entregável.
  • 2. Complexidade — ciclomática e cognitiva das funções que você refatorou, mostrando o antes/depois. O par calcFrete 14/22 → 3/4 é o exemplo a seguir.
  • 3. Manutenibilidade — o MI médio do projeto (plato) e o pior arquivo, com print do dashboard.
  • 4. Duplicação — a porcentagem do jscpd contra a meta de 5%.
  • 5. Dependências — quantos ciclos o madge achou (a meta é zero).
  • 6. Segurança — o resultado do npm audit (bibliotecas) e do eslint-plugin-security (código).
  • 7. Cobertura de tipos — a porcentagem do type-coverage contra a meta de 95%.

O entregável em uma frase

Anexe ao MR: cobertura + complexidade ciclomática + manutenibilidade + duplicação + dependências + segurança de bibliotecas + cobertura de tipos, com o relatório-modelo preenchido e os prints dos HTMLs. Principalmente a cobertura — é o que o professor pede primeiro.

16. O painel de qualidade local consolidado

Em vez de memorizar cada comando, registre o painel inteiro como scripts no package.json. Assim npm run quality executa todas as verificações em sequência e para na primeira que falhar — um único comando para o raio-x completo.

package.json — o painel como scripts
"scripts": {
  "test":       "jest",
  "test:cov":   "jest --coverage",
  "lint":       "eslint src --max-warnings 0",
  "complexity": "ts-complex src --threshold 10",
  "dup":        "jscpd src --threshold 5",
  "deps":       "madge --circular src",
  "types":      "type-coverage --at-least 95",
  "audit":      "npm audit --audit-level=high",
  "report":     "plato -r -d report src",
  "quality":    "npm run lint && npm run complexity && npm run dup && npm run deps && npm run types && npm run audit && npm run test:cov"
}

Cada ferramenta tem um número-alvo. Esta é a régua que define "código auditado e aprovado":

Todo o painel roda no seu terminal — nada de serviço externo, nada na nuvem. Você instala as bibliotecas, roda npm run quality e lê os números ali mesmo. É o raio-x na sua máquina, antes mesmo de abrir o MR.

17. O fluxo IA → auditoria

Junte tudo num ciclo repetível. Toda vez que a IA gera código, ele passa por estas seis etapas antes de virar commit:

  1. A IA gera — você descreve a intenção e recebe o código.
  2. Meça — rode npm run quality e npm test -- --coverage.
  3. Leia os números — compare cada métrica com a régua de limiares.
  4. Refatore os pontos vermelhos — ataque o que estourou (smell → refactoring de Fowler).
  5. Re-meça — os números caíram? Os testes continuam verdes? (Se não, você mudou comportamento — desfaça.)
  6. Preencha o relatório e comite — com o painel verde, documente no relatório-modelo e suba o MR.

Checklist de auditoria de código de IA

Antes de aceitar qualquer código gerado, passe por estas nove perguntas:

  • ✓
    A cobertura é suficiente? (≥ 80% statements, ≥ 70% branches)
  • ✓
    A complexidade está no limite? (ciclomática ≤ 10, cognitiva ≤ 15)
  • ✓
    Há duplicação abaixo de 5%?
  • ✓
    Existe alguma dependência circular? (alvo: zero)
  • ✓
    Há padrões inseguros no código? (SQL concatenado, eval, RegExp de input, fs com path)
  • ✓
    As dependências estão sem CVE high/critical?
  • ✓
    Os tipos estão cobertos? (sem any escondido, > 95%)
  • ✓
    O algoritmo é eficiente? (sem O(n²) onde cabe O(n))
  • ✓
    Os casos de borda das saídas estão tratados? (null, undefined, array vazio)

18. Boas práticas

Métrica é bússola, não meta

O alvo nunca é o número em si — é o código bom que o número aponta. Perseguir 100% de cobertura ou MI 100 produz testes artificiais e código contorcido para agradar a ferramenta. Use as métricas para encontrar problemas, corrija os problemas reais, e deixe o número subir como consequência.

Refatore em passos pequenos, com testes verdes

Nunca refatore um trecho grande de uma vez. Faça um movimento (extrair uma função, substituir uma condicional), rode os testes, veja o verde, comite. Passos pequenos mantêm a rede de segurança sempre ativa e tornam trivial localizar onde algo quebrou.

Meça antes de mexer

Sempre tire a foto "antes": rode o painel e anote os números antes de refatorar. Sem o antes, você não tem como provar que a refatoração melhorou — e às vezes ela piora sem que você perceba. O par antes/depois (como o calcFrete 14→3) é a prova objetiva de que valeu a pena.

Pequenas auditorias contínuas > uma grande no fim

Rodar o painel a cada feature, sobre um diff pequeno, é barato e mantém o código sempre saudável. Deixar para auditar tudo no fim do projeto transforma a auditoria num mutirão doloroso — e a essa altura os smells já se multiplicaram.

19. Anti-padrões

Perseguir 100% de cobertura

Os últimos pontos percentuais custam testes que afirmam pouco — só executam linhas para colorir o relatório. Cobertura é bússola para achar buracos relevantes, não troféu. Pare quando os caminhos importantes (lógica de negócio, ramos de erro) estiverem cobertos.

Refatorar sem teste

Reorganizar código sem uma rede de testes que prove que o comportamento não mudou é apenas "reescrever e torcer". Se não há teste cobrindo o trecho, primeiro escreva o teste (que documenta o comportamento atual), depois refatore.

Confiar cegamente na IA

Aceitar o código gerado porque "passou no teste" ignora tudo o que esta aula mostra: o teste não vê complexidade, duplicação, ciclos, brechas de segurança nem algoritmos quadráticos. "Funciona" é o piso, não o teto. Audite antes de aceitar.

Gold-plating de métrica

Otimizar um número que já está bom — espremer o MI de 88 para 92, ou a cobertura de 88% para 94% num módulo trivial — gasta tempo sem retorno. É o equivalente a polir o que já brilha enquanto outro arquivo está em 40. Direcione o esforço para onde o número está ruim.

O pior anti-padrão: métrica como teatro

Maquiar números para "passar no painel" — desabilitar regras de lint com comentários, excluir os arquivos ruins do collectCoverageFrom, baixar o threshold até tudo passar. Isso transforma a auditoria em encenação e devolve à IA o controle da qualidade. O painel só vale se você for honesto com ele.

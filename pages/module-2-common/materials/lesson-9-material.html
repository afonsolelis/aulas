Box Model e box-sizing

Todo elemento HTML é uma caixa retangular composta por quatro camadas: content (o conteúdo em si), padding (espaçamento interno), border (borda) e margin (espaçamento externo). O Box Model define como essas camadas interagem para determinar o tamanho final que um elemento ocupa na tela.

O comportamento padrão dos navegadores usa box-sizing: content-box: quando você define width: 300px, está definindo apenas a largura do conteúdo. Padding e border são adicionados por fora, fazendo o elemento ocupar mais do que os 300px declarados. Isso é contra-intuitivo e a fonte de incontáveis bugs de layout nos primórdios do CSS.

A solução modern e universal é aplicar box-sizing: border-box a todos os elementos. Com esse valor, a largura declarada inclui padding e border — o que você escreve é exatamente o espaço que o elemento ocupa. Essa linha se tornou a primeira de todo projeto CSS moderno:

CSS 
/* Reset universal — primeira coisa em qualquer projeto */
*, *::before, *::after {
  box-sizing: border-box;
}

/* Com content-box (padrão): ocupa 300 + 40 + 4 = 344px */
.content-box {
  box-sizing: content-box;
  width: 300px;
  padding: 20px;
  border: 2px solid #ccc;
}

/* Com border-box: ocupa exatamente 300px */
.border-box {
  box-sizing: border-box;
  width: 300px;
  padding: 20px;
  border: 2px solid #ccc;
}
Regra de ouro

Sempre comece com * { box-sizing: border-box }. Isso elimina toda uma categoria de bugs de layout antes mesmo de eles aparecerem.

Flexbox

O Flexbox (Flexible Box Layout) resolve o problema mais comum do CSS: distribuir e alinhar elementos ao longo de um único eixo (horizontal ou vertical). Antes do Flexbox, centralizar um elemento verticalmente era um exercício de criatividade e frustração. Hoje é uma linha de CSS.

O Flexbox opera com dois tipos de elementos: o container (o elemento com display: flex) e os items (filhos diretos do container). As propriedades do container controlam como os items são distribuídos; as propriedades dos items controlam o comportamento individual de cada um.

As propriedades mais importantes do container são: flex-direction (eixo principal — row ou column), justify-content (alinhamento no eixo principal), align-items (alinhamento no eixo cruzado), flex-wrap (permitir quebra de linha) e gap (espaçamento entre items sem margin).

CSS Grid

O CSS Grid é o primeiro sistema de layout bidimensional nativo do CSS. Enquanto o Flexbox controla elementos em uma dimensão (linha OU coluna), o Grid controla simultaneamente linhas E colunas. Isso o torna ideal para layouts de página inteiros, dashboards e grades de cards.

A unidade fr (fraction) é a grande inovação do Grid: representa uma fração do espaço disponível. Combinar fr com repeat() e minmax() produz layouts responsivos sem uma linha de media query:

CSS 
/* 3 colunas iguais */
.grid-3 {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: 24px;
}

/* Responsivo sem media query */
.cards {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(280px, 1fr));
  gap: 20px;
}

/* Layout de página com áreas nomeadas */
.layout {
  display: grid;
  grid-template-areas:
    "header header"
    "sidebar main"
    "footer footer";
  grid-template-columns: 260px 1fr;
  grid-template-rows: auto 1fr auto;
  min-height: 100vh;
}
.header  { grid-area: header; }
.sidebar { grid-area: sidebar; }
.main    { grid-area: main; }
.footer  { grid-area: footer; }

Flexbox vs Grid

Flexbox: use quando os items fluem em uma direção (navbar, botões alinhados, itens de lista). Grid: use quando você tem um layout bidimensional em mente (página completa, grades de cards, dashboards). Na prática, você usará os dois juntos — Grid para o layout macro, Flexbox para componentes internos.

Variáveis CSS e Temas Dinâmicos

As CSS Custom Properties (variáveis CSS) permitem definir valores reutilizáveis que podem ser alterados em tempo de execução via JavaScript ou media queries. Diferente de variáveis de pré-processadores (Sass, Less), as variáveis CSS são vivas no DOM — mudar o valor em :root com JavaScript atualiza instantaneamente todos os elementos que a usam.

A declaração usa a sintaxe --nome: valor e o consumo usa var(--nome). Variáveis CSS suportam fallback: var(--cor-primaria, #1863dc) usa #1863dc se --cor-primaria não estiver definida. Elas respeitam cascata e herança normalmente.

CSS 
/* Definir no :root (escopo global) */
:root {
  --bg: #ffffff;
  --text: #1e293b;
  --primary: #1863dc;
  --radius: 12px;
}

/* Tema escuro automático — 3 linhas */
@media (prefers-color-scheme: dark) {
  :root {
    --bg: #0f172a;
    --text: #f1f5f9;
    --primary: #60a5fa;
  }
}

/* Consumo */
body {
  background: var(--bg);
  color: var(--text);
}
.btn-primary {
  background: var(--primary);
  border-radius: var(--radius);
}

/* Tema alternado via JavaScript */
document.documentElement.style.setProperty('--primary', '#dc2626');
Vantagem sobre Sass/Less

Variáveis de pré-processadores são compiladas em tempo de build — não existem em runtime. Variáveis CSS nativas continuam presentes no navegador, permitindo alteração dinâmica sem recompilar nada.

Metodologia BEM

Conforme os projetos crescem, o CSS sem convenção vira um labirinto. A BEM (Block__Element--Modifier) é uma metodologia de nomenclatura que resolve esse problema tornando os seletores previsíveis e a estrutura explícita no nome da classe.

Um Block é um componente independente e reutilizável (.card, .navbar, .form). Um Element é uma parte do Block, sempre dependente dele (.card__title, .card__body, .form__input). Um Modifier é uma variação de estado ou aparência (.card--featured, .btn--primary, .form__input--error).

CSS — BEM 
/* Block */
.card { background: white; border-radius: 12px; padding: 20px; border: 1px solid #e2e8f0; }

/* Elements — nunca aninhados: .card .title (errado) */
.card__title { font-size: 1.1rem; font-weight: 700; color: #2e2640; }
.card__body  { color: #64748b; line-height: 1.6; margin-top: 8px; }
.card__footer { display: flex; justify-content: flex-end; margin-top: 16px; }

/* Modifiers */
.card--featured { border-color: #1863dc; border-width: 2px; }
.card--disabled { opacity: 0.5; pointer-events: none; }

/* Botões */
.btn { padding: 8px 16px; border-radius: 8px; font-weight: 700; cursor: pointer; }
.btn--primary { background: #1863dc; color: white; }
.btn--outline { background: transparent; border: 2px solid #1863dc; color: #1863dc; }
.btn--sm { font-size: .8rem; padding: 4px 10px; }
Regra de ouro BEM

Nunca aninhe seletores BEM: .card .card__title é errado. Use simplesmente .card__title. Isso mantém especificidade baixa e previsível, evitando guerras de especificidade no CSS.

Pseudo-classes, Pseudo-elementos e Animações CSS

As pseudo-classes representam estados especiais de um elemento: :hover (quando o cursor está sobre), :focus (quando recebe foco de teclado/tab), :active (momento do clique), :nth-child(n) (enésimo filho), :not(.classe) (negação). São fundamentais para feedback visual interativo.

Os pseudo-elementos ::before e ::after criam "elementos fantasmas" que existem no CSS mas não no HTML. Com a propriedade content, permitem adicionar decorações, ícones e badges sem poluir o HTML com elementos puramente visuais.

As transições CSS (transition) animam suavemente mudanças de estado. As animações CSS (@keyframes + animation) criam animações independentes de interação. A regra de performance: sempre anime transform e opacity — elas rodam na thread do compositor sem causar reflow. Evite animar width, height, top, left.

CSS — Transições e Animações 
/* Transição suave em hover */
.btn {
  background: #1863dc;
  transition: transform .15s ease, box-shadow .15s ease, background .2s;
}
.btn:hover {
  transform: translateY(-2px);
  box-shadow: 0 6px 20px rgba(24,99,220,.35);
}
.btn:active { transform: translateY(0); }

/* Pseudo-elemento — badge de notificação */
.notif-btn { position: relative; }
.notif-btn::after {
  content: attr(data-count);
  position: absolute;
  top: -6px; right: -6px;
  background: #dc2626; color: white;
  border-radius: 999px;
  font-size: .65rem; font-weight: 800;
  min-width: 18px; height: 18px;
  display: flex; align-items: center; justify-content: center;
}

/* Animação com @keyframes */
@keyframes fadeInUp {
  from { opacity: 0; transform: translateY(20px); }
  to   { opacity: 1; transform: translateY(0); }
}
.card {
  animation: fadeInUp .4s ease forwards;
}

/* Shimmer para skeleton screen */
@keyframes shimmer {
  from { background-position: -200% 0; }
  to   { background-position:  200% 0; }
}
.skeleton {
  background: linear-gradient(90deg, #f0f0f0 25%, #e0e0e0 50%, #f0f0f0 75%);
  background-size: 200%;
  animation: shimmer 1.5s infinite;
}

Closures e Module Pattern

Uma closure é a capacidade de uma função de "lembrar" o escopo em que foi criada, mesmo após esse escopo ter encerrado sua execução. Toda função em JavaScript é uma closure — ela captura as variáveis do escopo externo por referência, não por valor.

O Module Pattern usa uma IIFE (Immediately Invoked Function Expression) para criar um escopo privado. Variáveis definidas dentro da IIFE não poluem o escopo global, e o objeto retornado expõe apenas a API pública. Esse padrão era essencial antes dos módulos ES6 (import/export) e ainda é útil em scripts simples.

JavaScript 
// Closure simples
function criarContador(inicio = 0) {
  let count = inicio; // variável privada
  return {
    incrementar() { count++; },
    decrementar() { count--; },
    valor()      { return count; },
    reset()      { count = inicio; }
  };
}
const c = criarContador(10);
c.incrementar();
console.log(c.valor()); // 11

// Module Pattern (IIFE) — estado encapsulado
const carrinho = (() => {
  let items = [];

  return {
    adicionar(produto) {
      items.push(produto);
    },
    remover(id) {
      items = items.filter(p => p.id !== id);
    },
    total() {
      return items.reduce((s, p) => s + p.preco, 0);
    },
    listar() {
      return [...items]; // cópia superficial (imutabilidade)
    }
  };
})();

carrinho.adicionar({ id: 1, nome: 'Notebook', preco: 2500 });
console.log(carrinho.total()); // 2500

📖 Aprofundamento: Closures e loops — armadilha clássica

▼

Um erro comum com closures em loops: todos os callbacks capturam a mesma referência à variável do loop, resultando no valor final em todos os casos. A solução com let (escopo de bloco) ou com uma closure explícita:

JavaScript 
// Problema com var
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100); // imprime 3, 3, 3
}

// Solução com let (escopo de bloco)
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100); // imprime 0, 1, 2
}

Debounce e Throttle

Eventos como scroll, resize e input podem disparar centenas de vezes por segundo. Executar uma função pesada (chamada à API, recálculo de layout) em cada disparo degrada seriamente a performance. Debounce e Throttle são padrões para controlar essa frequência.

O Debounce adia a execução até que o usuário pare de disparar o evento por um intervalo mínimo. Exemplo: numa busca, você quer chamar a API apenas quando o usuário parar de digitar por 400ms, não a cada tecla. Implementação: cancela o timer anterior e inicia um novo a cada chamada.

O Throttle garante que a função execute no máximo uma vez por intervalo fixo, independentemente de quantos eventos ocorreram. Exemplo: atualizar a posição de um elemento no scroll — execute no máximo a cada 16ms (~60fps). Debounce espera a pausa; Throttle limita a frequência.

JavaScript 
// Debounce: executa só depois de parar por `delay` ms
function debounce(fn, delay) {
  let timer;
  return function(...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

// Throttle: executa no máximo 1x por `intervalo` ms
function throttle(fn, intervalo) {
  let ultimo = 0;
  return function(...args) {
    const agora = Date.now();
    if (agora - ultimo >= intervalo) {
      ultimo = agora;
      fn.apply(this, args);
    }
  };
}

// Uso: busca com debounce
const buscarProdutos = debounce(async (termo) => {
  const res = await fetch(`/api/produtos?q=${encodeURIComponent(termo)}`);
  const dados = await res.json();
  renderizar(dados);
}, 400);

document.getElementById('busca').addEventListener('input', e => {
  buscarProdutos(e.target.value);
});

// Uso: scroll com throttle
const onScroll = throttle(() => {
  console.log('Posição:', window.scrollY);
}, 16); // ~60fps
window.addEventListener('scroll', onScroll);

Core Web Vitals e Performance

O Google definiu os Core Web Vitals como métricas essenciais de experiência do usuário que impactam diretamente o ranking em buscas. Elas medem três aspectos: velocidade de carregamento, responsividade e estabilidade visual.

O LCP (Largest Contentful Paint) mede o tempo até o maior elemento visível ser renderizado — idealmente abaixo de 2.5 segundos. O INP (Interaction to Next Paint), substituto do FID, mede a latência de todas as interações — abaixo de 200ms é bom. O CLS (Cumulative Layout Shift) mede quanto o layout "pula" enquanto carrega — abaixo de 0.1 é bom. Shifts de layout acontecem quando imagens sem dimensões definidas ou conteúdo de anúncio empurra elementos já exibidos.

CSS + HTML — Prevenindo CLS 
/* Sempre definir aspect-ratio ou dimensões em imagens */
img {
  width: 100%;
  height: auto;
  aspect-ratio: 16/9; /* reserva o espaço antes de carregar */
}

/* Lazy loading nativo */
<img src="foto.jpg" alt="Descrição" loading="lazy" width="800" height="450">

/* Pré-carregar recursos críticos */
<link rel="preload" as="font" href="fonte.woff2" crossorigin>

/* font-display: swap evita texto invisível durante carregamento */
@font-face {
  font-family: 'Manrope';
  font-display: swap;
}
Ferramentas de diagnóstico

Lighthouse (DevTools → Lighthouse): relatório de performance, acessibilidade e SEO. Performance tab (DevTools): flame chart de execução JS. PageSpeed Insights: mede Core Web Vitals com dados reais de campo (CrUX dataset).

RM-ODP — Viewpoints de Arquitetura

O RM-ODP (ISO/IEC 10746) é um framework de 5 viewpoints para descrever sistemas distribuídos de forma completa. Na Aula 9, os viewpoints mais relevantes são Technology e Computational, que descrevem respectivamente as tecnologias concretas usadas e os componentes funcionais que o sistema expõe.

🏢
Enterprise Viewpoint
Propósito: a interface deve entregar valor ao usuário final, atendendo aos objetivos de negócio do sistema. Cada tela implementa um RF.
📋
Information Viewpoint
A interface consome e exibe dados do backend. O estado da UI (loading/success/error) reflete o estado do sistema de informação.
⚙️
Computational Viewpoint
Componentes visuais como interfaces funcionais: cada componente tem estado interno e expõe comportamento via eventos (click, submit).
🔧
Engineering Viewpoint
O browser é o runtime que executa HTML/CSS/JS. Event loop, DOM, e CSSOM são a infraestrutura que suporta a camada de apresentação.
💻
Technology Viewpoint ⭐ foco desta aula
As tecnologias concretas: CSS (Flexbox, Grid, Variáveis, BEM), JavaScript (Closures, Debounce, Event Delegation). A escolha de tecnologias impacta diretamente os RNFs de Usabilidade e Desempenho.

RF, RN e RNF — Camada de Apresentação

Os Requisitos Funcionais (RF) da camada de apresentação descrevem o que a interface deve fazer: exibir lista de produtos, permitir cadastro, mostrar estado de carregamento. Cada funcionalidade de negócio deve ter uma tela ou componente correspondente.

As Regras de Negócio (RN) definem constraints visuais: "o botão de salvar deve estar desabilitado enquanto o formulário tem erros", "o total do carrinho deve atualizar imediatamente ao adicionar um item". Essas regras são implementadas com CSS e JavaScript.

Os Requisitos Não-Funcionais (RNF) da camada de apresentação são principalmente de Usabilidade e Desempenho. Usabilidade: consistência visual (BEM, variáveis CSS), feedback visual (estados, transições), acessibilidade. Desempenho: Debounce em inputs, lazy loading de imagens, evitar reflows, Core Web Vitals.

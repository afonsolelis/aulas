1. Completando o CRUD na API

Na aula anterior construímos as rotas de leitura: GET /produtos para listar todos os recursos e GET /produtos/:id para buscar um único registro pelo identificador. Essas duas rotas são suficientes para consultar dados, mas uma API de produção precisa muito mais: os usuários e sistemas clientes também precisam criar, atualizar e remover registros. É exatamente isso que vamos construir hoje.

O acrônimo CRUD (Create, Read, Update, Delete) define as quatro operações fundamentais sobre qualquer recurso persistido. Em REST, cada operação mapeia diretamente para um método HTTP: POST para criação, GET para leitura, PUT ou PATCH para atualização e DELETE para remoção. Ter um CRUD completo significa que o consumidor da API tem controle total sobre o ciclo de vida de um recurso — sem precisar acessar diretamente o banco de dados ou usar outra ferramenta.

Antes de detalhar cada operação, vale relembrar o ciclo request/response em detalhe. Quando um cliente envia uma requisição HTTP, ela chega ao servidor com quatro elementos essenciais: o método (POST, PUT, DELETE...), a URL (qual recurso está sendo manipulado), os cabeçalhos (metadados como Content-Type: application/json) e o body (corpo com os dados, presente em POST e PUT/PATCH). O servidor processa a requisição e devolve uma resposta com status code, cabeçalhos e, opcionalmente, um body JSON. Compreender esse ciclo é essencial para depurar problemas e construir APIs que se comportam de forma previsível.

C
Create — POST /produtos Cliente envia dados no body. Servidor valida, persiste e responde 201 Created com o recurso criado.
R
Read — GET /produtos e GET /produtos/:id Leitura sem efeito colateral. Responde 200 OK com JSON ou 404 Not Found.
U
Update — PUT /produtos/:id ou PATCH /produtos/:id Substitui (PUT) ou atualiza parcialmente (PATCH). Confirma existência antes; responde 200 OK ou 204 No Content.
D
Delete — DELETE /produtos/:id Remove o recurso após confirmar existência. Responde 204 No Content.

2. POST — Criando Recursos

O método POST é usado para criar um novo recurso no servidor. Ao contrário do GET, o POST não é idempotente: chamar POST /produtos duas vezes com o mesmo body cria dois produtos diferentes. É exatamente por isso que criação usa POST e não PUT — o identificador do recurso ainda não existe no momento da chamada, ele é gerado pelo servidor.

O dado enviado pelo cliente chega no body da requisição. No Express, o body só fica disponível em req.body após configurar o middleware express.json() no arquivo principal da aplicação. Sem esse middleware, req.body é undefined e você vai gastar horas debugando sem encontrar o problema. Sempre verifique se app.use(express.json()) está declarado antes das rotas.

Após receber os dados, o controller deve validar os campos obrigatórios antes de tentar qualquer operação no banco. Se o campo nome é obrigatório e não foi enviado, a resposta correta é 400 Bad Request — não um erro 500 de banco de dados. A validação no controller é mais informativa do que deixar o banco lançar um erro de constraint. Por fim, quando o recurso é criado com sucesso, a resposta deve ter status 201 Created (não 200) e deve retornar o recurso recém-criado no body, para que o cliente tenha acesso ao id gerado.

POST
Cria um novo recurso
POST /produtos
Body obrigatório com dados do recurso. Retorna 201 Created e o objeto criado com o id gerado pelo banco.

const express = require('express');
const router = express.Router();
const ctrl = require('../controllers/produtoController');

router.get('/',        ctrl.listar);
router.get('/:id',     ctrl.buscarPorId);

router.put('/:id',     ctrl.atualizar);
router.delete('/:id',  ctrl.remover);

module.exports = router;
controllers/produtoController.js — função criar
const model = require('../models/produtoModel');

async function criar(req, res, next) {
  try {
    const { nome, preco, estoque } = req.body;

    // Validação mínima: campos obrigatórios
    if (!nome || preco == null) {
      return res.status(400).json({ erro: 'nome e preco são obrigatórios' });
    }

    const produto = await model.create({ nome, preco, estoque });

    // 201 Created — não 200 OK
    res.status(201).json(produto);
  } catch (err) { next(err); }
}
Por que 201 e não 200?

O HTTP define 201 Created especificamente para indicar que a requisição resultou na criação de um novo recurso. Usar 200 OK para criação é semanticamente impreciso e dificulta clientes que tomam decisões baseadas no status code. APIs bem definidas usam o status code correto para cada operação.

O que retornar no body da criação?

A convenção mais adotada é retornar o objeto completo criado, incluindo o id gerado pelo banco e campos de auditoria como criado_em. Isso evita que o cliente precise fazer uma segunda requisição GET para obter o recurso recém-criado. Alternativamente, pode-se retornar apenas { id: 42 }, mas isso obriga um round-trip a mais. Para recursos pequenos, retornar o objeto completo é quase sempre a escolha certa.

3. PUT e PATCH — Atualizando Recursos

A distinção entre PUT e PATCH é frequentemente mal compreendida, mas é fundamental para construir APIs com semântica correta. PUT substitui o recurso inteiro: se você enviar apenas { preco: 29.90 } via PUT, todos os outros campos do produto — nome, estoque, descrição — serão apagados ou definidos como nulos, porque o body enviado é o novo estado completo do recurso. PATCH atualiza parcialmente: apenas os campos presentes no body são modificados; os demais permanecem intactos.

Na prática, a maioria das APIs REST de uso geral adota PATCH para atualizações porque é mais flexível e menos propenso a perda acidental de dados. O PUT faz mais sentido em contextos onde o cliente sempre tem e envia o estado completo do recurso, como em uploads de arquivos ou substituição de documentos inteiros. Para APIs de gerenciamento de entidades com muitos campos, PATCH é quase sempre a escolha superior — o cliente envia apenas os campos que mudaram.

Independente de usar PUT ou PATCH, o controller deve verificar a existência do recurso antes de tentar atualizar. Se o id informado não existe no banco, a resposta correta é 404 Not Found — não um erro 500. Após a atualização bem-sucedida, você pode responder com 200 OK e o recurso atualizado no body, ou com 204 No Content sem body, se o cliente já conhece o estado final. A maioria das APIs modernas prefere 200 com o objeto atualizado para facilitar o consumo.

PUT
Substitui o recurso inteiro
PUT /produtos/:id
Todos os campos devem ser enviados. Campos ausentes são zerados. Idempotente.
PATCH
Atualiza parcialmente
PATCH /produtos/:id
Apenas campos enviados são alterados. Campos ausentes permanecem. Mais comum em APIs modernas.
controllers/produtoController.js — função atualizar
async function atualizar(req, res, next) {
  try {
    // 1. Verificar existência — 404 se não existe
    const produto = await model.findById(req.params.id);
    if (!produto) {
      return res.status(404).json({ erro: 'Produto não encontrado' });
    }

    // 2. Atualizar e retornar o recurso atualizado
    const atualizado = await model.update(req.params.id, req.body);
    res.json(atualizado); // 200 OK implícito
  } catch (err) { next(err); }
}
Armadilha: PUT sem todos os campos

Se o cliente usa PUT mas envia apenas os campos que quer mudar (comportamento de PATCH), e o servidor simplesmente faz UPDATE SET ... WHERE id = ? apenas com os campos recebidos, a API está mentindo sobre a semântica do método. Ou você implementa PUT de verdade (substituição total) ou muda o método para PATCH. Misturar as semânticas confunde consumidores e viola o contrato HTTP.

4. DELETE — Removendo Recursos

A rota de remoção é estruturalmente a mais simples do CRUD, mas esconde algumas decisões importantes. O fluxo básico é: receber o id via parâmetro de rota, verificar se o recurso existe, removê-lo e responder. A verificação de existência antes da remoção é obrigatória — se o id não existir, o servidor deve retornar 404 Not Found. Tentar deletar um registro inexistente e responder com sucesso (204) é enganoso para o cliente.

Quanto ao status de resposta bem-sucedida, a convenção REST é 204 No Content — recurso foi removido e não há mais nada para retornar. Alguns times preferem retornar 200 OK com uma mensagem descritiva como { mensagem: "Produto removido com sucesso" }, o que é igualmente válido mas vai contra a semântica HTTP estrita do 204. O 204 é mais preciso: "processado com sucesso, sem conteúdo de resposta". Para iniciantes, qualquer dos dois funciona; para APIs públicas, prefira 204.

Uma alternativa importante à deleção física é o soft delete: em vez de remover o registro do banco, você define um campo deleted_at com o timestamp da remoção. O registro continua existindo, mas todas as queries de listagem e busca filtram por WHERE deleted_at IS NULL. Soft delete é valioso quando você precisa de auditoria, recuperação de dados, ou quando registros removidos são referenciados por outros registros (por exemplo, produtos em pedidos antigos). O trade-off é maior complexidade nas queries e crescimento do banco ao longo do tempo.

controllers/produtoController.js — função remover
async function remover(req, res, next) {
  try {
    // Confirmar existência antes de deletar
    const produto = await model.findById(req.params.id);
    if (!produto) {
      return res.status(404).json({ erro: 'Produto não encontrado' });
    }

    await model.remove(req.params.id);

    // 204 No Content — sem body
    res.status(204).end();
  } catch (err) { next(err); }
}

Aprofundamento: soft delete com campo deleted_at

▼

Em vez de DELETE FROM produtos WHERE id = ?, o soft delete executa UPDATE produtos SET deleted_at = NOW() WHERE id = ?. Todas as outras queries precisam incluir WHERE deleted_at IS NULL para não retornar registros "deletados".

Vantagens: auditoria completa, recuperação de dados, integridade referencial em registros históricos. Desvantagens: crescimento ilimitado da tabela, complexidade nas queries, necessidade de índice parcial (CREATE INDEX ON produtos (id) WHERE deleted_at IS NULL).

Use soft delete quando: o domínio exige histórico de exclusões, registros removidos podem ser restaurados, ou há relatórios que precisam incluir registros inativos. Use deleção física quando: dados não têm valor histórico, privacidade exige remoção real (LGPD), ou o volume de dados cresce de forma insustentável.

5. Status Codes na Prática

Status codes são o idioma que o servidor usa para comunicar o resultado de uma operação. Um erro comum entre iniciantes é usar 200 OK para tudo — inclusive para erros — e colocar o status real dentro do body JSON ({ success: false, code: 404 }). Isso quebra qualquer cliente HTTP que lê o status code para tomar decisões, e vai contra toda a especificação REST. O status code deve sempre refletir com precisão o que aconteceu.

Para operações CRUD, um subconjunto relativamente pequeno de status codes cobre quase todos os casos. O importante é entender por que cada um existe e quando usá-lo, não apenas memorizar os números. Além disso, erros devem sempre incluir um body JSON com uma mensagem legível por humanos — a diferença entre { erro: "nome é obrigatório" } e um body vazio com status 422 é a diferença entre uma API utilizável e uma API que exige olhar o código-fonte para entender o erro.

Dois status que causam mais confusão são o 422 Unprocessable Entity e o 400 Bad Request. O 400 indica que a requisição está malformada — JSON inválido, campo de tipo errado, header obrigatório ausente. O 422 indica que a requisição está bem formada mas o conteúdo viola regras de negócio ou validação — campo obrigatório ausente, valor fora do intervalo permitido, duplicata. Na prática, muitas APIs usam 400 para os dois casos por simplicidade, o que é aceitável enquanto o body de erro for descritivo.

6. Validação com express-validator

Validação manual com if (!nome || preco == null) funciona para campos simples, mas rapidamente se torna difícil de manter quando há muitos campos com regras complexas: comprimento mínimo/máximo, formatos (email, URL, CPF), valores dentro de um conjunto permitido. A biblioteca express-validator resolve isso declarativamente: você descreve as regras de validação como um array de middlewares e a biblioteca acumula os erros, deixando para o controller apenas verificar se há erros e, em caso positivo, devolvê-los.

As funções principais são body() para validar campos do body, param() para parâmetros de rota e query() para query strings. Cada uma recebe o nome do campo e pode ser encadeada com validadores como .notEmpty(), .isFloat({ min: 0 }), .isEmail(), .isInt(), entre outros. O resultado de todas as validações fica disponível via validationResult(req), que retorna um objeto com método .isEmpty() e .array() para listar os erros.

O padrão recomendado é centralizar a verificação de erros em uma função utilitária que pode ser inserida como middleware entre o array de validações e o controller. Isso evita repetir o bloco if (!result.isEmpty()) return res.status(422).json(...); em cada controller. Ao retornar os erros de validação, inclua sempre o campo (path) e a mensagem para facilitar o trabalho do cliente — e padronize o formato de resposta de erro em toda a API.

validators/produtoValidator.js
const { body, param, validationResult } = require('express-validator');

// Middleware: verifica erros acumulados e aborta se houver
const validar = (req, res, next) => {
  const result = validationResult(req);
  if (!result.isEmpty()) {
    return res.status(422).json({ erros: result.array() });
  }
  next();
};

// Regras de validação para criação de produto
const regiasCriar = [
  body('nome').notEmpty().withMessage('nome é obrigatório')
           .isLength({ max: 100 }).withMessage('nome deve ter até 100 caracteres'),
  body('preco').isFloat({ min: 0 }).withMessage('preco deve ser número positivo'),
  body('estoque').optional().isInt({ min: 0 }).withMessage('estoque deve ser inteiro >= 0'),
  validar
];

// Validação de parâmetro id
const regiasId = [
  param('id').isInt({ min: 1 }).withMessage('id deve ser inteiro positivo'),
  validar
];

module.exports = { regiasCriar, regiasId };
routes/produtos.js — com validadores
const { regiasCriar, regiasId } = require('../validators/produtoValidator');

router.post('/',      ...regiasCriar, ctrl.criar);
router.put('/:id',    ...regiasId,    ctrl.atualizar);
router.delete('/:id', ...regiasId,    ctrl.remover);

7. Middleware de Erro Global

Mesmo com validação cuidadosa, erros inesperados acontecem: o banco pode estar indisponível, uma query pode lançar uma exception, uma dependência pode falhar. Sem um mecanismo centralizado de tratamento de erros, esses casos chegam ao Express como exceções não capturadas e retornam uma resposta padrão confusa ao cliente — frequentemente uma página HTML de erro com stack trace exposto, em vez de um JSON padronizado.

O Express suporta um tipo especial de middleware para tratamento de erros: um middleware com quatro parâmetros — (err, req, res, next). Qualquer middleware ou controller que chame next(err) com um erro vai desviar o fluxo para este handler global. Isso permite que todos os controllers sejam escritos com um simples catch (err) { next(err); } e deixem o tratamento real para um único ponto centralizado, garantindo respostas de erro consistentes em toda a API.

É importante distinguir dois tipos de erros: erros operacionais são condições esperadas que o sistema não controla — banco fora do ar, recurso não encontrado, dado inválido. Erros de programação são bugs no código — acesso a propriedade de undefined, variável não declarada, lógica incorreta. Erros operacionais devem ser tratados e comunicados ao cliente com status code adequado. Erros de programação devem ser logados em detalhe e retornar 500 ao cliente sem expor o stack trace.

app.js — middleware de erro global (4 parâmetros)
// IMPORTANTE: deve ser o ÚLTIMO middleware registrado no app
app.use((err, req, res, next) => {
  console.error(err.stack);

  // Se o erro tem status definido, use-o; caso contrário, 500
  const status = err.status || 500;
  const mensagem = err.isOperational
    ? err.message
    : 'Erro interno do servidor';

  res.status(status).json({
    erro: mensagem,
    // No ambiente local, inclui o stack trace
    ...(process.env.NODE_ENV === 'development' && { stack: err.stack })
  });
});
Nunca exponha stack traces em produção

Stack traces contêm caminhos de arquivo, nomes de módulos, versões de dependências e até fragmentos de código — informações valiosas para um atacante. Em produção, retorne sempre uma mensagem genérica como "Erro interno do servidor" e registre o detalhe nos logs internos, nunca no body da resposta.

Exemplo: lançar erros operacionais com status
// Em qualquer controller ou service:
const produto = await model.findById(id);
if (!produto) {
  const err = new Error('Produto não encontrado');
  err.status = 404;
  err.isOperational = true;
  throw err; // capturado pelo catch e passado ao next(err)
}

8. Organização de Rotas

Quando a API começa a crescer, definir todas as rotas diretamente no app.js torna o arquivo gigante e difícil de manter. O Express oferece express.Router(), que permite criar mini-aplicações com rotas próprias que depois são montadas no app principal. Cada recurso (produtos, usuários, pedidos) fica em seu próprio arquivo de rotas, e o app.js apenas importa e monta cada Router com um prefixo.

O arquivo de rotas deve ser fino: ele apenas declara quais métodos HTTP e paths existem e delega a execução para os controllers. Nenhuma lógica de negócio, nenhuma query de banco, nenhuma validação complexa deve existir dentro do arquivo de rotas. As validações do express-validator são adicionadas como middlewares na rota, mas são definidas em arquivos separados de validação. Dessa forma, o arquivo de rotas lê como uma tabela de conteúdos da API.

A montagem no app.js com app.use('/api/produtos', produtosRouter) significa que todas as rotas definidas em produtosRouter herdam o prefixo /api/produtos. Assim, router.get('/') responde a GET /api/produtos e router.get('/:id') responde a GET /api/produtos/:id. Isso permite renomear o prefixo em um único lugar sem tocar nos arquivos de rota.

app.js — montagem dos Routers
const express = require('express');
const app = express();

// Middlewares globais
app.use(express.json());

// Routers por recurso
app.use('/api/produtos', require('./routes/produtos'));
app.use('/api/usuarios', require('./routes/usuarios'));
app.use('/api/pedidos',  require('./routes/pedidos'));

// Middleware de erro global — SEMPRE por último
app.use(require('./middlewares/errorHandler'));

app.listen(3000, () => console.log('Servidor na porta 3000'));
Estrutura de pastas recomendada

routes/ — um arquivo por recurso, apenas registra rotas e middlewares
controllers/ — lógica de request/response, chama services/models
models/ — acesso ao banco de dados, queries SQL
validators/ — regras express-validator por recurso
middlewares/ — error handler global, autenticação, logging

9. Documentação Implícita — Contratos Claros

Uma API REST bem desenhada é autodocumentada por convenção: apenas olhando as rotas, qualquer desenvolvedor experiente já entende o que cada endpoint faz. Isso acontece quando você segue as convenções REST de forma consistente. A regra principal é: rotas usam substantivos (recursos), não verbos. O verbo HTTP já indica a ação; a URL deve apenas identificar o recurso.

Exemplos de rotas ruins versus boas ilustram bem o princípio. GET /buscarUsuarios, POST /criarProduto e DELETE /removerPedido/5 são verbos duplicados: o método HTTP já comunica a ação. As versões corretas são GET /usuarios, POST /produtos e DELETE /pedidos/5. A estrutura hierárquica de URLs também comunica relacionamentos: GET /usuarios/3/pedidos busca os pedidos do usuário 3, o que é legível e previsível.

Um contrato claro de API inclui também o formato do body esperado e o formato da resposta. Mesmo sem uma ferramenta de documentação como Swagger/OpenAPI, o código deve ser escrito de forma que um colega possa ler o arquivo de rotas e os validators e entender o contrato sem precisar executar o código. Comentários JSDoc ou anotações simples acima de cada rota são investimentos baratos com retorno alto em manutenibilidade e onboarding de novos membros.

10. RM-ODP — Visão Computacional e de Engenharia

O framework RM-ODP (ISO/IEC 10746) organiza a descrição de sistemas distribuídos em cinco visões complementares. Para a Aula 6, as visões mais relevantes são a Computacional e a de Engenharia, pois descrevem exatamente o que estamos construindo: interfaces formais e infraestrutura de comunicação.

A viewpoint Computational descreve o sistema como uma coleção de objetos com interfaces bem definidas. Cada endpoint da API é precisamente isso: uma interface formal que especifica uma operação (criar, atualizar, remover), os parâmetros de entrada (body, params, query), os parâmetros de saída (status code, body de resposta) e as pré e pós-condições (recurso deve existir para atualizar; campos obrigatórios devem estar presentes para criar). O controller é o objeto; o endpoint é a interface. Pensar em endpoints como contratos formais — e não apenas como rotas — muda a forma como você os projeta.

A viewpoint Engineering descreve como esses objetos computacionais são distribuídos e se comunicam: canais, protocolos, binding. No contexto Express, isso se traduz na infraestrutura de roteamento e middleware: o express.Router() é o canal que distribui requisições para os controllers corretos; o express.json() é o protocolo de deserialização do body; o middleware de erro global é o mecanismo de tratamento de falhas de comunicação. O middleware de 4 parâmetros existe exatamente porque, na visão de engenharia, precisamos de um ponto de interceptação de falhas que funcione independentemente do componente que falhou.

11. RF, RN e RNF — Requisitos desta Aula

Os Requisitos Funcionais (RF) descrevem as capacidades que o sistema deve oferecer. Para esta aula, os RFs principais são: o sistema deve disponibilizar endpoints para criação, atualização e remoção de recursos via HTTP; cada endpoint deve retornar o status code correto e o body adequado conforme a operação; operações de criação devem persistir o novo recurso e retornar o id gerado; operações de remoção devem verificar a existência antes de remover.

As Regras de Negócio (RN) definem as restrições e políticas do domínio que os endpoints devem reforçar: campos obrigatórios como nome e preço não podem ser nulos; preço deve ser um número positivo; estoque deve ser um inteiro não-negativo; um produto só pode ser atualizado ou removido se existir. Essas regras devem ser validadas no controller ou validator, antes de qualquer operação no banco de dados.

Os Requisitos Não-Funcionais (RNF) desta aula se concentram em três eixos fundamentais. Segurança: toda entrada de usuário deve ser validada antes de ser processada — nunca confie nos dados do cliente. Confiabilidade: o middleware de erro global garante que nenhum erro não tratado derrubará o servidor ou exponha informações internas; respostas são sempre consistentes mesmo em caso de falha. Suportabilidade: contratos REST claros (substantivos na URL, métodos HTTP como verbos, status codes semânticos) garantem que a API seja compreensível e manutenível por qualquer desenvolvedor que siga as convenções REST.

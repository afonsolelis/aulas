Day 3 — Prompt Refinement & Testing

Make the prompt robust and confirm the assumptions in the field
Slides ← Back

🎯 Goal of the day

In the morning, carry out the technical visit to the partner, observing the real context and confirming or correcting the assumptions recorded in the spec. In the afternoon, make the group's prompt robust: test it beyond the happy path and handle the AI's typical failures.

⚠️ Why test the prompt

The happy path is deceptive: almost every prompt works when the input is perfect. A solution's robustness shows at the extremes — when the input is incomplete, ambiguous, out of scope, or malicious.

Where prompts break

  • Empty or incomplete input: the user does not fill in all the fields.
  • Out-of-scope input: they ask for something that is not the solution's problem.
  • Malicious input: attempts to bypass the constraints (prompt injection).
  • Ambiguous input: several possible interpretations.
  • Volume: text that is too long or too short.

How to handle failures

  • Validate before sending: check required fields in the interface.
  • Instruct the fallback: tell the prompt what to do when data is missing.
  • Reinforce constraints: repeat the limits at the end of the prompt.
  • Clear messages: when the AI cannot respond, explain why.

Principle: a robust prompt fails predictably — never with a wrong output presented as if it were correct.

🧪 The test table

Record each case in a row, run them all, and note the actual result. Adjust the prompt and repeat until every case passes.

Incomplete inputAn important field is missingAsks for the data or uses a fallback

🏭 In the morning — Technical Visit

What to observe on the visit

Bring the group's spec. The visit is the opportunity to confirm or correct the Day 1 assumptions based on the real world.

  • How the problem appears in practice — not on paper.
  • Who the people who would use the solution are and how they speak.
  • Details of tone, vocabulary, and context that change the prompt.
  • What the spec got right and what needs to be revised.

When you return: revise the spec and the test table with what you learned, adjust the prompt if the real context requires it, and arrive calibrated for Thursday's development sprint.

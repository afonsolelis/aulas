Day 2 — Prompts from the Model

Turn the spec into precise instructions for the AI
Slides ← Back

🎯 Goal of the day

Convert the spec validated on Day 1 into a structured prompt, ask the AI for a structured output, and test it until the result meets the acceptance criteria. The prompt is not a loose piece of text: it is the direct translation of the model into instructions the AI understands and obeys. In a hiring agent — one that evaluates people — the prompt must also be fair and safe, and not only correct.

✅ Learning objectives

The day is built around what you should be able to do by the end of it. Use the list below as a self-check.

By the end of the day, you can

  • Derive a prompt from the spec — the five views defined on Day 1.
  • Structure a prompt with role, context, task, output format and constraints.
  • Ask for a structured output and calibrate the AI with examples.
  • Identify and fix a prompt failure.
  • Recognize bias and privacy risks in a selection agent.

Why it matters

  • The prompt is the bridge between the spec and the solution built in Lovable.
  • A robust prompt is predictable, fair and testable.
  • The quality of the selection funnel depends directly on it.

📋 From the spec to the prompt

Nothing in the prompt is invented: each part of the Day 1 spec becomes a part of the prompt. Reading the spec as the map below turns the prompt into a translation, not a guess.

Principle

Nothing in the prompt comes from nowhere — every line traces back to a decision recorded in the spec.

🔧 Anatomy of a good prompt

An effective prompt has six parts. Each one makes the instruction clearer and more verifiable, and each corresponds to something you already defined in the spec.

The 6 parts of an effective prompt

  1. Role: who the AI is in this context — for example, an impartial HR analyst.
  2. Context: the evaluation criteria and the role profile.
  3. Task: what exactly it must do — a clear verb, no ambiguity.
  4. Output format: how it should return the answer (a table, JSON).
  5. Constraints: what it must not do — derived directly from the non-functional requirements.
  6. Examples: a sample of what a good answer looks like.

Consider the spec from Day 1 — task: evaluate an interview answer; criteria: clarity, experience, fit to the profile; output: a 0–5 score per criterion with a justification; constraint: do not consider age, gender or origin. It translates into the prompt like this:

You are an impartial HR analyst.
Criteria: [criteria] · Role: [role].
Evaluate the answer [answer], giving a 0–5 score per criterion with one sentence of justification.
Do not use age, gender or origin. Format: table.

The parts in brackets are the parameters: what changes with each use. In Lovable, each parameter becomes a form field that the user fills in.

⚖️ Weak prompt vs. robust prompt

The same task can produce very different results. The example below evaluates the same interview answer with a weak prompt and with a robust one.

❌ Weak prompt

Evaluate this candidate: [answer]
  • No criteria: the AI decides on its own what matters.
  • No format: each answer comes out differently.
  • No constraints: it may consider age, gender or origin.
  • Result: inconsistent scores, subject to bias.

✅ Robust prompt

Role + criteria + format (table) + constraints (no protected attributes) + example
  • Always evaluates by the same criteria.
  • Standardized output — candidates can be compared.
  • Reduces bias and protects the candidate's data.
  • Result: predictable, fair and testable.

📊 Ask for a structured output

Free text is hard to compare across candidates. A fixed format — a table or JSON — turns loose text into data that can feed the ranking in the selection funnel. Always ask for the same three fields: a score per criterion, a justification, and a recommendation.

Respond in JSON:
{ "criterion": ..., "score": 0-5,
  "justification": ..., "recommendation": advance | review }

The same request can be returned as a table, which is easier to read during the evaluation:

CriterionScoreJustification
Clarity4Explained with concrete examples
Experience3Relevant, but little time in the role
Fit5Aligned with the role's competencies

Recommendation: advance to the next stage — the final decision remains with the recruiter.

🎯 Calibrate the AI with examples (few-shot)

Without a reference, each evaluation may vary. Including one or two examples of an answer together with its "correct" score teaches the AI the ruler you want it to use — a technique known as few-shot. The model then follows the same standard, which is exactly what the reliability requirement asks for. A few well-chosen examples are worth more than many.

Example of a score of 5: "answer with concrete data and a result"
Example of a score of 2: "generic answer, with no evidence"

Now evaluate to the same standard: [answer]

The examples teach the ruler: two similar answers start receiving similar scores.

🛡️ Responsible AI in recruitment

In an agent that evaluates people, responsible use is not optional. Three safeguards must be written explicitly into the prompt — never left implicit.

Three mandatory safeguards

  • Bias: instruct the AI to ignore age, gender, origin and appearance, and to evaluate only what relates to the work.
  • Privacy (LGPD): do not expose the candidate's personal data; use only what is necessary for the evaluation.
  • Human decision: the AI recommends, the recruiter decides. Never an automatic rejection.

These are constraints of the prompt, and they come straight from the spec's non-functional requirements (Day 1). They connect to ISO/IEC 25010 — the security and reliability characteristics. A good prompt states them clearly:

Constraints: evaluate only competencies related to the role; do not use sensitive personal data; flag uncertainty instead of inventing.

🔁 Iterate with test cases

A prompt is rarely good on the first version — and that is fine. It is adjusted with evidence. What sets an efficient group apart is a disciplined iteration method: change one part at a time and check the result against the spec.

The iteration cycle

  1. Write the first version with the six parts.
  2. Test with at least three sample answers.
  3. Compare the output against the spec's acceptance criteria.
  4. Locate which part of the prompt caused the deviation.
  5. Adjust only that part and test again.

Choose good test cases

  • A strong answer (should score high).
  • A weak answer (should score low).
  • A borderline or ambiguous answer.
  • If the AI gets the borderline case wrong, the prompt needs more criteria or another example.

Refinement and testing continue on Day 3, now informed by the technical visit to the partner.

🎛️ Parametrization and common failures

A well-written prompt serves many roles, not just one. The variable parts are marked as parameters, and the fixed parts — role, criteria structure, constraints — stay the same.

Parametrization

  • Mark what changes with brackets: [role], [criteria], [answer].
  • The same prompt then serves several roles.
  • In Lovable, each parameter becomes a form field.
  • Test with different parameters to make sure the prompt stays valid.

Most failures fall into a few recognizable patterns. Each has a direct fix in the prompt:

FailureCauseFix
Invents experienceThe prompt allows "filling gaps"Ask it to cite only what is in the answer
Inconsistent scoreNo criteria or examplesAdd criteria and calibration (few-shot)
Signs of biasNo constraint on attributesExplicitly forbid age / gender / origin
Broken formatFormat not specifiedRequire JSON or a table and give an example

🧾 Robust-prompt checklist

Answer "yes" to every item before considering the prompt done.

Is the prompt robust?

  • Does it give the AI a clear role?
  • Does it bring the context (criteria and role profile)?
  • Is the task a clear verb, without ambiguity?
  • Does it define the output format (table / JSON)?
  • Does it include bias and privacy constraints?
  • Was it calibrated with at least one example?
  • Was it tested on three answers (strong, weak, borderline)?
  • Was the output consistent across the tests?
  • Is it parameterized for other roles?
  • Does it make clear that the decision is human?

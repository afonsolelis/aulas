🧪 Prompt testing ⚠️ Edge cases 🏭 Technical visit

Why test the prompt

The happy path is misleading — robustness shows in the edge cases

Testing · Concept

⚠️ Where prompts break

  • Empty or incomplete input: the user does not fill in everything
  • Out-of-scope input: they ask for something that is not the problem
  • Malicious input: attempts to bypass the constraints (prompt injection)
  • Ambiguous input: several possible interpretations
  • Volume/size: text that is too long or too short

🛠️ How to handle failures

  • Validate before sending: check required fields in the interface
  • Instruct the fallback: state in the prompt what to do when data is missing
  • Reinforce constraints: repeat the limits at the end of the prompt
  • Clear messages: when the AI cannot respond, explain why
Principle
A robust prompt fails predictably — never with a wrong output presented as correct.

🔨 Hands-on — build the test table

🏭 In the morning — Technical Visit

The partner's real context, outside the classroom

Morning · Field

👁️ What to observe during the visit

Bring the group's spec. The visit is the chance to confirm or correct the Day 1 assumptions based on the real world.

  • How the problem appears in practice — not on paper
  • Who the people who would use the solution are and how they speak
  • Details of tone, vocabulary and context that change the prompt
  • What the spec got right and what needs to be revised

🔄 On returning

  • Review the spec and the test table with what you learned
  • Adjust the prompt if the real context requires it
  • Tomorrow is the main development sprint — arrive calibrated

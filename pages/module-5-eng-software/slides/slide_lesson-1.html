A Pirâmide de Testes

🧪 Unitários (Base)

Testam uma única função ou classe em isolamento. Rápidos, baratos e numerosos.

  • Sem dependências externas
  • Execução em milissegundos
  • Fáceis de manter

🔗 Integração (Meio)

Verificam a interação entre múltiplos componentes ou serviços.

  • Banco de dados, APIs
  • Mais lentos e complexos
  • Encontram bugs de contrato

🌐 E2E (Topo)

Simulam o comportamento real do usuário no sistema completo.

  • Playwright, Cypress
  • Mais lentos e frágeis
  • Alta confiança no fluxo
Regra de Ouro: Muitos unitários + alguns de integração + poucos E2E = equilíbrio ideal de velocidade e confiança.

Anatomia de um Teste (AAA)

📐 Padrão Arrange-Act-Assert

  • Arrange: configurar o estado inicial e dependências
  • Act: executar a função ou comportamento sob teste
  • Assert: verificar o resultado esperado

🎯 Boas Práticas

  • Um assert por teste sempre que possível
  • Nomes descritivos: should return error when input is empty
  • Testes independentes entre si
// Arrange const userService = new UserService(); const email = 'test@inteli.edu.br'; // Act const result = userService.validateEmail(email); // Assert expect(result).toBe(true);

Test-Driven Development (TDD)

🔴 Red — Escreva o Teste

Escreva um teste que falhe para o comportamento desejado. O teste define a interface.

🟢 Green — Faça Passar

Implemente o mínimo de código necessário para o teste passar. Sem otimização prematura.

🔵 Refactor — Melhore

Refatore o código mantendo os testes verdes. Elimine duplicação e melhore clareza.

Benefícios do TDD: Design emergente mais limpo, documentação viva, refatoração segura e confiança para mudanças.

Cobertura de Código

📊 Métricas de Cobertura

  • Line coverage: linhas executadas pelos testes
  • Branch coverage: ramificações if/else cobertas
  • Function coverage: funções chamadas
  • Statement coverage: statements individuais

⚠️ Cuidados com a Métrica

  • 100% de cobertura não garante ausência de bugs
  • Foque em testar lógica de negócio crítica
  • Evite testes sem assertions significativas
  • Meta razoável: 70-80% em projetos reais
// package.json — configuração do Jest com cobertura "jest": { "collectCoverageFrom": ["src/**/*.ts"], "coverageThreshold": { "global": { "lines": 80 } } }

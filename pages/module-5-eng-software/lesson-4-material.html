Objetivo da Aula

Compreender os valores e princípios do Manifesto Ágil, aplicar os frameworks Scrum e Kanban, e utilizar ferramentas práticas de gestão no projeto do módulo.

1) O Manifesto Ágil (2001)

17 desenvolvedores se reuniram em Snowbird, Utah, e definiram 4 valores e 12 princípios que mudaram a forma de desenvolver software.

🧑‍🤝‍🧑 Indivíduos e interações
mais que processos e ferramentas
✅ Software funcionando
mais que documentação abrangente
🤝 Colaboração com o cliente
mais que negociação de contratos
🔄 Responder a mudanças
mais que seguir um plano

Atenção: os itens à direita têm valor — apenas valorizamos os da esquerda mais.

2) Scrum

Papéis
  • Product Owner (PO): responsável pelo backlog e pela priorização de valor.
  • Scrum Master (SM): facilita o processo, remove impedimentos, protege o time.
  • Dev Team: auto-organizado, cross-funcional, entrega o incremento.
Cerimônias
  • Sprint Planning: seleciona itens do backlog para a sprint. Duração: 2–4h para sprint de 2 semanas.
  • Daily Scrum: 15 minutos. O quê fiz? O quê farei? Há impedimentos?
  • Sprint Review: demo do incremento para stakeholders. Coleta feedback.
  • Retrospective: o que funcionou? O que melhorar? Ações concretas.
Artefatos
  • Product Backlog: lista priorizada de tudo que precisa ser feito.
  • Sprint Backlog: itens comprometidos para a sprint atual.
  • Incremento: produto potencialmente entregável ao final de cada sprint.

3) Kanban

Kanban é um método de fluxo contínuo focado em visualizar o trabalho e limitar o WIP (Work In Progress).

📥 A Fazer
Configurar CI/CD
Escrever testes unitários
⚙️ Em Progresso (WIP: 2)
Implementar login
Migração do banco
✅ Concluído
Scaffold do projeto
Setup do ambiente

Limite de WIP: ao restringir o trabalho em progresso, o time é forçado a terminar antes de começar algo novo, reduzindo context switching e aumentando qualidade.

5) Ferramentas Práticas

  • GitHub Projects — integrado ao repositório, gratuito, issues vinculadas a commits e PRs.
  • Jira — completo, enterprise, Scrum e Kanban configuráveis.
  • Trello — Kanban simples, ótimo para times pequenos.
  • Linear — rápido, voltado para times de engenharia.
  • Notion — wiki + tasks, flexível mas menos estruturado.

Recomendação para o módulo: use GitHub Projects com issues nomeadas como User Stories no formato "Como [persona], quero [ação] para [benefício]".

6) Entregável da Aula 4

  • Configurar um board no GitHub Projects para o projeto do módulo.
  • Criar ao menos 5 issues no formato de User Story com critérios de aceitação.
  • Definir o backlog da primeira sprint com estimativa de story points (1, 2, 3, 5, 8).
  • Realizar uma Daily Scrum simulada com a equipe e documentar impedimentos.
